# models package

## Purpose

SQLAlchemy ORM definitions plus value objects. Aggregate roots own their child entities and expose business methods so services stay thin. Geometry columns are plain `Column(String)` storing ISO WKT (`POINT Z (lon lat alt)`, `LINESTRING Z (...)`, `POLYGON Z ((...))`) - the model layer holds raw strings; conversion to/from Shapely lives in `app.core.geometry` and is consumed by services.

## Public API surface

- `Mission` (`mission.py`) — aggregate root for missions, inspections, configs, flight plans, operator-attached constraint rules. Owns the status state machine.
- `DroneProfile` (`mission.py`) — drone hardware + performance limits, plus capability bits (`supports_geozone_upload: bool` — true when the airframe accepts a geofence polygon embedded in its mission file; gates the geozone bundle in `export_service`).
- `Inspection`, `InspectionConfiguration`, `InspectionTemplate` (`inspection.py`) — inspection aggregate; `InspectionConfiguration._MERGE_FIELDS` and `CONFIG_FIELDS` drive override merging. `InspectionConfiguration.lha_selection_rules` is a JSONB per-AGL rule dict (`{agl_id: {mode, params}}`) sitting alongside the canonical `lha_ids` list — it is recorded intent only; `app.services.lha_selection.apply_lha_selection` resolves it into `lha_ids` on every save. APPROACH_DESCENT carries two nullable Float columns on the configuration: `descent_start_distance` (meters back of the runway touchpoint where the descent begins; null falls back to `DEFAULT_DESCENT_START_DISTANCE = 1000 m` in `trajectory/types.py`) and `descent_glide_slope_override` (operator-pinned descent angle in degrees; null uses the resolved PAPI glide slope). Both are part of `CONFIG_FIELDS`, so `resolve_with_defaults` carries them through and `Mission.duplicate()` clones them. Migration `0008_approach_descent_config` adds the columns. SURFACE_SCAN carries 11 more nullable columns: `scan_surface_id` (UUID FK → `airfield_surface`, `ON DELETE SET NULL` — SURFACE_SCAN is AGL-agnostic and targets a surface, not an LHA), `scan_length_mode` / `scan_width_side` / `scan_run_orientation` (String columns whose CHECK constraints are rendered from the `ScanLengthMode` / `ScanWidthSide` / `ScanRunOrientation` enums), and `scan_length_from` / `scan_length_to` / `scan_width` / `scan_height` / `scan_run_count` / `scan_sidelap_percent` / `scan_frontlap_percent` (the last drives the PHOTO along-track capture spacing; null falls back to `DEFAULT_SURFACE_SCAN_FRONTLAP_PERCENT = 0`, mirroring the sidelap default plumbing). All 11 are in `CONFIG_FIELDS` so they merge and clone like the rest. Migration `0015_surface_scan_config` adds the columns, the FK, and the CHECK constraints; `0016_surface_scan_frontlap` adds the additive nullable `scan_frontlap_percent` column.
- `Airport`, `AirfieldSurface`, `Runway`, `Taxiway`, `Obstacle`, `SafetyZone` (`airport.py`) — airport aggregate; single-table inheritance for surfaces. `AirfieldSurface.paired_surface_id` is a nullable self-FK to the reciprocal RUNWAY direction (`ON DELETE SET NULL`), exposed via the `paired_surface` relationship (`remote_side=[id]`, `post_update=True`); pair invariants (reversed centerline, swapped threshold/end, reciprocal heading, identifier rename block, cascade-through-pair on delete) are owned by `airport_service`, not the model.
- `AGL`, `LHA` (`agl.py`) — runway lighting; `AGL.calculate_lha_center_point()` returns the centroid for trajectory geometry. `LHA.sequence_number` is a per-AGL dense ordinal (1..N) with `UNIQUE (agl_id, sequence_number)` and a `> 0` check; `LHA.validate_sequence_target(target, n_lhas)` is the model-side range guard used by the airport service. `LHA.lens_height_msl_m` / `LHA.lens_height_agl_m` are two nullable `Float` columns holding the PAPI lens optics height (MSL and AGL; null for non-PAPI units) — surveyed metadata, not `position.z`; migration `0010_lha_lens_height` adds both.
- `FlightPlan`, `Waypoint`, `ValidationResult`, `ValidationViolation`, `ExportResult` (`flight_plan.py`). `ValidationViolation.violation_kind` is a nullable `String` written by the trajectory orchestrator / persist paths at emission (`surface_crossing`, `battery`, validator-emitted kinds, `camera_obstruction`); null only on legacy rows persisted before the `violation_kind` column was introduced. `ValidationViolation.is_warning` is a backwards-compat computed property (`category != "violation"`), not a column. `Waypoint.agl` and `Waypoint.camera_target_agl` are nullable `Float` columns persisting rendering-only height above sampled ground — populated by the trajectory orchestrator at generation time and refreshed only by `flight_plan_service` (moved/new rows on edit, every row on terrain-source change, lazy backfill on first read for legacy rows). The trajectory pipeline still flies at the persisted MSL in `Waypoint.position`; AGL columns exist so the 3D viewer renders without firing per-read Open-Elevation calls.
- `ConstraintRule` + subclasses (`flight_plan.py`, but FK is `mission_id` — back-populates `Mission.constraints`). Subtypes: `AltitudeConstraint`, `SpeedConstraint`, `BatteryConstraint`, `RunwayBufferConstraint`, `GeofenceConstraint`. Single-table inheritance on `constraint_type`.
- `User` (`user.py`) — auth + role.
- `AuditLog` (`audit_log.py`) — immutable per-action row. Carries `user_id` (FK, ON DELETE SET NULL), `action`, `entity_type` / `entity_id` / `entity_name`, `details` (JSONB), `ip_address`, and a denormalized `airport_id` (FK → `airport.id`, ON DELETE SET NULL, indexed) used for fast per-airport activity scoping.
- `DroneMediaFile` (`drone_media_file.py`) — one uploaded original, either hub-reported (keyed by a `fingerprint`) or a manual per-inspection upload. Carries `object_key`, nullable `fingerprint` (the hub idempotency key; null for manual uploads), device-reported `captured_at` (never server receive time), `capture_position` (nullable WKT point), `device_sn` (indexed), `raw_callback` (hub callback payload verbatim, JSONB), `status` (`MediaFileStatus`, CHECK generated from the enum; hub rows arrive as `RECEIVED`, manual rows as `MATCHED`), and `updated_at` (onupdate `now()` — the audit trail for matching + manual reassignment). `origin` (`MediaOrigin`, CHECK from the enum; `HUB` default for hub rows, `MANUAL` for uploads), `filename` / `size_bytes` (`BigInteger`) are the upload-listing metadata. `mission_id` (FK → mission, ON DELETE SET NULL, indexed) is null while unmatched; `inspection_id` (FK → inspection, ON DELETE SET NULL, indexed) and `order_index` (dense 1..N per inspection) attach the file to one inspection in the upload form — both null until attached, both set once attached (paired-null CHECK `ck_drone_media_file_order_inspection`), `order_index > 0` CHECK, and `UNIQUE (inspection_id, order_index)`. Fingerprint uniqueness is a **partial** unique index (`uq_drone_media_file_fingerprint WHERE fingerprint IS NOT NULL`) so null-fingerprint manual rows coexist. Status transitions live on the model: `assign_to_mission(mission_id)` (→ `MATCHED`), `mark_unassigned()` (→ `UNASSIGNED`, clears `mission_id`), `mark_ingested()` (→ `INGESTED`, idempotent no-op returning false when already ingested); the first two raise `DomainError(409)` once the row is `INGESTED`. `validate_order_target(target, n)` is the model-side range guard (1..n) used by the move path. Migrations `0011_drone_media_file`, `0013_drone_media_updated_at`, `0016_drone_media_per_inspection`.
- `WaylineDispatch` (`wayline_dispatch.py`) — one mission's wayline in the field hub's route library: the mission ↔ wayline mapping persisted by a *Send to drone* dispatch. `mission_id` is a unique FK (ON DELETE CASCADE), so a re-dispatch updates the existing row in place; `wayline_id` is the stable uuid presented to Pilot 2's route list (never regenerated, so Pilot sees an updated route instead of a duplicate); `device_sn` stays null until a flight execution binds one — flight-task progress events aren't persisted yet, so `drone_media_service` matching treats it as optional (equality enforced only when both sides carry one); `status` defaults to the module constant `DISPATCH_STATUS_DISPATCHED = "DISPATCHED"` (flight-task tracking values land when progress events are persisted). `mark_dispatched()` resets status and bumps `dispatched_at` for a re-dispatch. Migration `0012_wayline_dispatch`.
- `Measurement` (`measurement.py`) — ORM row for one inspection's video-measurement run: `inspection_id` (FK → inspection, ON DELETE CASCADE, indexed `ix_measurement_inspection_id`), `status` (String, CHECK `ck_measurement_status` generated from `MeasurementStatus` via `enum_check_values`), `label` (nullable — operator free-text run name; blank clears it back to the inspection-label fallback, added by migration `0019_measurement_label`), `runway_heading`, the JSONB columns `reference_points` / `light_boxes` / `summaries` / `media_object_keys` (default `[]`) + `annotated_video_keys` (default `{}`), the object-storage pointers `first_frame_object_key` / `object_key` (the gzipped per-frame results blob — never stored in Postgres), `error_message`, and `created_at` / `updated_at`. This is a **dumb data holder for the sqlalchemy adapter only** — all measurement business logic (status machine, scoring, reference snapshot) lives on the domain aggregate `app.domain.measurement.entities.Measurement`, persisted through `app.infra.measurement.sqlalchemy_repository`.
- `CameraPreset`, `SystemSettings` — supporting tables.
- `value_objects.py` — `Coordinate`, `Speed`, `AltitudeRange`, `IcaoCode`. Immutable, range-validated primitives.
- `mission.py` module-level: `TRAJECTORY_FIELDS` (set of update-keys that regress to DRAFT), `TRANSITIONS` (status state machine), `MAX_INSPECTIONS = 10`, `_DJI_HEADING_MODE_VALUES = ("smoothTransition", "towardPOI", "followWayline")` (mirrors the CHECK constraint emitted by migration `0002_dji_heading_mode` so the python literal and the DB constraint cannot drift).
- `Mission.dji_heading_mode` (`String(20)`, default `'smoothTransition'`) — operator's last-used per-export DJI WPML `waypointHeadingMode`. Only consumed by KMZ / WPML generators; ignored by every other format. The export endpoint accepts a per-export override and writes it back here on success; deliberately NOT in `TRAJECTORY_FIELDS` so flipping the export shape never regresses status.

## Invariants

- **Status changes only via `mission.transition_to(target)`.** Never assign `mission.status` directly — the state machine in `TRANSITIONS` is the only legal source of transitions. Two exceptions live on the model itself: `regress_to_planned` and `invalidate_trajectory`, which intentionally bypass the forward-only state machine. `mark_measured()` is NOT a bypass — it routes through `transition_to(MissionStatus.MEASURED)`, gated so it only fires while the mission is in `POST_PLAN_STATUSES` (idempotent on repeat measurement kickoffs).
- **`MEASURED` belongs to exactly one status-set frozenset: `EXPORT_ELIGIBLE_STATUSES`.** `TRANSITIONS` allows `VALIDATED → MEASURED` (skipping EXPORTED), `EXPORTED → MEASURED`, and `MEASURED → {COMPLETED, CANCELLED}`. Beyond the export/dispatch gate, `MEASURED` was deliberately left out of the other five frozensets below: a MEASURED mission is not treated as terminal (it stays deletable), is not post-plan, and is not in the waypoint-edit / regression sets — those gates simply never fire for it. The one set it does join is `EXPORT_ELIGIBLE_STATUSES`, because re-export / re-dispatch is read-only against the persisted plan + artifacts that survive measurement. Don't add `MEASURED` to any of the other five to "fix" a gate; that would change deletion/edit/regression semantics the feature intentionally left alone. Editing IS blocked for MEASURED, but via an explicit guard clause in `invalidate_trajectory()` (raises `cannot modify mission after measurement`), not via a frozenset — the lock is a direct status check, so the directive holds: don't reach for a frozenset to express it.
- **Status set frozensets live on `Mission`, not in service-side locals.** Six class-level frozensets are the canonical "which statuses count for this gate" definitions and every caller must reuse them rather than re-deriving inline tuples or sets:
  - `TERMINAL_STATUSES = frozenset({MissionStatus.COMPLETED, MissionStatus.CANCELLED})` — `EXPORTED` is *not* terminal; it can still be deleted, transitioned, or duplicated.
  - `NON_DRAFT_WITH_PLAN_STATUSES = frozenset({PLANNED, VALIDATED, EXPORTED})` — has a persisted flight plan but isn't terminal; `invalidate_trajectory()` and the airport renormalize loop use it to decide when to regress to DRAFT.
  - `POST_PLAN_STATUSES = frozenset({VALIDATED, EXPORTED})` — the auto-regress branch in the trajectory orchestrator, plus the `mark_measured` / `regress_to_planned` gates. No longer the export gate (that moved to `EXPORT_ELIGIBLE_STATUSES`).
  - `EXPORT_ELIGIBLE_STATUSES = frozenset({VALIDATED, EXPORTED, MEASURED})` — export/dispatch gate (`export.orchestrator.export_mission`; `wayline_dispatch_service.dispatch_mission` reuses it). The only frozenset MEASURED is in; kept deliberately separate from `POST_PLAN_STATUSES` so widening export eligibility never alters the regress / measurement-kickoff gates that consume `POST_PLAN_STATUSES`. The `VALIDATED → EXPORTED` bump still keys on `VALIDATED`, so a measured mission stays MEASURED after export.
  - `PRE_EXPORT_EDITABLE_STATUSES = frozenset({DRAFT, PLANNED, VALIDATED})` — waypoint write gate.
  - `PRE_PLAN_STATUSES = frozenset({DRAFT, PLANNED})` — trajectory-generation gate and the bulk-drone-change SQL filter.

  Frozensets hold `MissionStatus` enum members, not bare strings. `MissionStatus(str, Enum)` means both `mission.status == "VALIDATED"` (ORM column comes back as a string) and `mission.status == MissionStatus.VALIDATED` evaluate true against the same frozenset; SQLAlchemy `in_(frozenset(...))` iterates the frozenset for `Mission.status.in_(Mission.PRE_PLAN_STATUSES)`. Don't re-introduce open-coded `(MissionStatus.X, MissionStatus.Y)` tuples or `{"X", "Y"}` literals at call sites — that's the duplication these frozensets collapsed.
- **Keep-stale flight plan contract**: `Mission.invalidate_trajectory()` regresses status, sets `has_unsaved_map_changes = True`, and resets computation status — but it does NOT clear `mission.flight_plan`. The previously computed plan stays so the frontend can render it as a stale reference until the operator triggers a fresh recompute. Mission deletion still wipes the row via the DB-level CASCADE (the relationship uses `passive_deletes=True`). Before any of that it raises `ValueError` on `TERMINAL_STATUSES` (`cannot modify mission in completed or cancelled state`) and on `MEASURED` (`cannot modify mission after measurement`) — `inspection_service` / `mission_service` map both to `DomainError(409)`. Since `add_inspection` / `remove_inspection` / `change_drone_profile` / `regress_if_trajectory_changed` all funnel through this method, the single guard locks every plan-edit path for a MEASURED mission: the footage was already scored against the planned LHA ground truth, so editing afterward would orphan the measurement.
- **Constraints are mission-owned, not flight-plan-owned.** `ConstraintRule.mission_id` is the FK; `Mission.constraints` uses `cascade="all, delete-orphan"`. Operator-attached rules survive flight-plan regeneration — replacing `mission.flight_plan` does not touch them. Mission deletion cascades through. Don't move the FK back onto `FlightPlan`; with the FK on `FlightPlan` every recompute silently dropped the operator-attached rules.
- **Aggregate boundaries**: child entities (Inspection, AirfieldSurface, Obstacle, SafetyZone) are created through aggregate-root methods (`mission.add_inspection`, `airport.add_surface`, etc.), not constructed and appended directly.
- **Max 10 inspections per mission** — enforced in `Mission.add_inspection`.
- **Enums import from `app.core.enums`**, not from the deleted `app.models.enums` shim.
- **Constants** like `MIN_TRANSIT_ALTITUDE_AGL_M` and `DEFAULT_BUFFER_DISTANCE_M` come from `app.core.constants`. Don't redefine numeric thresholds locally.
- **CHECK constraints** on enum-string columns are generated from the python enum via the shared `enum_check_values(Enum)` helper in `app.core.enums` (`_OBSTACLE_TYPE_VALUES = enum_check_values(ObstacleType)`, `_TERRAIN_SOURCE_VALUES`, `_MEDIA_FILE_STATUS_VALUES`, etc.) - reuse that helper, not a hand-rolled `", ".join(...)`, when adding a new discriminator column so the DB constraint and the python enum can't drift. The rendered SQL is byte-identical to the old inline join (pinned by `test_enum_check_values.py`), so swapping in the helper needs no migration.
- **`AuditLog.airport_id` is denormalized scope, not derived at read time.** Producers (routes via `log_audit`) decide once and write the value into the row; readers filter on the column directly. The FK uses `ON DELETE SET NULL` so deleting an airport leaves orphan rows pointing nowhere rather than cascading the audit history. Don't drop the index — the per-airport activity panel is the primary read pattern.
- **`Mission.dji_heading_mode` is export-time presentation, not a trajectory input.** The trajectory pipeline never reads it — waypoint (lon, lat, alt) and `wp.heading` are independent of the chosen mode. Keeping it out of `TRAJECTORY_FIELDS` is intentional so the export endpoint can write back the operator's per-export choice without regressing status to DRAFT. The CHECK constraint is rebuilt from `_DJI_HEADING_MODE_VALUES`; widen both together if a fourth WPML mode is ever added.

## Cross-package dependencies

- Imports from: `app.core` (database Base, enums, constants, exceptions), `app.utils.geo`. No GeoAlchemy2 - geometry round-trip is a service concern via `app.core.geometry`.
- Imported by: `app.services` (heavy use), `app.infra` (the measurement sqlalchemy adapter maps the `Measurement` ORM row), `app.schemas` (only when a schema needs an enum or a constant — most enums now live in `app.core.enums` to avoid this), tests.
- Models must NOT import from `app.services` or `app.schemas` — that would invert the dependency rule.

## Gotchas

- **`Mission.duplicate()` returns a detached aggregate and carries the trajectory over conditionally.** It does not call `db.add` / `db.flush` - the service attaches and flushes. A non-DRAFT original WITH a flight plan deep-copies the whole `FlightPlan` aggregate (waypoints + validation result + violations) and promotes the copy to PLANNED; a DRAFT original - even one holding a stale flight-plan row from the keep-stale contract - clones to a clean DRAFT with no plan, preserving the "PLANNED ⇒ current trajectory" invariant. Copied waypoints rebind to the freshly-minted inspection rows via an old→new inspection map (the raw `inspection_id` FK would dangle otherwise; None for TAKEOFF/TRANSIT/LANDING), and `ValidationViolation.waypoint_ids` are remapped to the copy's pre-assigned waypoint ids. NOT carried over: `ConstraintRule`s, the `ValidationViolation.constraint_id` remap, and `ExportResult` rows; the trajectory is a verbatim deep-copy, never regenerated.
- **`Mission.regress_if_trajectory_changed(data)` does not apply field values.** It only decides whether to regress. The caller still owns `apply_schema_update` / `setattr`.
- **`Mission.modify_inspections(callback)` invalidates the trajectory whether or not the callback mutates anything** — the contract is "after this call, the trajectory is stale." Don't pass cheap read-only callbacks.
- **`has_unsaved_map_changes` is owned by the model.** `invalidate_trajectory()` is the single source of truth for flipping it on; services must not set the flag directly to keep all invalidation paths consistent.
- **`Inspection.validate_method_agl_compat`** does an in-method import of `METHOD_AGL_COMPAT` to dodge a circular at module load. Keep it that way.
- **Single-table inheritance** for `AirfieldSurface` / `Runway` / `Taxiway` uses `surface_type` as the discriminator — don't add subtypes without updating both the python class hierarchy and the CHECK constraint.
- **`paired_surface_id` is opt-in metadata, not a planner construct.** The trajectory pipeline reads two surfaces regardless of pairing. Don't move pair logic onto the model — the `SELECT FOR UPDATE` partner lock, geometry propagation, threshold-driven AGL `distance_from_threshold` recompute, and identifier-rename block all live in `airport_service` because they need the session and a transactional boundary. The relationship uses `post_update=True` to side-step a chicken-and-egg insert order when both sides reference each other; the migration adds a partial unique index keyed on the pair so the link cannot fan out.
- **`lha_selection_rules` is recorded intent, `lha_ids` is the truth.** The trajectory pipeline never reads `lha_selection_rules` — it consumes `lha_ids` only. Don't pull selection logic into the model: the resolver lives in `app.services.lha_selection` because it needs DB access to load the parent AGLs and surfaces. `_MERGE_FIELDS` includes `lha_selection_rules` so `Mission.duplicate()` carries it across, but assigning it directly without re-resolving will leave `lha_ids` stale.
- **`Mission.supports_geozone_upload` is a property, not a column.** It proxies through `Mission.drone_profile.supports_geozone_upload` and returns `None` when no drone profile is attached, so the export panel can distinguish "no drone selected" from "drone selected but incapable". Surfaced on `MissionResponse` to avoid a second round-trip. Don't materialize it as a column - the source of truth lives on `DroneProfile`.
- **LHA sequence_number is owned by the airport service, not the model.** The model only enforces the unique/check constraints and the `validate_sequence_target` range guard. Don't assign `lha.sequence_number` directly — the dense-1..N invariant relies on the service's `SELECT FOR UPDATE` + sentinel + ±1 shift dance, and on `_apply_papi_invariant` to keep PAPI letters aligned (1=A..4=D).
- **`Waypoint.agl` / `Waypoint.camera_target_agl` are owned by `flight_plan_service`, not the model.** Nullable means "not yet computed"; build_enriched_response's one-shot lazy backfill will compute and persist on first read. TAKEOFF / LANDING force `agl=0` regardless of sampled ground. Don't write these columns from models or routes — every write path lives behind `_refresh_persisted_agl` (waypoint moves / inserts), `_compute_waypoint_data_agl` (initial persist from the trajectory orchestrator), or `_backfill_waypoint_agl` (legacy-row read-path backfill) so TAKEOFF/LANDING handling and the batched-provider-call discipline stay in one place. The columns are intentionally absent from `TRAJECTORY_FIELDS` — refreshing them on a terrain-source change is rendering-only and must NOT regress status.
- **`ValidationViolation.violation_kind` is set at emission, never by the model.** The trajectory orchestrator tags each soft warning/violation with a structured kind and the persist paths write it onto the row; the model only declares the nullable column. A `null` value means a legacy row written before the `violation_kind` column existed — the *schema* (`ValidationViolationResponse`) classifies those from `message` as a fallback, the model does not. Don't add a model-side default or classifier; that would mask which rows are genuinely structured.
- **`DroneMediaFile` rows are created by two paths; transitions go through the model methods.** Hub rows come from `field_link_service.record_media_event` (`origin=HUB`, `status=RECEIVED`); manual uploads come from `drone_media_service.complete_upload` (`origin=MANUAL`, `status=MATCHED`, `fingerprint=None`). Fingerprint uniqueness (partial index, null-fingerprint rows excluded) is the hub-retry idempotency contract — a repost returns the existing row untouched. The `RECEIVED → MATCHED / UNASSIGNED → INGESTED` transitions are owned by `drone_media_service` (auto-matching, manual reassignment, ingest confirm) and must route through `assign_to_mission` / `mark_unassigned` / `mark_ingested` — never assign `status` or `mission_id` directly, the ingest block and the idempotency contract live on the methods. **The dense `order_index` (1..N per inspection) is owned by `drone_media_service`, not the model** — the model only enforces the constraints + the `validate_order_target` range guard; the sentinel-shift renumber + the parent (mission) row lock live in the service (mirrors the LHA `sequence_number` protocol).
- **`WaylineDispatch` rows are written only by `wayline_dispatch_service.dispatch_mission`.** The unique `mission_id` makes re-dispatch an in-place update — don't insert a second row per mission, and don't regenerate `wayline_id` on update (the stable uuid is what keeps Pilot 2's route list from growing a duplicate per re-dispatch). A persisted row means the hub acknowledged the wayline: the service raises 502 before the route commits when registration fails, and the row rolls back.
- **Vertical profile climb bookends live on `InspectionConfiguration`.** `angle_source` (`'PAPI'` or `'CUSTOM'`, null = CUSTOM legacy) selects how the trajectory pipeline resolves the VP climb top/start; `angle_start` and `angle_end` carry the CUSTOM bookends; `angle_offset_above` / `angle_offset_below` carry the PAPI offsets (the old `angle_offset` was renamed to `angle_offset_above`). The previous `vertical_profile_height` column is gone — migration `c5d6e7f8a9b0_vertical_profile_angles` backfilled `angle_end = atan2(vertical_profile_height, COALESCE(horizontal_distance, 400))` with `angle_source='CUSTOM'` so existing rows keep the geometric top bookend. All four columns are part of `CONFIG_FIELDS`, so `resolve_with_defaults` carries them through and `Mission.duplicate()` clones them.
- **There are TWO `Measurement` classes — don't confuse them.** `app.models.measurement.Measurement` is the ORM row (this package); `app.domain.measurement.entities.Measurement` is the domain aggregate that owns the status machine, scoring, and the LHA reference snapshot. The ORM model is deliberately logic-free: do NOT add methods, a status state machine, or scoring to it — that would fork the rule the domain aggregate owns. The `app.infra.measurement.sqlalchemy_repository` adapter is the only place the two meet (it maps the aggregate's fields to/from this row). The `status` CHECK is rendered from the `MeasurementStatus` enum via `enum_check_values`, same protocol as every other discriminator column.
