# schemas package

## Purpose

Pydantic v2 DTOs that define the HTTP request/response shapes. Routes use schemas at the boundary; services convert schemas to ORM data via `geometry_converter` and back via response models.

## Public API surface

- One file per resource: `mission.py` (mission/inspection wrappers), `inspection_config.py` (inspection-config schemas + `LhaSelectionRule*` union), `surface.py`, `obstacle.py`, `safety_zone.py`, `agl.py` (AGL/LHA/bulk), `airport.py`, `flight_plan.py`, `inspection_template.py`, `drone_profile.py`, `camera_preset.py`, `auth.py`, `admin.py`, `export.py`, `openaip.py`, `field_link.py`, `wayline_dispatch.py`, `measurement.py` (video-measurement DTOs). `infrastructure.py` is a pure re-export shim over `surface`/`obstacle`/`safety_zone`/`agl`; `mission.py` re-exports the moved `inspection_config` names; `_violation_classifier.py` is the private stdlib-only legacy-classification module that `flight_plan.py` re-exports. Old import paths (`from app.schemas.infrastructure import ...`, the moved names off `app.schemas.mission`, `from app.schemas.flight_plan import _classify_violation`) all still resolve — that is deliberate compat surface, do not "fix" importers to point at the new modules.
- `flight_plan.py` — `WaypointResponse` carries the persisted columns (`position`, `heading`, `speed`, `camera_action`, `waypoint_type`, `camera_target`, `gimbal_pitch`, etc.) plus two rendering-only fields: `agl: float | None` (per-WP AGL above locally sampled ground; `0.0` for TAKEOFF / LANDING) and `camera_target_agl: float | None` (same shape for the LHA aim point when set). Both are populated by `flight_plan_service.build_enriched_response` from a single batched elevation lookup; the 3D viewer reads them directly. `FlightPlanResponse` carries the global flight envelope (`min/max_altitude_msl/_agl`) and per-inspection stats. `ValidationViolationResponse` exposes the stored `violation_kind` plus computed `constraint_name` (human-readable name mapped from the kind), `waypoint_ref`, `is_warning`, and `severity`; a `model_validator(mode="after")` classifies `violation_kind` from `message` only when the persisted value is `None` (legacy rows written before the `violation_kind` column existed). The classification machinery (`_VIOLATION_KIND_RULES`, `_classify_violation`, `_CONSTRAINT_NAME_MAP`, `_WP_REF_RE`, `_extract_waypoint_ref`) physically lives in the private `_violation_classifier.py` (stdlib-only, no service/model imports) and is re-exported by `flight_plan.py` so `from app.schemas.flight_plan import _classify_violation` and `test_flight_plan_schema` stay unbroken — consume via the re-export, not the private module.
- `surface.py` — `SurfaceResponse` exposes `paired_surface_id: UUID | None` so the frontend can dedup paired RUNWAY rows and render the chain affordance. Pair-link request bodies: `SurfaceCoupleRequest` (`target_surface_id`, `primary: Literal["self", "target"] = "self"`) and `SurfaceCreateReverseRequest` (optional `identifier` override; service derives the reciprocal when omitted). `SurfaceResponse.agls: list["AGLResponse"]` is a cross-module forward ref after the split — `surface.py` imports `AGLResponse` from `agl.py` and calls `SurfaceResponse.model_rebuild()` at module end; keep that call (previously resolved implicitly via the shared `infrastructure.py` namespace).
- `export.py` — `ExportRequest` (formats list + opt-in `include_geozones` / `include_runway_buffers` / `acknowledge_altitude_clamps`, plus optional `dji_heading_mode_override`), `GEOZONE_CAPABLE_FORMATS` (the `{MAVLINK, JSON, UGCS, KMZ, KML}` set shared with `export_service` for capability gating), and the clamp-warning response shapes `AltitudeClamp` (one record: `waypoint_index`, `intended_alt`, `clamped_alt`, `reason: Literal["below_takeoff"]`) + `ExportResponse` (`{altitude_clamps: list[AltitudeClamp]}`, returned as the 409 body when `acknowledge_altitude_clamps=False` and KMZ/WPML found below-takeoff placemarks).
- `geometry.py` — `PointZ`, `LineStringZ`, `PolygonZ` Pydantic models. Inbound WKT strings are parsed via `app.core.geometry.wkt_to_geojson`.
- `common.py` — shared literals (`CaptureModeStr`, `FocusModeStr`, `WhiteBalanceStr`, etc.), `ListMeta` pagination wrapper, `DeleteResponse` (`{deleted: bool, warnings: list[str] = []}`, the shared `response_model` for every delete route - template / drone / admin user delete; `warnings` is additive and the frontend reads only `.deleted`), and `validate_range_order(from_val, to_val, message, *, allow_equal=False)` - the shared from/to ordering check reused by the three real from/to validators (`LhaSelectionRuleRangeParams._check_bounds` with `allow_equal=True` for inclusive bounds; the two `_check_angle_band` validators strict). It lives here rather than in `inspection_config.py` so `inspection_template.py` reuses it without importing a module-private name across schema modules; both sites already depend on `common`.
- Naming: `{Entity}Create`, `{Entity}Update`, `{Entity}Response`, `{Entity}DetailResponse`, `{Entity}ListResponse`. Nested `*Override` schemas (e.g. `InspectionConfigOverride`) for partial-update payloads.
- Enum-typed fields are typed with `Literal[...]` aliases (`MissionDirectionStr`, `SurfaceTypeStr`, etc.) so OpenAPI shows the discriminated union.
- `LhaSelectionRule` (`inspection_config.py`, re-exported via `mission.py`) — discriminated union (`mode` discriminator) of `LhaSelectionRuleAll`, `LhaSelectionRuleRange`, `LhaSelectionRuleFromThreshold`, `LhaSelectionRuleCustom`. Reused by `inspection_template.py` (still imports it off `mission.py`) so templates and inspections speak the same shape. Range params alias `from_` ↔ `from` so the JSON wire format keeps the natural keyword.
- `inspection_config.py` and `inspection_template.py` — APPROACH_DESCENT fields `descent_start_distance: float | None = Field(default=None, gt=0)` (meters back of touchpoint) and `descent_glide_slope_override: float | None = Field(default=None, gt=0, le=10)` (degrees) ride on both `InspectionConfigOverride` (mission-attached) and `InspectionConfigCreate` (template defaults), with matching pass-through entries on the `*Response` shapes. The schemas only enforce the numeric bounds; the "needs a runway touchpoint" precondition is raised at compile by `app.services.trajectory.methods._prepare_approach_descent`. SURFACE_SCAN's 12 fields + the INTERVAL-ordering validator live on the shared `ScanConfigFields(BaseModel)` mixin in `inspection_config.py` (#868); all four config classes inherit it — `InspectionConfigOverride` / `InspectionConfigResponse` here and `InspectionConfigCreate` / `TemplateConfigResponse` in `inspection_template.py` — so the `scan_*` fields ride every `*Override` / `*Create` / `*Response` shape from one declaration. Edit a scan field once on the mixin, not per class. The fields: `scan_surface_id: UUID | None`, `scan_length_mode` (`ScanLengthModeStr = Literal["FULL", "MAX_LENGTH", "INTERVAL"]`), `scan_length_anchor` (`ScanLengthAnchorStr = Literal["THRESHOLD", "ENDPOINT"]` — which runway end the length window is measured from), `scan_length_from` / `scan_length_to` (`ge=0`), `scan_width` (`gt=0`), `scan_width_side` (`ScanWidthSideStr = Literal["LEFT", "RIGHT"]`), `scan_height` (`gt=0`), `scan_run_count` (`ge=1`), `scan_run_orientation` (`ScanRunOrientationStr = Literal["LENGTH_WISE", "WIDTH_WISE"]`), `scan_sidelap_percent` (`ge=0, le=80`), and `scan_frontlap_percent` (`ge=0, le=80` — forward overlap driving the PHOTO along-track capture spacing, symmetric with sidelap). `ScanConfigFields` and the four scan literal aliases live in `inspection_config.py` and are re-exported via `mission.py` (which `inspection_template.py` imports both from). The "scan surface must belong to the mission's airport" precondition is a 422 raised by `inspection_service`, NOT the schema.
- `admin.py` — `AuditLogResponse` exposes `airport_id: UUID | None` alongside the existing `entity_*` triple so the super-admin UI can scope the activity feed without a second lookup. `AuditLogListResponse` wraps it with `meta`. The two admin list endpoints get their own response wrappers so they stop returning bare dicts: `UserListResponse` (`data: list[UserAdminResponse]` + `meta: UserListMeta`) and `AdminAirportListResponse` (`data: list[AirportAdminResponse]`, no meta); `delete_user` returns the shared `DeleteResponse` from `common.py`. `AirportAdminResponse` is the admin-overview airport row: identity fields plus `user_count` / `coordinator_count` / `operator_count` / `mission_count` / `drone_count` (all `int = 0`) — computed aggregates filled by `admin_service.list_airports_admin`'s subqueries, not ORM columns, so a new count needs both a schema field and a subquery. `coordinator_count == 0` is what the super-admin UI renders as the orphaned/`Unassigned` flag. Frontend twin: `AirportAdminResponse` in `frontend/src/types/admin.ts` — keep the two in lockstep. `SystemSettingsResponse` carries `maintenance_mode`, `cesium_ion_token`, `elevation_api_url`, `elevation_api_fallback_enabled` (master toggle), `elevation_api_provider` (`ElevationApiProviderStr = Literal["OPEN_ELEVATION"]` — narrow union today, future backends land as new literal values), and `elevation_api_key` (`str | None`, returned masked as the `ELEVATION_API_KEY_MASK = "••••••"` sentinel when set; `None` when no key is configured). `SystemSettingsUpdate` is the partial-update mirror (every field `| None = None`). PUT bodies echoing the mask sentinel are a no-op on the api-key column so a GET → edit-other-field → PUT cannot clobber the key. The fallback toggle is the DB-backed gate that supersedes the env var on existing installs; the provider + key rows keep the wire format provider-agnostic.
- `airport.py` — `AirportResponse.terrain_source: Literal["FLAT", "DEM_UPLOAD", "DEM_API", "DEM_SRTM"]` (default `FLAT`) and `ElevationAtPointResponse` (`elevation: float`, `source: Literal["FLAT", "DEM_UPLOAD", "DEM_API", "DEM_SRTM", "API"]`, the response model for `GET /airports/{id}/elevation`) both carry `DEM_SRTM` (offline-staged Copernicus GLO-30 DEM) so a staged airport serializes with no downstream changes. The `API` label replaces the earlier `API_FALLBACK` label — keep frontend literal unions in lockstep. Out-of-range lat/lon are rejected by the route's `Query(..., ge=..., le=...)` constraints (422), not by this schema. `PhotoMetadataItem` (`filename`, `coordinates: PointZ | None`, `lens_height_msl_m` / `lens_height_agl_m: float | None`, `error: str | None`) and `PhotoMetadataResponse` (`items`, `has_dem: bool`) are the read-only response shapes for `POST /airports/{id}/extract-photo-metadata`: `coordinates` is null and `error` is set when a photo carries no GPS (the batch never fails), and `has_dem=false` means every `lens_height_agl_m` is null so the frontend prompts for manual AGL.
- `field_link.py` — status shapes: `FieldLinkDevice` (`sn`, `model_name`, `model_key`, `domain`, `online`, `bound`, `gateway_sn`) and `FieldLinkStatusResponse` (`hub_online`, `rc_connected`, `broker_connected`, `devices`, plus `connect_url` / `public_host` — the device-facing connect address proxied from the hub, both `str | None` and null when no host is derivable), mirroring the field hub's internal status payload. The four signals are distinct (#109): `hub_online` (backend reaches the hub), `rc_connected` (Pilot's HTTP session is live — defaults `False`), `broker_connected` (hub↔broker link), and `devices[].online` (a drone live on MQTT) — don't conflate them. Frontend twin is `frontend/src/types/fieldLink.ts` — keep the two in lockstep. Wayline shapes (#162): `FieldLinkWayline` (`id`, `mission_id`, `name`, `drone_model_key`, `payload_model_keys`, `favorited`, `username`, plus `create_time` / `update_time` as hub-reported epoch ms) and `FieldLinkWaylineListResponse` (`waylines`), the operator-facing view of the hub's wayline library (the hub's internal `object_key` is dropped at the service `_map_wayline`). Frontend twins `FieldLinkWayline` / `FieldLinkWaylineListResponse` in `frontend/src/types/fieldLink.ts`. Media-event shapes: `MediaEventCreate` (inbound from the *hub*, not the browser — `object_key`, `fingerprint`, device-reported `captured_at`, `position: PointZ | None`, `device_sn`, `raw_callback: dict | None` persisted verbatim) — two `field_validator(mode="before")` hooks harden it against dirty device data (#162): `captured_at` strips nul/control bytes and empties to `None`, and `raw_callback` is recursively nul-scrubbed via the module-private `_scrub_nul` because Postgres `jsonb` rejects nul bytes; a junk field degrades instead of 422ing. `DroneMediaFileResponse` (`status: MediaFileStatusStr`, the `Literal` alias mirroring `app.core.enums.MediaFileStatus`; carries `updated_at`, the reassignment audit trail). Frontend twin: `frontend/src/types/droneMedia.ts`.

Per-inspection upload (Phase 1): `UploadUrlRequest` (`filename`, `content_type`) / `UploadUrlResponse` (`object_key`, `upload_url`), `CompleteUploadRequest` (`mission_id`, `inspection_id: UUID | None`, `object_key`, `filename`, `size_bytes`), `MediaMoveRequest` (`inspection_id: UUID | None`, `order_index: int | None`), `MediaReorderRequest` (`ordered_ids: list[UUID]`), `InspectionMediaGroup` (`inspection_id`, `method`, `sequence_order`, `files`), `MissionInspectionMediaResponse` (`mission_id`, `mission_name`, `inspections`, `unassigned`), and `MediaViewUrlResponse` (`url` — the presigned GET url for `GET /drone-media/{id}/view-url`, #162). File entries reuse `DroneMediaFileResponse` from `field_link.py`, now extended with `inspection_id`, `order_index`, `origin` (`MediaOriginStr = Literal["HUB", "MANUAL"]`), `filename`, `size_bytes`, and a nullable `fingerprint`. Frontend twin: `frontend/src/types/droneMedia.ts`.

Both are derived from already-emitted columns and serialize on every response, so the frontend can branch the export UI without a second lookup. The constants table import is intentional — see invariant below.
- `measurement.py` — video-measurement DTOs for the two-step operator flow. `MeasurementResponse` is the full aggregate (refs, boxes, summaries, object-storage pointers + the operator `label`); `MeasurementUpdate` is the partial-update body for the rename route (`label: str | None`); `MeasurementStatusResponse` is the lightweight progress poll (`status` doubles as the phase, `error_message` only on ERROR); `MeasurementPreviewResponse` carries the first-frame presigned GET url + the detected/pre-placed boxes; `ConfirmLightsRequest` is the operator's confirmed boxes that start processing. `MeasurementListItemResponse` is one row of the airport-scoped Results list - status + `label` + mission context (`mission_id`, `mission_name`, added in #59 so the row spans every mission) + inspection context (`inspection_method`, `inspection_sequence_order`) + the rollup the list page routes on (`has_results`, `pass_count` / `fail_count`, `error_message`), all derived by `measurement_service._list_item` from the aggregate's `summaries` / `object_key` so the row needs no per-row results fetch. Nothing consumes the raw channels yet (deferred chart work); the frontend `LightSeriesPoint` in `frontend/src/types/measurement.ts` mirrors the four fields in lockstep. `has_results` stays false and the series empty until the run is DONE with a blob in object storage. Building blocks: `ReferencePointResponse` (one PAPI light's snapshotted ground truth), `LightBox` (percentage-coord box), `LightSummaryResponse` (per-light PASS/FAIL rollup). `MeasurementStatusStr = Literal[...]` mirrors `app.core.enums.MeasurementStatus`. The `measurement_service` package maps the `Measurement` ORM row onto these via `to_response` / `light_boxes_to_schema` (`measurement_service/_mapping.py`); the mapping is hand-rolled, so the route hands schema objects straight through with no `from_attributes` ORM coupling here.

## Invariants

- **Schemas import enums from `app.core.enums`**, not `app.models.enums` (that shim was removed). For literal unions, prefer the local `*Str = Literal["..."]` aliases that mirror the python enum values.
- **Constants** live in `app.core.constants` — `MIN_TRANSIT_ALTITUDE_AGL_M`, `DEFAULT_BUFFER_DISTANCE_M`, `MIN_VERTICAL_PROFILE_ANGLE_DEG` / `MAX_VERTICAL_PROFILE_ANGLE_DEG`, `DEFAULT_GLIDE_SLOPE_DEG`, `DEFAULT_LHA_TOLERANCE_DEG`, `EARTH_RADIUS_M`, `SAFE_IDENTIFIER_RE`, `DJI_WPML_ENUMS` (drone-model → WPML drone/payload enum tuple). Don't duplicate numeric defaults in field definitions; reference the constant. `DEFAULT_LHA_TOLERANCE_DEG` is shared with the ORM column default on `LHA.tolerance` and the service-side fallback in `bulk_generate_lhas`, so the wire default, persisted default, and runtime fallback cannot drift. `DJI_WPML_ENUMS` is the same constant the DJI exporter reads from — keeping it in `core` lets `DroneProfileResponse.supports_dji_wpml` derive its flag without the schemas → services import bug.
- **Schemas never import from `app.services` or `app.models`.** That would tie the wire format to internal types and create a circular import. Validators that need cross-cutting rules (e.g. transit-altitude min) re-implement the rule against `app.core.constants`.
- **Pydantic v2** — use `model_dump(exclude_unset=True)` for partial updates; `model_validate` (not v1's `parse_obj`) for inbound parsing.
- **Frontend parity**: literal unions in this package must match `frontend/src/types/enums.ts`. `backend/tests/test_enum_parity.py` enforces parity for `ConstraintType`, `MissionStatus`, `WaypointType` — extending parity is the right move when adding a new enum.

## Cross-package dependencies

- Imports from: `app.core` (constants, enums for non-literal use), `app.schemas` siblings (geometry, common), `pydantic`.
- Imported by: `app.api.routes` (request/response types), `app.services` (DTO in/out), tests.

## Gotchas

- **`MissionUpdate` partial updates** — services rely on `model_dump(exclude_unset=True)` to detect which fields were actually sent so they can decide whether to regress to DRAFT. Adding a default value to a field that should be "unset until provided" will silently break the regress logic; use `Field(default=None)` plus a sentinel or a `__pydantic_fields_set__` check instead.
- **Geometry round-trip**: inbound `PointZ`/`LineStringZ`/`PolygonZ` are converted to WKT strings in `app.services.geometry_converter` (`geojson_to_wkt`). Outbound, the response models accept the WKT string straight off the ORM column and parse it back into geojson via `app.core.geometry.wkt_to_geojson`. Don't try to expose raw geometry blobs.
- **Schema-only validators duplicate model-level rules** (e.g. `_validate_transit_altitude` mirrors `Mission.validate_transit_altitude`). That duplication is intentional to keep the layering rule — both must reference `MIN_TRANSIT_ALTITUDE_AGL_M` from `app.core.constants` so they can't drift.
- **`config` not `config_override`** in `InspectionCreate`/`InspectionUpdate` — the field was renamed; the override semantics live in nested `InspectionConfigOverride`.
- **`measurement_density` is floored at `ge=1`** on both `InspectionConfigOverride` (mission-attached overrides) and `InspectionConfigCreate` (template defaults in `inspection_template.py`). The schema rejects `0` and negatives; positive ints and `None` (inherit) pass. The orchestrator still defends against ORM hand-crafted writes that produce zero-waypoint passes - see `services/CLAUDE.md` - so the schema floor is a wire-level guard, not the only one.
- **`lha_selection_rules` is keyed by AGL UUID, not LHA UUID.** The dict shape is `dict[UUID, LhaSelectionRule]` on both `InspectionConfigOverride` and the template's `InspectionConfigCreate` / `TemplateConfigResponse`. The schema does not validate that the AGL ids are reachable from this inspection — that's the service's job (`apply_lha_selection` looks them up and silently drops unknown AGLs because the form should never produce a rule for one). RANGE params accept either `from` (wire) or `from_` (python) thanks to `populate_by_name`; the `model_validator` rejects `from > to` via the shared `common.validate_range_order(..., allow_equal=True)` (inclusive, so `from == to` passes).
- **`LHACreate.sequence_number` / `LHAUpdate.sequence_number` are `int | None = Field(default=None, ge=1)`.** `None` on create means "server picks `max + 1`"; an explicit value triggers a parent-locked sibling shift in `airport_service`. `LHAResponse.sequence_number: int` is always present. The schema does NOT enforce the upper bound (`<= count`) — the service does, because the count is per-AGL and the schema has no DB context. For PAPI parents the wire format is happy to accept either `unit_designator` (a letter A-D) or `sequence_number`; the service translates and rejects conflicting letter+seq pairs with 422. `LHACreate` / `LHAUpdate` / `LHAResponse` also carry `lens_height_msl_m` / `lens_height_agl_m` (`float | None`, PAPI-only); the schema does NOT enforce PAPI-only-ness (the form does), and the extractor handoff pre-fills them on an LHA seed.
- **`ExportRequest` enforces the runway-buffer dependency at the schema layer too.** A `model_validator` rejects `include_runway_buffers=true` without `include_geozones=true` (the same rule fires again in `export_service`, returning 400 instead of 422). `MissionResponse.supports_geozone_upload: bool | None` mirrors `DroneProfile.supports_geozone_upload` through the `Mission` proxy property; `None` means "no drone profile attached", not "incapable" — keep the tri-state when consuming it on the frontend. `ExportRequest.dji_heading_mode_override: DjiHeadingModeStr | None` carries an optional per-export override that wins for the export AND persists to `mission.dji_heading_mode` as the operator's last-used preference; only consumed by the KMZ / WPML generators, ignored elsewhere. Pydantic rejects unknown values at 422. `ExportRequest.acknowledge_altitude_clamps: bool = False` is the operator opt-in to ship a KMZ/WPML file whose DJI placemark altitudes were clamped to the takeoff reference; the schema does not validate the flag's meaning — `export_service` collects clamp records during generation and raises `DomainError(409, extra={"altitude_clamps": [...]})` when the list is non-empty and the flag is false. The 409 body's `detail` mirrors `ExportResponse` so the frontend can parse a blob-response error and surface the clamps.
- **`SafetyZoneUpdate` only inspects the inbound payload.** The `_validate_altitude_range` model validator rejects `type == AIRPORT_BOUNDARY` together with `altitude_floor` / `altitude_ceiling`, but partial patches that omit `type` cannot tell whether the persisted row is a boundary. `airport_service.update_safety_zone` re-checks against the merged `target_type` and additionally nulls any stale `altitude_floor` / `altitude_ceiling` when the resolved type is `AIRPORT_BOUNDARY`. Schema and service are deliberately layered - don't move the null-out into the schema (it has no DB context) and don't drop the schema check (it short-circuits obviously bad payloads before the service runs).
- **Pair-link request bodies live alongside `SurfaceUpdate`, not as a separate schema module.** `SurfaceCoupleRequest` and `SurfaceCreateReverseRequest` sit in `surface.py` so the wire types track the surface CRUD shape. Identifier override on create-reverse is `Field(default=None, min_length=1, max_length=10)` matching the column; bare-string overrides bypass the reciprocal derivation in the service. There is no dedicated decouple body — the route uses an empty `POST`.
- **VP angle band is rejected at the schema boundary.** `InspectionConfigOverride` and `InspectionConfigCreate` both run a `model_validator(mode="after")` that raises 422 on `angle_start >= angle_end` when both are supplied (both delegate to the shared strict `common.validate_range_order`, so the operator and the `"angle_start must be less than angle_end"` message stay byte-identical across the two schemas); `angle_start` / `angle_end` fields are `Field(default=None, ge=MIN_VERTICAL_PROFILE_ANGLE_DEG, le=MAX_VERTICAL_PROFILE_ANGLE_DEG)` so the `[1.0°, 16.5°]` envelope is enforced at the wire too. The bounds live in `app.core.constants` and both `inspection_config.py` and `inspection_template.py` import them directly; the `AngleSourceStr = Literal["PAPI", "CUSTOM"]` alias lives in `inspection_config.py` and is re-exported via `mission.py` (which `inspection_template.py` still imports from). The "PAPI requires every selected LHA to have a `setting_angle`" check is a 422 raised by `inspection_service`, NOT by the schema — the schema has no AGL/LHA context.
- **SURFACE_SCAN INTERVAL ordering is rejected at the schema boundary.** The `_check_scan_interval` `model_validator(mode="after")` lives on the shared `ScanConfigFields` mixin (#868), so it fires on ALL FOUR config classes — `InspectionConfigOverride` / `InspectionConfigCreate` AND both `*Response` shapes. It raises 422 when `scan_length_mode == "INTERVAL"` and `scan_length_from >= scan_length_to`, gated on the mode (FULL / MAX_LENGTH don't trip it), and inlines the comparison rather than going through `common.validate_range_order` because the check only fires for one mode. The responses inheriting the validator is benign — they build from already-validated DB rows — but two side effects are real and intentional: the interval check now runs on the response shapes too, and inherited fields sort first in Pydantic so `model_dump` / OpenAPI **key order** shifts (JSON clients are order-agnostic; the FE 47-key hydration test asserts key count, not order). The per-field bounds (`scan_length_from`/`to` `ge=0`, `scan_width`/`scan_height` `gt=0`, `scan_run_count` `ge=1`, `scan_sidelap_percent` `ge=0, le=80`, `scan_frontlap_percent` `ge=0, le=80`) ride the mixin and are enforced at the wire too; everything else (surface-belongs-to-airport, FOV-driven run count) is the service/compile layer's job.
- **`DroneProfileResponse.supports_dji_wpml` / `is_dji` are computed, not persisted.** Both `@computed_field`s serialize on every response so the frontend ExportPanel can branch the pre-export confirm modal (unmapped DJI vs non-DJI vs no drone) without a second lookup. `supports_dji_wpml` flips when a new drone model is added to `app.core.constants.DJI_WPML_ENUMS`; the response shape stays the same. The schema can read `DJI_WPML_ENUMS` directly because the constant lives in `core` — keep it there, don't move the table back into the DJI export service (that would require schemas to import services and break the layering).
- **`violation_kind` is the structured selector; message classification is the legacy-only fallback.** The orchestrator now persists `violation_kind` on every row, so consumers (PDF report, frontend panels) must branch on `violation_kind`, never re-parse `message`. The `_classify_violation(message)` keyword table and `_VIOLATION_KIND_RULES` (now in the private `_violation_classifier.py`, re-exported by `flight_plan.py`) exist solely to backfill `None` (rows written before the column existed); `constraint_name` is mapped from the resolved kind via `_CONSTRAINT_NAME_MAP`. Don't promote message parsing back to the primary path or move the classifier into a service — keeping it in the schemas package (stdlib-only, no service/model imports) is what bounds the regex coupling to a single legacy seam. The extraction into a private module did not change the layering: it is still schema-side, still the fallback, still re-exported from `flight_plan.py` for its consumers.
