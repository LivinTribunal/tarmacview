# api.routes package

## Purpose

FastAPI routers - the HTTP boundary. Routes parse requests, call services, attach audit-log rows for state-changing requests, commit, and serialize responses. They hold no business logic and never touch the ORM directly.

## Public API surface

One router per resource, registered in `app/main.py`:

- `auth.py` - login, logout, `me`. Issues JWTs via `auth_service`. Logs `LOGIN` / `LOGOUT`. `POST /refresh` rotates the refresh-token cookie on every exchange.
- `admin.py` - super-admin endpoints (user management, audit-log read/export, system settings). Logs `INVITE_USER`, `DEACTIVATE_USER`, `ASSIGN_AIRPORT`, `SYSTEM_SETTING_CHANGE`. `update_user`, `deactivate_user`, and `delete_user` pass `current_user.id` as the actor so `admin_service`'s self-guard and last-super-admin guard fire. `GET /audit-log` and `GET /audit-log/export` accept an optional `airport_id` query param that filters on `AuditLog.airport_id`. `GET /system-settings` is gated to any authenticated user (not just super-admin) so the coordinator UI can read shared toggles (`elevation_api_fallback_enabled`, `maintenance_mode`); for non-super-admin callers the admin-only fields `cesium_ion_token`, `elevation_api_url`, and `elevation_api_key` are blanked in the response so credentials don't leak through the widened read path. `elevation_api_key` is masked even for super-admin (sentinel string when set). `PUT /system-settings` stays super-admin only and, after `db.commit()`, invalidates the runtime cache for each key actually touched (`elevation_api_fallback_enabled`, `elevation_api_provider`, `elevation_api_key`) so a toggle / provider switch / key rotation takes effect on the next request. The `SYSTEM_SETTING_CHANGE` audit row redacts secret-bearing fields (`cesium_ion_token`, `elevation_api_key`) to `"***"` in `details`.
- `airports/` (package) - airport + child entities (surface / obstacle / safety zone / AGL / LHA), DEM upload, bulk drone change, per-point elevation lookup. `airports/__init__.py` owns the `prefix="/api/v1/airports"` / `tags=["airports"]` router plus the two empty-path collection routes (`GET`/`POST ""`), then `include_router`s the prefix-less sub-routers in source-block order: `core`, `terrain`, `photo_metadata`, `surfaces`, `obstacles`, `safety_zones`, `agls`, `lhas`. `main.py` still does `from app.api.routes.airports import router` - the package `__init__` resolves the same symbol, so registration is unchanged. Logs `CREATE` / `UPDATE` / `DELETE` against the model class names verbatim (`Airport`, `Surface`, `Obstacle`, `SafetyZone`, `AGL`, `LHA`, `TerrainDEM`). The airport-create route (`POST ""` on the package router) threads `creator=current_user` into `airport_service.create_airport` so coordinator creators are auto-assigned to the new airport - see the two-audit-row gotcha below. `POST /{airport_id}/extract-photo-metadata` (in `photo_metadata.py`) is a read-only multipart upload (NO audit row) that parses per-image GPS + PAPI lens height from EXIF/XMP and returns coordinates for the coordinator's "extract from image" seed flow; `CoordinatorUser`-gated via `check_airport_access`, caps at 50 images / 50 MB each, returns 403 when the coordinator isn't assigned to the airport (the frontend surfaces a distinct no-access message). `GET /{airport_id}/elevation?lat=&lon=&allow_api=` is a read-only lookup (no audit row) that returns the active provider's `(elevation, source ∈ {FLAT, DEM_UPLOAD, DEM_API, API})` for the operator's takeoff/landing placement and drag handlers; the optional `allow_api: bool = False` query param opts the lookup into the configured remote backend (used by the LHA placement flow), default keeps every other resolver on the bare DEM-or-flat tier. `OperatorUser` gate via `check_airport_access`, lat/lon ranges enforced at the schema boundary. Four terrain-affecting endpoints (`PUT /{airport_id}`, `POST /{airport_id}/terrain-dem`, `POST /{airport_id}/terrain-download`, `DELETE /{airport_id}/terrain-dem`) accept a `rewrite_existing: bool = True` query param that is forwarded to the service as `renormalize=`; for `PUT /{airport_id}` the flag only actually applies when `airport_service.elevation_fields_changed(body)` is true (the audit row carries `{"rewrite_existing": ...}` in `details` only in that case so the activity feed isn't polluted by no-op renames). Three pair-link endpoints share the existing surface router: `POST /{airport_id}/surfaces/{surface_id}/create-reverse` (creates the reciprocal direction and auto-couples), `POST .../couple` (couples two RUNWAY surfaces, primary side overwrites geometry), `POST .../decouple` (clears the pair link on both sides). Every pair write emits `paired_surface_id` in `log_audit` `details`; `delete_surface` emits a per-side audit row when the cascade picks up the partner.
- `missions/` (package) - mission CRUD, duplicate, validate, export, field-hub dispatch, status transitions, inspection child resources, mission report. `missions/__init__.py` owns the `prefix="/api/v1/missions"` / `tags=["missions"]` router plus the two empty-path collection routes (`GET`/`POST ""`), then `include_router`s the prefix-less sub-routers in source-block order: `core` (mission CRUD + duplicate + validate + export + dispatch + report + status transitions), `inspections` (add / reorder / update / delete inspection children), and `measurements` (read-only `GET /{mission_id}/measurements` - the mission-scoped results list, `OperatorUser` + `check_mission_access`, no audit; it imports `measurement_service` directly, not the four re-exported below, and returns `list[MeasurementListItemResponse]`). `main.py` still does `from app.api.routes.missions import router` - the package `__init__` resolves the same symbol so registration is unchanged. The service modules consumed by the `core` + `inspections` sub-routers (`mission_service`, `inspection_service`, `export_service`, `mission_report_service`) are re-exported at the package namespace so legacy `mock.patch("app.api.routes.missions.mission_service.X")` targets still resolve; tests that previously hung audit-failure assertions off the file-level `log_audit` now patch the submodule that actually owns the import (`from app.api.routes.missions import core as missions_route`).
- `flight_plans.py` - trajectory generation, plan retrieval, revalidation (`POST /{mission_id}/revalidate`), waypoint batch-update, transit insert/delete. T3 path. Flight plans cascade-delete with the mission (FK `ondelete=CASCADE`) and are replaced by the orchestrator on regeneration.
- `field_link.py` - three endpoints with different callers and gates. `GET /api/v1/field-link/status` (OperatorUser): proxies the field hub's internal status through `field_link_service`. Read-only, no audit row, no db session; never 500s over hub state - an unset `FIELDHUB_URL` or an unreachable/malformed hub returns the degraded `{hub_online: false, devices: []}` shape with HTTP 200 (the same shape leaves `connect_url` / `public_host` null). The export-panel chip polls it every 10 s, so keep it cheap. `GET /api/v1/field-link/ca-cert` (OperatorUser): downloads the local CA cert so Pilot 2 trusts the hub. Reads `settings.fieldhub_ca`, returns it via `FileResponse` (stdlib only, no new backend dependency) with media type `application/x-x509-ca-cert` and filename `fieldhub-ca.crt`; 404 when the path is unset or the file is missing. No audit row, no db session. `POST /api/v1/field-link/media-events` (`require_hub_secret`, NOT a user JWT - the caller is the field hub, not a browser): persists one hub-reported media arrival via `field_link_service.record_media_event`, idempotent on fingerprint, then runs `drone_media_service.match_media_file` on newly created rows (failure-safe - a matching error leaves the row `RECEIVED` for the listing sweep to retry). The audit row (`CREATE`, `entity_type="DroneMediaFile"`, `user_id=None` - a system action with no human actor) attaches only when a row was actually created; a repost returns the existing row as 201 with no second audit entry.
- `drone_media.py` - operator-facing drone-media surface (`/api/v1/drone-media`, `OperatorUser`-gated). `GET ""` returns mission-grouped media + the unassigned bucket; it commits without an audit row because the listing runs the lingering-`RECEIVED` matching sweep (a system retry, no human actor). `POST /confirm-ingest` (declared before the `{media_id}` sibling) marks a mission's rows `INGESTED` — idempotent, audits `STATUS_CHANGE` with `entity_type="DroneMediaFile"`, `entity_id=mission.id`, `details={mission_id, ingested_count}`, `airport_id=mission.airport_id`. `POST /{media_id}/assign` manually moves one file to a mission or (null `mission_id`) the unassigned bucket — audits `UPDATE` with the target mission in `details`; `airport_id` is the target mission's airport, null on a move to unassigned. Reassign-after-ingest surfaces as 409 from the model. **Per-inspection upload (Phase 1):** `POST /upload-url` mints a presigned PUT target (no row, no audit); `POST /complete-upload` records the manual row and audits `CREATE`; `PUT /{media_id}/move` reassigns a file to an inspection/position (audits `UPDATE`); `PUT /inspections/{inspection_id}/reorder` renumbers an inspection's media (audits `UPDATE`, `entity_id=inspection_id`); `DELETE /{media_id}` deletes one manual upload (204, audits `DELETE` with `details={object_key}`, then drops the stored object via `object_storage.delete_object` **after** the commit so a failed commit can't orphan a deleted-row reference — manual-only 422, ingested 409). The reorder route is declared on the drone-media router (no `/api/v1/inspections` router exists) and its `inspections/{id}` literal segment sits ahead of the `/{media_id}` siblings. The per-inspection read is `GET /api/v1/missions/{mission_id}/drone-media` (in `missions/core.py`, `check_mission_access`-gated, no audit).
- `measurements.py` - video-measurement endpoints (`OperatorUser`-gated), the HTTP front of the two-step measure flow. `POST /api/v1/inspections/{inspection_id}/measurement` creates/starts a run (snapshots refs, queues first-frame extract), `GET /api/v1/measurements/{id}` returns the full aggregate, `GET .../status` is the progress poll (status doubles as the phase), `GET .../preview` returns the first-frame presigned GET url + detected boxes, `POST .../confirm-lights` confirms the boxes and kicks off full processing. The two state-changing routes (`create`, `confirm-lights`) log `AuditAction.MEASURE` and, after `db.commit()`, hand work to the Celery worker via `measurement_service.enqueue_*`; `airport_id` for the audit row is resolved through `measurement_service.airport_id_for_inspection`. The domain->wire mapping lives in `measurement_service` (`to_response` / `light_boxes_to_schema`), so the route imports only `app.schemas` / `app.services` / `app.core` - no `app.domain` and never the ORM. `confirm-lights` passes the request's wire boxes straight to `measurement_service.confirm_lights`, which builds the domain boxes. Registered in `main.py`.
- `inspection_templates.py` - template CRUD (super-admin authoring).
- `drone_profiles.py`, `camera_presets.py` - reference data CRUD.

Cross-cutting helpers live in `app/api/dependencies.py`: `OperatorUser` / `CoordinatorUser` / `SuperAdminUser` (role gates), `check_airport_access`, `check_mission_access`, `get_user_airport_ids`, and `require_hub_secret` (gate for hub-to-backend internal endpoints: 503 while `FIELDHUB_SHARED_SECRET` is unconfigured, 403 on a missing/mismatched `X-Hub-Secret` header, constant-time compare, never logged - mirrors the hub-side gate). Routes consume them via `Depends`.

## Invariants

- **Routes call services, never models.** Importing from `app.models` outside of type hints is a layering bug.
- **Same-transaction audit guarantee.** Every state-changing route follows this exact shape: call the service (which `flush()`es), call `log_audit(...)` (which also flushes), then `db.commit()` once. The audit row and the mutation persist atomically; if either fails, both roll back. Don't `commit()` inside the service and audit afterwards - that closes the transaction before the audit row attaches and breaks the guarantee.
- **`AuditAction` matches the user-visible verb, not the HTTP method.** `POST /missions/{id}/validate` logs `VALIDATE`, not `STATUS_CHANGE`. `POST /missions/{id}/generate-trajectory` logs `GENERATE_TRAJECTORY`. `POST /missions/{id}/export` logs `EXPORT`. `POST /missions/{id}/dispatch` logs `DISPATCH`. The measurement create + confirm-lights routes log `MEASURE` (`entity_type="Measurement"`). Plain CRUD uses `CREATE` / `UPDATE` / `DELETE`. The full set lives in `app.core.enums.AuditAction`.
- **`entity_type` strings are model class names verbatim** (`Mission`, `Inspection`, `Airport`, `Surface`, `Obstacle`, `SafetyZone`, `AGL`, `LHA`, `TerrainDEM`, `FlightPlan`, `Waypoint`, `DroneMediaFile`). Don't lowercase or pluralize - the super-admin audit-log filter relies on the exact class name.
- **`entity_name` is set whenever the model has a human-readable identifier** (`Mission.name`, `Airport.name`, `AirfieldSurface.identifier`, `LHA.unit_designator`). `Waypoint` rows omit it.
- **Pass `airport_id` to `log_audit` whenever the action is scoped to a single airport.** Airport routes use `airport.id`; mission/flight-plan/waypoint routes read it off the eagerly-loaded `mission.airport_id`. Auth, super-admin user management, system settings, templates, drone profiles, and camera presets deliberately leave it `None` — `ASSIGN_AIRPORT` stays a single row per call rather than fanning out per airport. The denormalized column is what powers the per-airport activity feed; missing values disappear from that view.
- **`ip_address` is sourced from `request.client.host`**, gated by `if request.client else None`. Inject `request: Request` into the signature - don't rebuild the IP from headers.
- **Auth gate first, then service call, then audit, then commit.** The role-typed `Depends` (`OperatorUser` etc.) gates the request before the body executes; `check_mission_access` / `check_airport_access` enforces row-level scope. Never log audit rows for requests that can't proceed.
- **Routes own all `db.commit()` calls.** Services flush. The only exceptions are the trajectory `generate` route, which commits twice on purpose (once after `mark_computing` so the polling endpoint sees the in-progress state, again after the audit row), and the failure paths in the same route which roll back, refresh, and commit `mark_computation_failed` so the failed state survives.
- **HTTP errors come from `HTTPException`.** Catch `NotFoundError` / `DomainError` from services and re-raise as `HTTPException(status_code=e.status_code, detail=...)`. Never let a bare service exception bubble up.

## Cross-package dependencies

- Imports from: `app.api.dependencies`, `app.core` (config, enums, dependencies, exceptions), `app.schemas`, `app.services`, `app.utils.audit`. FastAPI primitives.
- Imported by: `app.main` (router registration) and tests only.

## Gotchas

- **Path ordering matters.** In `missions/inspections.py`, `PUT /{mission_id}/inspections/reorder` is declared before `PUT /{mission_id}/inspections/{inspection_id}` so "reorder" isn't parsed as a UUID. Keep static-suffix routes above their `{uuid}` siblings. The same rule spans sub-routers in both `airports/` and `missions/`: `__init__` includes the sub-routers in source-block order so FastAPI's in-order match stays byte-identical to the pre-split single file (airports: `GET /summary` in `core.py` must resolve before `GET /{airport_id}`; missions: the `core` block resolves before `inspections`, then `measurements` last - its read-only `/{mission_id}/measurements` suffix never shadows a `core` sibling). Don't reorder the `include_router` calls, and don't move the empty-path collection routes off the package router - a prefix-less sub-router can't carry an empty path through `include_router`.
- **`bulk_change_drone` emits a single audit row, not one per mission.** The row's `entity_type=Airport`, `entity_name=airport.name`, and `details={"drone_id", "mission_count", "mission_ids"}`. Tests assert this shape; don't fan out per-mission rows.
- **Coordinator airport creation emits two audit rows, one commit.** `POST /api/v1/airports` passes `creator=current_user`; the service auto-assigns a coordinator creator to the new airport, and the route records that as a second `ASSIGN_AIRPORT` row (`entity_type="User"`, `entity_id=current_user.id`, `details={"airport_ids": [...]}` - the airport rides in `details`, not `airport_id`, mirroring the admin assignment route) alongside the `CREATE` row before the single `db.commit()`. Super-admin creators get only the `CREATE` row - their airports stay unassigned by design and surface as orphaned in the admin overview. Tests pin both shapes (`test_create_airport_by_coordinator_logs_assign_audit` / `..._super_admin_logs_no_assign_audit`); don't fold the assignment into the `CREATE` row's details.
- **`bulk_generate_lhas` emits a single aggregate audit row.** `entity_type=LHA`, `entity_id=agl_id`, `airport_id=airport.id`, and `details={"airport_id", "surface_id", "agl_id", "count", "lha_ids"}`. The service `flush()`es so the route attaches the audit row and commits once - same atomicity guarantee as every other write path. Don't emit one row per generated LHA.
- **`reorder_inspections` audit details carry `{"reordered": [<inspection_id>, ...]}`** - the new sequence as strings. The route serializes UUIDs explicitly because JSON columns don't accept raw `UUID` instances.
- **`generate_trajectory` uses `GENERATE_TRAJECTORY`, not `CREATE`.** The trajectory orchestrator persists the `FlightPlan` via `flush()` only; this route's `db.commit()` after `log_audit` covers both the plan persistence and the audit row in one transaction.
- **`insert_transit_waypoint` raises 500 if it can't locate the new waypoint** in the response by `sequence_order`. The earlier behavior emitted `entity_id=None`, which produced unanchored audit rows; the explicit failure is intentional.
- **Terrain DEM upload/download own filesystem cleanup.** Both `POST /airports/{id}/terrain-dem` (upload, in `terrain.py`) and `POST /airports/{id}/terrain-download` keep `log_audit` + `db.commit()` inside the same `try` that owns the freshly-written GeoTIFF: if the audit insert or commit raises, the `except` branch unlinks the new file via `cleanup_path` and re-raises. `airport_service.upload_terrain_dem` returns `(airport, old_dem_path)` and the route only `os.unlink`s the *previous* file after `db.commit()` succeeds. Don't hoist the audit/commit out of the try block (a failed audit/commit would otherwise leave the new GeoTIFF orphaned on disk) or unlink the old DEM before the commit.
- **`generate` route's three commits are deliberate.** First commit publishes `mark_computing` for the polling endpoint. Pipeline runs. On failure the route rolls back, refreshes the mission, calls `mark_computation_failed`, and commits again so the error state is durable. On success it `mark_computation_completed`, audits, and commits. Don't collapse the first commit into the success path - the polling endpoint needs to see `COMPUTING` mid-flight.
- **`revalidate` deliberately skips the audit row and commits inside the service.** `POST /{mission_id}/revalidate` re-runs the safety pipeline against the persisted flight plan; it never touches `Mission.status`, never regenerates waypoints (UUIDs and lon-lat-alt stay byte-identical), and only swaps the `ValidationResult` row (cascade delete + insert). The single `db.commit()` lives in `flight_plan_service.revalidate_flight_plan` so the function can return the enriched `FlightPlanResponse` after reload. No `log_audit` call - this is the documented exception to the "routes own commits" invariant. Do not add an audit row here without first deciding what `AuditAction` would mean for a no-op revalidation. Returns 409 when the mission has no flight plan, 404 when the mission is missing.
- **Super-admin self-protection lives in the service.** The route just forwards `current_user.id` as the actor; `admin_service` raises `DomainError("cannot deactivate your own account")`, `cannot change the role of your own account`, `cannot delete your own account`, or `cannot remove the last active super admin`, all of which surface as HTTP 400. Don't open-code the same checks in the route - the service is the single point of truth so future non-HTTP callers inherit the invariant.
- **Pair-link audit details always carry `paired_surface_id`** when the surface is coupled at the time of the write. `update_surface` adds it after-the-fact (read off the refreshed `surface.paired_surface_id`). `delete_surface` emits two `DELETE` rows when the cascade picked up the partner — the second row's `details` includes `paired_surface_id` pointing at the originally-targeted surface so the trail is symmetric. `couple` / `decouple` rows include `operation` ("couple" or "decouple") and, for couple, `primary` ("self" or "target") so the audit log shows which side won the geometry overwrite. Don't strip these fields — the super-admin activity feed renders them.
- **`export_mission` transitions VALIDATED -> EXPORTED inside `app.services.export`** before the route logs `EXPORT` and commits. The route imports the package as `from app.services import export as export_service` so the call shape (`export_service.export_mission(...)`) stays identical to the pre-split layout. The service returns `(files, safe_name, altitude_clamps)`; the route discards `altitude_clamps` after counting it into the audit row, since a non-empty list with `acknowledge_altitude_clamps=False` already raised 409 inside the service before status changed. The transition flushes; the route's commit makes both the status change and the audit row atomic. The `EXPORT` audit row's `details` carry `{"formats", "include_geozones", "include_runway_buffers", "dji_heading_mode_override", "acknowledge_altitude_clamps", "altitude_clamps_count"}` — the geozone flags, the heading-mode override, and the clamp ack flag are recorded even when false / null / 0 so the audit log shows the operator's choice. The service additionally writes `mission.dji_heading_mode = override` as a side effect when the override differs from the persisted column (last-used preference); no separate UPDATE audit row is emitted. Don't drop any of these fields.
- **The export route never commits on the 409 altitude-clamp path.** When `export_mission` raises `DomainError(status_code=409, extra={"altitude_clamps": [...]})` because the KMZ/WPML pass found below-takeoff placemarks and `acknowledge_altitude_clamps=False`, the route's `log_audit` + `db.commit()` lines never execute — the exception propagates through the `DomainError` → `HTTPException` mapper in `app.utils.errors`, which serializes `detail = {message, extra}` so the frontend can read `response.data.detail.altitude_clamps`. The session rolls back at request teardown, so the prior `mission.transition_to(EXPORTED)` and `mission.dji_heading_mode = override` flushes inside the service do NOT persist; the mission stays VALIDATED and the operator's heading-mode pick is not stored. This is the same atomicity guarantee every other write path uses — don't try to log a partial audit row before re-raising, and don't reorder the service call below `log_audit`.
- **The dispatch route shares the export route's no-commit-on-failure shape.** `POST /missions/{id}/dispatch` (in `missions/core.py`) calls `wayline_dispatch_service.dispatch_mission`, which routes through `export_mission(["KMZ"])` — so the 409 altitude-clamp `DomainError` propagates exactly like the export route's, and a 502 (hub unconfigured/unreachable) propagates the same way: in both cases `log_audit` + `db.commit()` never run and the VALIDATED → EXPORTED transition plus the dispatch-row upsert roll back. The `DISPATCH` audit row carries `{"wayline_id", "acknowledge_altitude_clamps"}` in `details` and the mission's `airport_id`. The optional request body is `DispatchRequest | None` — a bare `POST` with no body is valid and means `acknowledge_altitude_clamps=False`.
- **Delete-route shape: read entity_name BEFORE the service delete call.** `delete_template` / `delete_drone` follow `service_get(...)` → capture `entity_name` (and any return value like `warnings`) into a local → `service_delete(...)` (flushes the delete) → `log_audit(...)` with the captured name → `db.commit()` once. This is the canonical service → audit → commit shape every other write path uses. The earlier inverted shape (audit-then-delete) only worked because the service committed internally and fanned both rows in one transaction; now that services flush and the route commits, accessing `template.name` after the delete mixes attribute access with a session that just deleted the row, so capture up front.
- **Measurement routes enqueue Celery work AFTER the commit, never before.** `create_measurement` and `confirm_lights` follow service flush → `log_audit` → `db.commit()` → `measurement_service.enqueue_*(measurement.id)`. The enqueue is the last line on purpose: the worker reads the row by id on its own session, so handing the task to Celery before the row commits would race the worker against an uncommitted insert (task picks up a row that isn't there yet). Don't move the enqueue above `db.commit()`. The enqueue seam is a lazy import inside `measurement_service`, so importing the route never pulls in `celery`.
