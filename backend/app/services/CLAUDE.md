# services package

## Purpose

Holds all business logic. Routes call into services; services orchestrate models, value objects, and external systems. Pydantic schemas come in as DTOs and ORM models go out ŌĆö services own the database session and transactional boundaries.

## Public API surface

- `mission_service.py` ŌĆö `create_mission`, `update_mission`, `delete_mission`, `duplicate_mission`, `transition_mission`, `list_missions`, `get_mission`. Update is the regress-on-trajectory-change entrypoint.
- `inspection_service.py` ŌĆö `add_inspection`, `update_inspection`, `delete_inspection`, `reorder_inspections`. All four use `mission.modify_inspections(...)` so trajectory invalidation stays atomic with the mutation. Inbound config payloads pass through `lha_selection.apply_lha_selection` before they hit the ORM so `lha_selection_rules` is normalized and `lha_ids` reflects the resolved union. SURFACE_SCAN saves additionally validate that the chosen `scan_surface_id` belongs to the mission's airport (422 otherwise).
- `inspection_template_service.py` ŌĆö template CRUD + AGL/method compatibility checks (`METHOD_AGL_COMPAT`). Default-config payloads also pass through `lha_selection.apply_lha_selection` so templates can carry helper-mode rules alongside an explicit `lha_ids`. The AGL-agnostic template list filter is generalized (HOVER_POINT_LOCK + SURFACE_SCAN both carry zero AGL targets), and bulk-create seeds a standalone "Surface Scan" template row ŌĆö its zero-target validation rides the existing `validate_method_agl_compat` short-circuit.
- `lha_selection.py` ŌĆö `apply_lha_selection(db, config_data)` mutates an inbound config dict in place: normalizes `lha_selection_rules` to JSONB-friendly shape, loads every affected AGL (the rules dict plus every AGL that owns any pre-existing `lha_ids` entry), runs `resolve_rules_to_lha_ids`, and overwrites `lha_ids` with the resolved union. `resolve_rule(rule, agl, surface)` evaluates a single rule (ALL / RANGE / FROM_THRESHOLD / CUSTOM); the FROM_THRESHOLD path uses an equirectangular along-track projection anchored at the chosen runway endpoint. Mirrored by `frontend/src/utils/resolveLhaSelection.ts`.
- `airport/` ŌĆö package; airport, surface, obstacle, safety-zone CRUD; cascades through WKT geometry strings + Shapely operations. Was the 2171-line `airport_service.py`, split by aggregate in #596; `airport_service.py` is now a thin re-export shim, so `from app.services import airport_service` and every `airport_service.<symbol>` reference stays byte-identical ŌĆö the symbol names below are unchanged, only the file you open to edit them moved. The route passes `creator=current_user` and records the auto-assignment as a second `ASSIGN_AIRPORT` audit row. Owns the per-airport altitude renormalize protocol (`altitude.renormalize_airport_altitudes(db, airport_id)`) which re-runs `provider.get_elevation` against every obstacle ring, AGL/LHA position, `mission.takeoff/landing_coordinate`, and every flight-plan waypoint's persisted `agl` + `camera_target_agl` for the airport (the waypoint pass delegates to `flight_plan_service._refresh_persisted_agl` so the ground-level waypoint set stays in one place; it does NOT call `mission.invalidate_trajectory` - per-WP AGL is display-only and the takeoff/landing alt branch already owns regression semantics for a real terrain shift). Fires automatically from `update_airport`, `upload_terrain_dem`, and `delete_terrain_dem` when their `renormalize: bool = True` kwarg is left at the default, and returns a `dict[str, list[UUID]]` of per-entity-type skip ids (`obstacles`, `agls`, `lhas`, `missions`, `waypoints`). `elevation_fields_changed(schema)` (single source of truth used by both the route and `update_airport`) returns true iff the inbound `AirportUpdate` touches `elevation` / `terrain_source` / `dem_file_path`; the audit `rewrite_existing` flag and the actual renormalize trigger stay in lockstep through this helper. `_normalize_position_altitude` is the single-write entrypoint used by AGL / LHA / obstacle create + update paths (constructs a fresh `create_elevation_provider`-built provider per call; LHA call sites pass `allow_api=True`, obstacle / AGL stay on the bare DEM-or-flat default; bulk callers reuse one instance). Update paths short-circuit before constructing a provider when the inbound `(lon, lat)` matches the stored value at 7 dp (~1 cm): `update_agl` / `update_lha` route through `_position_unchanged(schema.position.coordinates, _stored_point_coords(row.position))` and keep `position[2]` at the stored z (the inbound `alt` is ignored ŌĆö matches the "leave existing position.alt untouched" contract). `update_obstacle` routes its polygon through `_renormalize_boundary_with_stored(submitted, stored_wkt, airport)`, which uses `_ring_diff` to compute the set of moved vertex indices, copies stored z onto unchanged vertices byte-for-byte, and resamples only the moved indices via `_normalize_boundary_altitude(..., indices=moved)`. `_normalize_boundary_altitude` accepts an `indices: Iterable[int] | None` filter ŌĆö `None` preserves the create-path whole-ring default, an empty iterable short-circuits before any provider is built. `update_safety_zone` does NOT normalize ring z today, so the optimization is a no-op there. The renormalize loop opens two providers ŌĆö a bare one (`allow_api=False`) for the obstacle / AGL / mission-coordinate / waypoint sweep so the resample does not fan out remote-elevation calls per entity, and a separate `allow_api=True` provider used only for the LHA pass. `get_elevation_at_point(db, airport_id, lat, lon)` returns `(elevation, source_label)` for `GET /airports/{id}/elevation`. Also owns the LHA `sequence_number` reorder protocol: `create_lha` / `update_lha` / `delete_lha` / `bulk_generate_lhas` lock the parent AGL row with `SELECT ... FOR UPDATE`, park the moving (or newly-inserted) row at a `max + 1` sentinel, shift siblings by ┬▒1, then call `_apply_papi_invariant` to relabel PAPI letters from sequence (1=A, 2=B, 3=C, 4=D). Also owns the runway pair-link write paths: `couple_surfaces`, `decouple_surfaces`, `create_reverse_surface`, plus `update_surface` / `delete_surface` cascade-through-pair behavior. Pair invariants (symmetric link, reversed centerline, swapped threshold/end, reciprocal heading, equal length/width/buffer/boundary) are enforced via `_load_paired_for_update` (`SELECT FOR UPDATE` on the partner), `_propagate_pair_geometry`, `_overwrite_pair_geometry`, `_reverse_linestring_wkt`, `_reciprocal_heading`, `_recompute_agls_distance_from_threshold` (refreshes both sides' AGL distances after threshold/end edits), and `_reciprocal_runway_identifier` (01Ōåö19, 09LŌåö27R, C stays C).

## Invariants

- Services own the `Session`. Models never call `db.add` / `db.delete` / `db.flush`.
- **Services flush, routes commit.** Every state-changing service entrypoint that a route calls ends in `db.flush()` (and `db.refresh()` if the route needs the persisted ids), never `db.commit()`. The route then attaches an audit row via `log_audit` and calls `db.commit()` once - mutation and audit row land atomically. This is the same-transaction audit guarantee; tests under `test_audit_coverage.py` monkeypatch `log_audit` to raise and assert the mutation rolls back. The trajectory orchestrator follows the same rule: `generate_trajectory` flushes the persisted `FlightPlan` and lets the route commit. Documented exceptions: background / non-route paths (`seeder.py`); `flight_plan_service.revalidate_flight_plan`, which commits inside the service because the route attaches no audit row and the function reloads + returns the enriched response in one call; `flight_plan_service._backfill_waypoint_agl`, which commits the one-shot lazy AGL backfill on a read so the next read costs zero provider calls (mirrors `revalidate_flight_plan`'s read-path-commit shape); and the `measurement_service` worker runners (`run_first_frame` / `run_processing` / `_mark_failed`), which run off-request in the Celery worker and own their own commits so the polling endpoint sees each phase transition land ŌĆö the route-facing measurement entrypoints (`create_measurement` / `confirm_lights`) still flush-only.
- **Keep-stale flight plan contract**: `Mission.invalidate_trajectory` and `Mission.modify_inspections` deliberately do NOT delete `mission.flight_plan` ŌĆö the frontend renders the previously computed plan as a stale reference until a fresh recompute lands. Do not re-introduce a `delete_flight_plan_if_exists` helper around these calls. Mission deletion still wipes the row via the DB-level CASCADE on `flight_plan.mission_id` (`flight_plan` relationship uses `passive_deletes=True`).
- Routes call services. Services may call other services. Services never call routes.
- Status changes go through `mission.transition_to()` ŌĆö never assign `mission.status = ...` directly.
- Trajectory-affecting field updates use `mission.regress_if_trajectory_changed(data)`; inspection mutations use `mission.modify_inspections(callback)`. Don't open-code the regress-and-invalidate dance, and don't set `has_unsaved_map_changes` directly ŌĆö `invalidate_trajectory` owns that flag.
- `export.export_mission` is the only place that transitions VALIDATED ŌåÆ EXPORTED. `wayline_dispatch_service.dispatch_mission` reaches the transition *through* `export_mission` ŌĆö don't add a second transition site there.
- Use enums from `app.core.enums`, not the legacy `app.models.enums` re-export (it's gone).
- Magic numbers (cruise floor, default buffer, unit conversions like `METERS_PER_FOOT` / `METERS_PER_NM` / `METERS_PER_KM`, the openaip nearby-search radius, the VP angle envelope `MIN_VERTICAL_PROFILE_ANGLE_DEG` / `MAX_VERTICAL_PROFILE_ANGLE_DEG`, the PAPI fallback `DEFAULT_GLIDE_SLOPE_DEG`, the WGS84 `EARTH_RADIUS_M`, the filesystem `SAFE_IDENTIFIER_RE`, and the named float-comparison epsilons `COST_COMPARISON_EPSILON` (heading-optimizer tie-breaker, `1e-9`), `NEGLIGIBLE_OFFSET_M` (fly-over horizontal-offset skip threshold, `1e-9`), `DETERMINANT_DEGENERACY_EPSILON` (pathfinding 2x2 cross-product near-zero guard, `1e-15`)) live in `app.core.constants` ŌĆö import, don't redefine. Two of those epsilons share the value `1e-9` today; they stay separate constants because the names encode three different questions (deterministic tie-break vs sub-mm physical skip vs numerical-degeneracy guard) and collapsing them would let a future tweak silently change unrelated code. `app.utils.geo` and `app.utils.local_projection` re-export `EARTH_RADIUS_M` for backwards compatibility, but new code should import from `app.core.constants` directly. Geodesic helpers (`point_at_distance`, `bearing_between`, `distance_between`, plus `linestring_length` for whole-polyline length - import it for multi-segment paths, don't re-roll the per-segment `distance_between` loop) and the EGM96 / HAE conversion (`egm96_undulation`, `msl_to_hae`, used by the DJI KMZ export for the `takeOffRefPoint` z and the template `ellipsoidHeight`) live in `app.utils.geo` - same rule.
- **WKT-to-tuple extraction routes through `app.core.geometry`.** Use `point_lonlatalt(wkt) -> (lon, lat, alt)` (strict ŌĆö raises `ValueError` on empty/None/non-Point; a missing waypoint position is a data bug, not a `(0, 0, 0)` Null Island fallback), `polygon_xy(wkt | None)` (exterior ring only, `[]` on empty/None), and `linestring_xy(wkt | None)` (`[]` on empty/None). Don't reintroduce per-service `_extract_coords` / `_extract_polygon_coords` / `_extract_line_coords` shims; they were removed and the seam consolidated into `app.core.geometry`.
- **Obstacle buffer resolution is centralized.** Every Shapely-based site (per-pass safety validation, visibility graph edges, reroute search radius, vertex extraction, fast-path segment checks, inter-pass transit pre-flight) goes through `resolve_obstacle_buffer(obs, override)`. Priority chain: caller `override > 0` ŌåÆ `obstacle.buffer_distance > 0` ŌåÆ `DEFAULT_OBSTACLE_RADIUS` (15 m, defined in `trajectory/types.py`). A zero or `None` override means "no caller-level override", not "zero buffer". Don't reintroduce inline `obs.buffer_distance or X` fallbacks.
- **Surface buffer is baked into `LocalSurface.polygon`.** `app.utils.local_projection` inflates each surface polygon by `surface.buffer_distance` when the planner builds `LocalSurface`, so the runway-crossing penalty in the visibility graph and Shapely-based crossing checks see the buffered no-go region directly. Perpendicular candidate-node spacing in the shared `pathfinding._surface_edge_nodes` helper (called by `_collect_graph_nodes_in_circle` per-pass and `_surface_edge_nodes_in_region` inter-pass) adds the same `buffer_distance` to its `half_w` (`width/2 + buffer_distance + vertex_buffer_m`) so nodes land outside the buffered region, and the grid fill in both `_collect_graph_nodes_in_circle` and `_grid_fill_in_region` appends every `LocalSurface.polygon` to `exclusion_polys` so no grid node lands inside the buffered runway/taxiway region. Don't re-add the buffer at the use site - it would double-count and gum up tight corridors. Don't drop surfaces from the grid-fill exclusion set either - omitting them let grid nodes land inside the buffered runway/taxiway region.
- **Per-pass A* and `compute_inter_pass_transits` share node sources.** Both code paths feed their visibility graph from the same helpers: surface edge nodes (centerline ┬▒`half_w` at `SURFACE_NODE_SPACING`), perpendicular crossing nodes from `_runway_crossing_node_pairs` for the from->to line so A* always has a short crossing edge alongside any go-around, and `GRID_NODE_SPACING` grid fill (`_grid_fill_in_region` for the inter-pass case) clipped to the dilated unified hull. If you add a new node source to one path, mirror it in the other - a hull-vertex-only graph silently loses crossing edges on long runways. There is now exactly one inter-pass A* call site (`_assemble_core` -> `compute_inter_pass_transits`) shared by every scope. The surface-edge ┬▒`half_w` walk is a single shared helper `_surface_edge_nodes(surfaces, contains_fn)` - `_collect_graph_nodes_in_circle` calls it with a search-circle predicate, `_surface_edge_nodes_in_region` with a dilated-hull predicate - so the `half_w` / `num_points` spacing is structurally locked across the per-pass and inter-pass paths and can no longer drift by hand. The remaining shared sources (perpendicular crossing nodes, grid fill) still need manual mirroring across the two paths if you add a new one; `compute_transit_path`'s bookend invocations consume the same per-pass helper.
- **`RUNWAY_CROSSING_PENALTY_PER_METER = 15`** is calibrated to the buffered surface footprint, not arbitrary. A perpendicular crossing through the buffered region (~75 m incl. `vertex_buffer` each side) costs ~1125 m equivalent: detours under ~1 km still beat crossing, longer perimeter walks lose. Parallel crossing on a 3700 m runway costs ~55500 m equivalent - effectively forbidden. Bumping this without first guaranteeing crossing nodes exist in both code paths is what makes `compute_inter_pass_transits` walk the perimeter on long runways.
- **`BOUNDARY_EGRESS_PENALTY_PER_METER = 15.0`** (`trajectory/types.py`, deliberately co-equal with `RUNWAY_CROSSING_PENALTY_PER_METER`) is the per-mission keep-inside-airport-boundary nudge. The orchestrator threads `mission.keep_inside_airport_boundary` plus `local_geoms.boundary_zones` into `_build_visibility_graph`, `compute_transit_path`, `compute_inter_pass_transits`, `resolve_inspection_collisions`, and `revalidate_existing_plan`. When the flag is off the branch is skipped and the graph is byte-identical to the no-preference shape; when on, every meter of an edge lying outside the boundary union adds the penalty. This is an *edge-cost* contributor, NOT a node source - unlike the runway-crossing nodes it does not need mirroring across the per-pass / inter-pass paths. Both fast paths (`compute_transit_path` and `_try_fast_inter_pass_transit`) fall through to A* when a direct line exits the boundary (`_segment_exits_airport_boundary`, the rename of `_segment_violates_boundary_preference`). The common mission shape (operator + LHAs + takeoff/landing all inside the controlled-airspace polygon) makes the branch inert: the penalty fires but adds zero outside-length, so the flight plan stays byte-identical and the nudge only bites when a transit actually has to leave. The three-way `AirportBoundaryPreference` enum (`IGNORE`/`PREFER_INSIDE`/`PREFER_OUTSIDE`) was collapsed to this boolean - `PREFER_OUTSIDE` zigzagged because the sparse visibility-graph hull never encloses the outside-of-boundary corridor; re-adding an outside preference needs boundary-polygon vertices in the graph plus an extended hull (recorded future work).
- **Cruise clearance is 2.5-D for obstacles, 2-D for hard zones.** `pathfinding._check_cruise_clearance` runs after the cruise-altitude rewrite and only raises when a segment both intersects an obstacle footprint AND its altitude band overlaps `[base_alt, base_alt + height]`. A 5 m fence under a 30 m cruise no longer aborts. Hard safety zones (`HARD_ZONE_TYPES`) keep the 2-D semantics because they aren't modeled with a height. Per-pass validation in `safety_validator` is unrelated and unchanged.
- **Final-assembled buffer monotonically tightens.** The orchestrator tracks the per-pass buffer each `validate_inspection_pass` actually used and feeds `_compute_final_buffer(buffers_used) = max(buffers_used, default=settings.vertex_buffer_m)` into `compute_inter_pass_transits`, both bookend transits, and the final assembled validation. A pass cleared at `X` is never re-validated at a smaller value. Final hard-violation messages must report this effective buffer, not the per-pass one.
- **`settings.vertex_buffer_m` aliases `DEFAULT_BUFFER_DISTANCE_M`.** The env-override seam (`Settings.vertex_buffer_m`) and the persisted-column default on `AirfieldSurface` / `Obstacle` are seeded from the same constant in `app.core.constants` so they cannot drift.
- **Privileged user mutations are actor-aware.** `admin_service.deactivate_user`, `update_user`, and `delete_user` take an `actor_id` (the calling super admin's id) and route through `_assert_not_self` plus `_assert_not_last_super_admin` so a super admin cannot deactivate, demote, or delete their own account, and cannot strip the system of its last active super admin. The guards live at the service layer so future callers (CLI, scripts, new routes) inherit the invariant ŌĆö don't move them up into the route. The route gate plus `_assert_not_self` make the last-super-admin path unreachable through HTTP today; the service-level test exercises it directly so the invariant survives if a future caller bypasses the self-guard.
- **Invitation tokens are credentials.** `deactivate_user` nulls out `invitation_token` and `invitation_expires_at` so a previously issued setup/reset link cannot re-activate the account. `setup_password` and `reset_password` both check `is_active` *and* token validity; `admin_service.reset_password` refuses to mint a fresh token for an inactive user (403). If you add a new flow that consumes an invitation token, mirror the `is_active` gate.
- **LHA selection rules are resolved at write time, not at read time.** `apply_lha_selection` mutates the inbound dict so the canonical `lha_ids` array stays the single source of truth ŌĆö the trajectory pipeline reads `lha_ids` only and never sees `lha_selection_rules`. `resolve_rules_to_lha_ids` always feeds the resolver every AGL that owns any pre-existing `lha_ids` entry, so a partial rules dict (covering only the AGLs the form actually touched) cannot silently drop selections on untouched AGLs. CUSTOM-mode AGLs and AGLs absent from the rules dict preserve their portion of the pre-existing `lha_ids` (intersected with the AGL's lhas).

## Gotchas

- **Trajectory pipeline (T3)**: anything under `trajectory/` is `harness.config.json` Tier 3. Changes need extra test coverage and a human reviewer.
- **`trajectory/methods/` re-export is the stable seam - but mock where it's used.** `methods/__init__.py` re-exports every `_prepare_*` / `_*_handler` from `_prepare.py` / `_dispatch.py` by name (the `from ._prepare/_dispatch import ŌĆ”` lines bind them in the package namespace; they are then referenced by `METHOD_SPECS`, so no `__all__` sync list is needed - that was removed in #862), so production import paths survive the split unchanged. `unittest.mock.patch` targets do NOT: a `_prepare_*` body resolves helpers (`get_threshold_position`, `determine_start_position`, etc.) in the `methods._prepare` namespace, so `@patch("app.services.trajectory.methods.get_threshold_position")` patches a name nothing reads and is a silent no-op. Patch `app.services.trajectory.methods._prepare.get_threshold_position` instead (mock-where-it's-used); same rule for handler-internal symbols, which bind in `methods._dispatch`. Add new methods with a single `MethodSpec(method, prepare, handler, is_papi_glide_slope=ŌĆ”)` entry in `METHOD_SPECS` plus the matching prepare + handler - `PREPARE_REGISTRY` / `METHOD_REGISTRY` / `_PAPI_GLIDE_SLOPE_METHODS` all derive from that list, and prepares take `(ctx: MethodContext)` while handlers take `(ctx, prep)` (no per-method signature edits).
- **`update_inspection` ordering**: apply config-blob updates first, then wrap the scalar dict update inside `modify_inspections(lambda: apply_dict_update(inspection, data))`. The `modify_inspections` wrapper is what guarantees trajectory invalidation runs atomically with the mutation ŌĆö bypassing it desyncs the regress.
- **Rules-without-lha_ids partial update**: `update_inspection` and `update_template` both reseed `config_data["lha_ids"]` from the existing config when the caller sent `lha_selection_rules` but no `lha_ids`. Without that backfill, CUSTOM-mode AGLs would resolve against an empty custom set and lose their saved selections.
- **`reorder_inspections`**: reorder logic runs inside the `modify_inspections` callback so trajectory invalidation and sequence updates commit atomically.
- **`duplicate_mission`**: delegates to `Mission.duplicate()` which returns a *detached* aggregate. The service is responsible for `db.add(copy)` and the flush/refresh; the route commits.
- **`airport_service.upload_terrain_dem` returns `(airport, old_dem_path)`**, not just the airport. The route uses the returned path to `os.unlink` the previous DEM only after `db.commit()` succeeds - rolling the disk delete forward of the commit risks orphaning a filesystem mutation if the audit row fails to insert. The service runs `renormalize_airport_altitudes` after the source swap by default so existing obstacle / AGL / LHA / mission-coord altitudes resample against the new DEM. `delete_terrain_dem` is symmetric - snaps positions back to whatever the post-deletion FLAT / API-fallback provider returns. Both, plus `update_airport`, accept `renormalize: bool = True`; callers (route or test) pass `renormalize=False` to skip the per-entity resample (the "Rewrite existing structures" opt-out). When skipped, existing entities keep their old ground references and non-DRAFT missions stay at their persisted status - the operator must re-compute manually. Default-on inherits the renormalize's status-regression semantics for affected non-DRAFT missions.
- **`renormalize_airport_altitudes` regresses non-DRAFT missions only on a real alt change.** Every mission of the airport is scanned. For each non-terminal mission, takeoff/landing alts are compared against the resampled ground; a no-op rewrite preserves status, a real shift on a `PLANNED` / `VALIDATED` / `EXPORTED` mission calls `Mission.invalidate_trajectory()` (which regresses to DRAFT, flips `has_unsaved_map_changes`, resets computation status, and keeps the stale flight plan as a reference). Use `invalidate_trajectory()` here, not `regress_to_planned()` - the latter only handles `VALIDATED`/`EXPORTED` ŌåÆ `PLANNED`. `COMPLETED` / `CANCELLED` missions are skipped entirely.
- **Status-set gates reuse the class-level frozensets on `Mission`.** Every status-set guard in this package routes through one of `Mission.TERMINAL_STATUSES`, `Mission.NON_DRAFT_WITH_PLAN_STATUSES`, `Mission.POST_PLAN_STATUSES`, `Mission.PRE_EXPORT_EDITABLE_STATUSES`, or `Mission.PRE_PLAN_STATUSES` (defined on the model ŌĆö see `backend/app/models/CLAUDE.md` for the contents). Current sites: export gate (`export/orchestrator.py` uses `POST_PLAN_STATUSES`), trajectory generation gate (`trajectory/orchestrator/` uses `POST_PLAN_STATUSES` for auto-regress and `PRE_PLAN_STATUSES` for the can-generate check), waypoint write paths in `flight_plan_service.py` use `PRE_EXPORT_EDITABLE_STATUSES`, the bulk-drone-change SQL filter (`airport/core.py`) and the renormalize regression loop (`airport/altitude.py`) use `PRE_PLAN_STATUSES` and `NON_DRAFT_WITH_PLAN_STATUSES` / `TERMINAL_STATUSES` respectively. `MissionStatus(str, Enum)` means the frozensets compare equal against both `mission.status` (ORM string) and `MissionStatus.X` enum members, so call sites don't need `.value` accessors any more. Don't re-introduce open-coded inline tuples or local `{"X", "Y"}` literals ŌĆö that is the duplication the frozensets removed.
- **Litchi CSV has format-level invariants enforced inside the generator, not by the trajectory.** `generate_litchi_csv(flight_plan, mission_name, airport_elevation, *, mission=None)` is the only generator that takes `mission` as a kwarg, because three Litchi-specific constraints can only be resolved at write time: (1) **strictly positive `speed(m/s)` on every row** ŌĆö chain is `wp.speed` (when `> 0`) ŌåÆ `mission.measurement_speed_override` for MEASUREMENT/HOVER ŌåÆ `mission.default_speed` for everything else ŌåÆ `_LITCHI_DEFAULT_CRUISE_SPEED = 5.0`, clamped at `_LITCHI_MIN_SPEED = 0.1`. A zero or null persisted speed silently zeroed the row before the fix and Litchi flew the mission as stop-at-each-waypoint hops. (2) **`_LITCHI_MIN_3D_DIST = 0.6` collocation merge** ŌĆö consecutive waypoints under the 3D floor are folded into one row by `_group_collocated`; Litchi rejects the file with "Invalid 3D distance" otherwise. The kept row preserves every dropped waypoint's `camera_action` and represents the dwell as a Litchi `stay for` action (`actiontype=0`, param = ms). `_group_actions` reads `hover_duration` off **any** waypoint type, not just `HOVER` (#754 - the merged recording bookends now ride directly on the first/last `MEASUREMENT` and carry `hover_duration = config.recording_setup_duration`, so a HOVER-only filter would silently drop the camera-startup pause in the Litchi CSV). The action header widens to as many `actiontypeN`/`actionparamN` pairs as the busiest group needs, capped at `_LITCHI_MAX_ACTION_PAIRS = 15`. A defensive second pass in `_group_collocated` re-walks the merged list until stable so near-but-not-exact duplicates can't leave two reps still under the floor. (3) **spacing-aware `curvesize`** ŌĆö `_curvesize_for` returns `min(dist_prev, dist_next) * _LITCHI_CURVE_FRACTION` (0.5), pinned to `0` on takeoff, landing, and any row carrying a `_LITCHI_STOP_TYPE_ACTIONS = {0, 2, 3}` action (stay-for / start-record / stop-record ŌĆö a curved fly-through would skip them). MEASUREMENT/HOVER waypoints with a `camera_target` populate the `poi_*` columns and set `gimbalmode=1` (focus POI); the rest use `gimbalmode=2` (interpolate). The orchestrator's `LITCHI` dispatch branch threads `mission=mission` so the generator can see `default_speed` / `measurement_speed_override` ŌĆö don't fold the dispatch back into the default `generator(flight_plan, name, elevation)` shape, or every row's speed falls through to the 5 m/s constant fallback. KMZ/WPML and other format fixes do NOT belong in `litchi.py`; this is a Litchi-only fix.
- **Geozone bundle gating** (`include_geozones` / `include_runway_buffers` on `export_mission`): rejection order is `include_runway_buffers && !include_geozones` ŌåÆ 400; mission/flight-plan/airport lookup; status gate; generator support; THEN under `include_geozones=True`: any selected format outside `GEOZONE_CAPABLE_FORMATS = {MAVLINK, JSON, UGCS, KMZ, KML}` ŌåÆ 400, missing/incapable drone (`drone_profile is None or not drone_profile.supports_geozone_upload`) ŌåÆ 400. Don't reorder ŌĆö moving the drone check before the format check would mask "format unsupported" for capable drones. `build_geozone_payload` filters out `is_active=False` and `AIRPORT_BOUNDARY` safety zones server-side; per-format generators consume the shared dict and never re-filter. Empty airports yield empty arrays, not errors. MAVLink swaps content-type via `_resolve_export_content_type(fmt, with_geozones=...)` ŌĆö the same helper feeds both the single-file `Content-Disposition` and the zip filename, so don't index `_EXPORT_CONTENT_TYPES` directly when geozones may be on.
- **Dispatch must not commit a row the hub never stored.** `wayline_dispatch_service.dispatch_mission` raises `DomainError(502)` from `_post_kmz_to_hub` when the hub is unconfigured, unreachable, or returns non-2xx ŌĆö *after* the export + row upsert flushes but *before* the route's `log_audit` + `db.commit()` ever run, so the dispatch row, the `mark_dispatched()` bump, and the VALIDATED ŌåÆ EXPORTED transition all roll back with the uncommitted session. Don't move the hub POST after a commit and don't commit inside the service ŌĆö the invariant is "a persisted dispatch row means the hub acknowledged the wayline". Re-dispatch goes through the same path: the row is updated in place (unique `mission_id`), never duplicated, and `wayline_id` is never regenerated.
- **DJI altitude clamp gate fires *after* file generation, *before* the status transition.** `export_mission` always runs every requested generator and collects KMZ/WPML clamp records first; the 409 (`extra={"altitude_clamps": [...]}`) is raised only when the collected list is non-empty AND `acknowledge_altitude_clamps=False`, and the raise sits ahead of both the `VALIDATED ŌåÆ EXPORTED` transition and the `dji_heading_mode` write-back so neither side effect persists when the route does not commit. The route surfaces the 409 verbatim via the `DomainError` ŌåÆ `HTTPException` conversion in `app.utils.errors` so axios sees `response.data.detail.altitude_clamps`. Don't reorder the gate above generator dispatch (you'd lose the clamp list) or below the status transition (you'd ship an unacknowledged side effect). The `EXPORT` audit row carries both `acknowledge_altitude_clamps` and `altitude_clamps_count` in `details` so the trail records the operator's choice even when the count is 0. Non-DJI formats never collect clamps; KMZ and WPML each emit `clamps=altitude_clamps` into `_append_placemark`, which only appends on the waylines pass (`in_waylines=True`) so the template/waylines double-emit cannot duplicate a record.
- **Heading optimizer cap**: solver brute-forces 2^k where k is the number of AUTO inspections; `MAX_AUTO_INSPECTIONS = 10` keeps the search bounded. Removing the cap will blow up runtime.
- **`flight_plan_scope` branches** (`FULL`, `MEASUREMENTS_ONLY`): the two scopes share one canonical core - `MEASUREMENTS_ONLY` *is* that core, and `FULL` is constructed by *adding* at-transit-altitude TRANSIT bookends to it. `_assemble_core` filters every pass to `MEASUREMENT`/`HOVER` and joins adjacent passes via `pathfinding.compute_inter_pass_transits` (unified convex-hull visibility graph + bounded dilation fallback) between MH boundaries - regardless of scope. For `FULL` the rendered pass equals the original full pass byte-for-byte (intra-pass TRANSITs like the vertical-profile descent stay at their original positions). Bookend transits target the first/last MH waypoint, not the first/last full-pass waypoint - `_pass_boundary` returns MH endpoints unconditionally. `FULL` adds at-transit-altitude TRANSIT bookends from above-takeoff to first MH and from last MH to above-landing; no ground TAKEOFF/LANDING is ever emitted. Result: the `MEASUREMENTS_ONLY` waypoint list is a contiguous byte-for-byte slice of `FULL`. Single-pass missions get no inter-pass transit. The per-pass descent-to-transit-AGL waypoint is no longer emitted; `compute_transit_path` owns the climb on its first segment. Scope-branching code keys on `MEASUREMENTS_ONLY` only - the "else" branch is the airborne `FULL` path. Pre-#755 the enum carried a third `NO_TAKEOFF_LANDING` value and `FULL` was a ground-takeoff/landing scope; both legacy paths were collapsed into the current airborne `FULL` (operator hand-launches, triggers wayline mid-air; wayline carries no ground TAKEOFF/LANDING and never descends to ground at WP1).
- **`compute_transit_path` is bookend-only.** the only callers are `_build_takeoff_transit_bookend` and `_build_landing_transit_bookend`. inter-pass connections are owned by `compute_inter_pass_transits` for every scope, so there is one inter-pass A* implementation to maintain. don't reintroduce a per-segment `compute_transit_path` call between passes - that's the two-codepath shape that kept the runway-crossing regression alive.
- **`FULL` bookend assembly assumes `pass.waypoints[0]` IS the first MH.** The bookend lands on the first MH (via `_pass_boundary`) and `_assemble_core` then concatenates the original full pass starting at index 0. Today every method either emits MH at index 0 directly (HORIZONTAL_RANGE) or has had its pre-MH descent-to-transit-AGL waypoint removed (VERTICAL_PROFILE), so bookend-end and pass[0] coincide on (lon, lat) and the byte-for-byte slice invariant `MEASUREMENTS_ONLY ŌŖé FULL` holds. If a future method emits a pass shaped `[T_pre_MH, MH1, ...]` where `T_pre_MH` is at different (lon, lat) than `MH1`, the assembled `FULL` trajectory would land on MH1, then fly back to `T_pre_MH`, then forward to MH1 again. The slice test in `test_orchestrator_layering.py` exercises HR-only fixtures and would not catch this. Either skip pre-MH TRANSITs in `rendered_passes` for `FULL` or retarget the bookend to `pass.waypoints[0]` if you ever introduce such a method.
- **Empty inspection passes are dropped before phase 5.** `_process_inspection` returns `None` for any pass that produced zero waypoints (e.g. an ORM-level `measurement_density=0` that bypasses the schema floor) or has no LHA positions, emitting the soft warning naming the inspection; `_generate_trajectory_inner` then `continue`s past the `None` so the pass never reaches `inspection_passes`. - **LHA reorder is parent-locked**, not optimistic. Every write path that can change the per-AGL ordinal (`create_lha` with explicit sequence, `update_lha`, `delete_lha`, `bulk_generate_lhas`) opens with `SELECT ... FOR UPDATE` on the parent `AGL` row. Two concurrent `create_lha` calls on the same AGL would otherwise race for `max + 1` and violate `uq_lha_agl_sequence`; two reorders racing into the same target slot would deadlock or skip the shift. Don't shrink the lock scope.
- **`bulk_generate_lhas` rejects with 422 when fewer than 2 designator slots remain.** PAPI parents cap at 4 letters (A-D); generating a series across `[first_position, last_position]` requires at least 2 free slots so linear interpolation has a denominator. The early `remaining_slots < 2` return guards the cap, and a sibling `len(available_designators) < 2` check guards the PAPI partial-fill case (e.g. 3 of 4 letters used) where `remaining_slots` underreports the constraint. Without it the generator divided by `count - 1 == 0` and surfaced as a 500 `ZeroDivisionError`. The service flushes and the route commits the aggregate audit row alongside the LHA inserts.
- **PAPI `unit_designator` is derived, not stored intent.** For PAPI parents, `unit_designator` is a presentation of `sequence_number` (1=A, 2=B, 3=C, 4=D ŌĆö closest-to-runway is D=seq 4). `_apply_papi_invariant` rewrites letters via a two-pass lowercase-sentinel ŌåÆ uppercase final pass to dodge the `(agl_id, unit_designator)` unique constraint mid-rewrite. `update_lha` translates an inbound PAPI letter into a `sequence_number` shift; non-A-D letters and conflicting letter+seq pairs return 422. Test fixtures (`tests/data/trajectory.py::DESIGNATOR_MAP`, etc.) follow the same convention ŌĆö keep them aligned.
- **Sentinel value is `max + 1`, not `max + 1000`.** The earlier wider-gap sentinel was defensive coding from before the parent-row lock landed; with `SELECT FOR UPDATE` in place, `max + 1` is sufficient and matches what `bulk_generate_lhas` already used.
- **PAPI lens heights are plain pass-through columns.** `airport/lha.py` carries `lens_height_msl_m` / `lens_height_agl_m` through `create_lha` / `update_lha` verbatim ŌĆö they are surveyed-optics metadata, NOT a `position.z`, so the ground-altitude renormalize protocol never touches them and a terrain-source change does not resample them. `photo_metadata_service` is the only producer that derives lens AGL from a DEM; everywhere else the coordinator types both values.
- **PAPI methods preserve elevation angle, not AGL.** `center.alt` is the surveyed LHA target altitude, NOT a ground-truthed terrain reading. `_apply_terrain_delta` bumps each waypoint by `terrain_at_wp - terrain_at_center` so commanded AGL survives terrain undulation - the right invariant for fly-over and parallel-side-sweep, the wrong one for PAPI methods (HORIZONTAL_RANGE, VERTICAL_PROFILE, APPROACH_DESCENT) where the elevation angle from the LHA / touchpoint defines the all-white-zone edge. PAPI methods route through `_apply_papi_glide_slope_terrain` instead, which rebuilds each measurement/hover altitude geometrically (`anchor.alt + horiz_dist * tan(angle) + altitude_offset`) and clamps upward only when local terrain would drop the waypoint below `MIN_TRANSIT_ALTITUDE_AGL_M`. The anchor is the LHA center for HR/VP and the runway touchpoint for APPROACH_DESCENT ŌĆö APPROACH_DESCENT's glide slope is anchored on the touchpoint, not the PAPI, and the orchestrator passes `terrain_anchor = touchpoint` into the helper. The operator's `config.altitude_offset` (baked into `wp.alt` by every PAPI generator at emission) is re-added on the geometric rebuild branch only: the `else` branch (VP fallback / zero horiz) reads `wp.alt` verbatim where the offset already rides through, so re-applying there would double-count. Gimbal pitch is recomputed toward each waypoint's own `wp.camera_target` (angle-neutral for HR/VP because `camera_target` *is* the LHA center for them; APPROACH_DESCENT keeps `camera_target = LHA center` even though the geometry anchor is the touchpoint, so the framing stays on the PAPI). The post-recompute regression nets are split per method: `validate_papi_angle_band` checks every HORIZONTAL_RANGE measurement against `max(setting_angles) + HOVER_ANGLE_TOLERANCE`. `validate_vertical_profile_angle_band` checks each VP climb at the resolved bookends (`angle_start`, `angle_end` from `resolve_vertical_profile_angles`) against `HOVER_ANGLE_TOLERANCE`, plus a "band not fully covered" soft warning when PAPI mode does not span the `[min, max]` of `setting_angles`. The HR-style every-measurement check no longer applies to VP because the climb is *designed* to sweep below the all-white edge.

- **Vertical profile climb bookends.** `inspection_configuration.angle_source` is `"PAPI"` or `"CUSTOM"` (or null = CUSTOM legacy). `resolve_vertical_profile_angles(config, setting_angles)` clamps bookends into `[MIN_VERTICAL_PROFILE_ANGLE_DEG, MAX_VERTICAL_PROFILE_ANGLE_DEG]` (1.0┬░ / 16.5┬░, defined in `app.core.constants`) and falls back to legacy 1.9┬░ / 6.5┬░ when CUSTOM angles are unset. `inspection_service.add_inspection` and `update_inspection` reject PAPI-mode saves with 422 when any selected LHA is missing `setting_angle`, naming the offending units. The `vertical_profile_height` column is gone - `c5d6e7f8a9b0_vertical_profile_angles` backfilled `angle_end = atan2(vertical_profile_height, COALESCE(horizontal_distance, 400))` with `angle_source='CUSTOM'` so existing rows keep their geometric top bookend.
- **Safety-validator severity is intentional, not a default.** Transit AGL violations are *hard* because `_adjust_transit_altitude_for_terrain` should already have clamped them ŌĆö anything below `MIN_TRANSIT_ALTITUDE_AGL_M` at validation time means the elevation provider failed. Measurement/hover stay *soft* because PAPI 3┬░ glide-slope geometry inherently places measurements low. Boundary-egress is *soft* and gated on `mission.keep_inside_airport_boundary` (off ŌåÆ no warning at all and the pathfinder skips the penalty; on ŌåÆ soft warning on transit/takeoff/landing waypoints outside the polygon, measurement/hover always exempt). `ConstraintRule` checks stay post-generation advisory (soft). `check_battery` returns a soft `Violation` rather than `None` when endurance is unknown so the gap surfaces. See SPEC.md "Safety-semantics stance" for the full audited list before changing severities.
- **Fly-over waypoints sit behind the LHA, not above it.** `methods/fly_over.calculate_fly_over_path` shifts each waypoint back along the reverse heading by `height * tan(90┬░ + camera_gimbal_angle)` so a tilted optical axis lands on the LHA. Default gimbal is `-70┬░` (`DEFAULT_FLY_OVER_GIMBAL` in `trajectory/types.py`), which puts the drone ~5.46 m back at the 15 m default height; an explicit `-90┬░` gimbal hits the `abs(back_offset_m) <= NEGLIGIBLE_OFFSET_M` early-return branch and produces unshifted coords. `camera_gimbal_angle > -1┬░` is rejected up front (`MIN_TILT_BELOW_HORIZONTAL`) so `tan(90 + gimbal)` cannot blow up. `camera_target` still references the original LHA so trajectory-direction reversal and per-waypoint gimbal pitch are unchanged. `parallel_side_sweep` does NOT mirror this offset yet ŌĆö its lateral geometry differs and is tracked separately.
- **Constraints come off the mission, not the flight plan.** `generate_trajectory` reads `mission.constraints` (joined-loaded with the mission) and feeds it into `MissionData.constraints`. The FK lives on `mission`, so operator-attached rules survive flight-plan regeneration: replacing `mission.flight_plan` does not delete them. Don't reintroduce a `flight_plan.constraints` lookup or pass an empty list to validation as a workaround - with the FK on `FlightPlan` every recompute silently dropped the rules.
- **Visibility graph runs one batched predicate query per polygon set, not a per-edge loop.** `_build_visibility_graph` collects every surviving `(i, j)` pair (after the `grid_start_index` / `GRID_EDGE_RADIUS` cutoff), builds all candidate edges in a single `shapely.creation.linestrings(coord_array)` call, and resolves blocking via one `STRtree.query(lines, predicate="intersects")` against buffered obstacles + hard zones - predicate-mode returns the exact intersection set in one GEOS call, so there is no separate `prepared.intersects(...)` follow-up. The runway-crossing penalty uses the same shape against `surface_polys`, then `np.lexsort((poly_idxs, line_idxs))` sorts hits into surface-list order and a single batched `shapely.intersection(lines, polys)` + `shapely.length(...)` resolves all surviving `(line, poly)` pairs in one call before per-line accumulation so multi-surface FP penalties match the flat-scan reference byte-for-byte. Edge weights are likewise vectorized: `np.hypot(dx, dy) + penalty_per_edge` over the pair-coord array replaces the per-edge `math.sqrt` loop. Don't reintroduce a per-edge `for line in lines: tree.query(line)` loop, a `prep(...)` wrap, or a `for k in range(num_lines): segment_runway_crossing_length(...)` accumulator - the wrapper-dispatch and Python-level overhead is what the vectorization removed. Don't move the `grid_start_index` cutoff to after the index build, and don't drop the lexsort - both invariants are pinned by `TestVisibilityGraphSTRtreeEquivalence` (5 cases) and `TestVisibilityGraphVectorizedEquivalence` (6 cases) in `test_trajectory_pathfinding.py`.
- **`resolve_inspection_collisions` inherits per-waypoint metadata from the source slice, not from the bounding anchors.** Earlier shape hardcoded every rerouted waypoint as `MEASUREMENT` / `PHOTO_CAPTURE` / `camera_target=center` / `alt=anchor_before.alt`, which corrupted VIDEO_CAPTURE recordings (RECORDING_START -> PHOTO_CAPTURE truncated mid-pass), FLY_OVER per-LHA targets, PARALLEL_SIDE_SWEEP terrain deltas, and VERTICAL_PROFILE altitude bands. The right contract: pick the nearest source waypoint by 2-D distance in projected local coords and copy `waypoint_type`, `camera_action`, `camera_target`, `alt`, `speed`, `heading`, `inspection_id`, `hover_duration`. Heading is *inherited* (not recomputed toward the next anchor) so row-direction methods preserve their along-row heading and the `MAX_TURN_ANGLE` guard stays meaningful; `gimbal_pitch` is recomputed because the (lon, lat, alt) -> camera_target geometry changed. When `elevation_provider` is supplied, terrain queries are batched into a single `get_elevations_batch` call (source positions then rerouted positions) and the rerouted altitude is `src.alt + (terrain_at_rerouted - terrain_at_source)`. Don't reintroduce a centroid-target / anchor-altitude shortcut.
- **Flight envelope is opt-OUT, not opt-in.** `build_enriched_response` filters waypoints by `wp.waypoint_type not in _GROUND_LEVEL_WAYPOINT_TYPES = {TAKEOFF, LANDING}`. Earlier shape used an opt-in `{MEASUREMENT, TRANSIT}` set that silently dropped HOVER, so MEASUREMENTS_ONLY hover-only inspections (hover-point-lock, meht-check) returned `min/max_altitude_msl/agl = None`, and FULL-scope HOVERs above transit altitude under-reported the peak. The set name and the docstring are the contract ("takeoff/landing sit on the ground") - if a future waypoint type lands on the ground, add it to `_GROUND_LEVEL_WAYPOINT_TYPES`; never re-narrow the keep-set. Per-inspection stats are unaffected because they key on `inspection_id`, which TAKEOFF/LANDING lack.
- **Runway pair-link writes are partner-locked.** Every write that can touch the link (`couple_surfaces`, `decouple_surfaces`, `update_surface`, `delete_surface`) opens with `_load_paired_for_update` so the partner row is held under `SELECT FOR UPDATE` for the duration. `update_surface` rejects `identifier` rename while coupled (decouple first), then propagates geometry-affecting fields across the pair through `_propagate_pair_geometry` (reversing centerline, swapping threshold/end, reciprocating heading, mirroring length/width/buffer/boundary) and re-runs `_recompute_agls_distance_from_threshold` on both sides if `threshold_position` or `end_position` moved. `delete_surface` returns `(deleted, paired_deleted)` so the route can attach a per-side audit row; `paired_deleted` is `None` when the surface was uncoupled. `couple_surfaces` rejects same-id, cross-airport, non-RUNWAY, and already-coupled inputs at 422; `create_reverse_surface` rejects when the base is already coupled. The trajectory pipeline is unchanged - `paired_surface_id` is configuration metadata, not a planner construct.
- **Taxiway heading is server-derived; an explicit heading always wins.** `create_surface` fills a missing `heading` on a TAXIWAY from the centerline first-to-last bearing (via the model's `recalculate_dimensions()`); `update_surface` re-derives only when `geometry` is in the payload's `model_fields_set` and `heading` is not. `_derive_taxiway_heading` no-ops for runways, and a degenerate centerline (<2 points) leaves heading NULL. Migration `0014_taxiway_heading_backfill` filled pre-existing NULL-heading taxiway rows the same way (downgrade clears taxiway headings only). The frontend feature panels are type-agnostic and render the heading + compass/flip widget once the value exists - no UI branch to maintain.
- **Unchanged-position update paths intentionally ignore the inbound `alt`.** `update_agl` / `update_lha` / `update_obstacle` compare the inbound `(lon, lat)` against the stored value at 7 dp (~1 cm). On a match they skip the elevation provider entirely and keep the stored z ŌĆö even if the inbound payload carries a different `position.coordinates[2]`. This matches the "leave existing position.alt untouched" contract: ground elevation is the truth, and identity round-trips (rename, recolor, height edit, idempotent PUT) pay zero provider construction cost. The 7 dp threshold is tight enough that any real coordinator drag trips re-resolve; `_position_unchanged` treats `(None, None)` as unchanged so a payload that omits position cannot accidentally trigger a provider build. Polygon updates do the same per-vertex via `_ring_diff` ŌĆö a ring with some-but-not-all vertices moved resamples only the moved indices and copies stored z byte-for-byte onto the rest. If the stored WKT is unparseable or the vertex count changed, `_renormalize_boundary_with_stored` falls back to a full whole-ring resample. Do NOT "fix" this by always persisting the inbound `alt` ŌĆö that would re-introduce drift between the displayed elevation and the actual ground reference, and would re-open per-update remote elevation fan-out on flat airports with the API toggle on.
- **Wire freshly-flushed children via the relationship, not the FK.** `inspection_template_service.create_template` / `update_template` assign `template.default_config = config` rather than `template.default_config_id = config.id`. The previous internal `db.commit()` forced an `expire_on_commit` reload that papered over the difference; now that the service flushes and the route commits, setting the FK alone leaves `template.default_config` unattached in Python until session expiry, so the post-flush response would render with `default_config=null`. Assigning the relationship wires both sides in one step and lets `_enrich(template, db)` build the response without an explicit `db.expire(template)` + reload. Same shape applies anywhere a service flushes a child and returns the parent in the same call.
- **Mission-report surface crossings are kind-driven and validation-sourced, not recomputed.** `_build_crossing_analysis_page` reads the persisted `ValidationViolation` rows off `data.violations` and selects crossings by `violation_kind == "surface_crossing"` (`_surface_crossing_parts`), with message classification used only for legacy `violation_kind is None` rows. This covers RUNWAY *and* TAXIWAY - the old `" crosses RUNWAY "` message filter silently dropped TAXIWAY crossings. Columns are `Surface | Type | Waypoint # | Min Altitude AGL`, where Waypoint # and Min Altitude AGL are resolved from each warning's structured `waypoint_ids` against `airport_elev` (a `NameError` here means the `airport_elev` definition was dropped - it is required by the AGL resolution). `"No runway crossings detected."` fires only when validation genuinely has no crossing warning, so the PDF and the validation page read the same rows and cannot diverge. The *Waypoints Near Runway Thresholds* recompute is deliberately kept independent (justified by an in-code comment): the validator emits no kind for threshold proximity because it is advisory geometry, not a constraint. Don't reintroduce a from-scratch point-proximity recompute for the crossing table - that was the `_point_near_polygon` heuristic that was deleted because a transit clipping a runway away from its endpoints made the PDF disagree with the validation page. `_point_in_polygon` is kept (still used by the safety-zone-passes section) - don't delete it as dead code alongside `_point_near_polygon`.
- **Inter-pass visibility-graph builds run in parallel, gated by `TRAJECTORY_PARALLEL_VG`.** `compute_inter_pass_transits` dispatches each transit to a `ThreadPoolExecutor` (max workers = `min(len(transits), os.cpu_count())`); Shapely 2 batched predicates release the GIL so the 6 builds on the reference mission run concurrently. Determinism is preserved because results are collected in transit-index order and warnings appended afterwards. Each transit is computed by the module-scope `_compute_one_transit(ctx, transit_idx)`, dispatched via `functools.partial(_compute_one_transit, ctx)` on both the executor and sequential branches; `ctx` is an immutable `_TransitContext` NamedTuple bundling the 10 read-only captured vars. The per-worker mutable `local_cache` + `cur_dilation` MUST stay function-locals inside `_compute_one_transit` - never hang them off `_TransitContext` (shared read-only across workers); that is what keeps parallel transits from sharing dilation state, and is why the cross-transit attempt-0 cache hit from the old sequential closure is gone (replaced by per-thread rebuilds, ~one dict insert of overhead). Set `TRAJECTORY_PARALLEL_VG=0` to fall back to the sequential path on runners where the executor misbehaves. The parallel path is pinned by `TestComputeInterPassTransitsParallelEquivalence`, the lift by `TestComputeOneTransitLifted`, and the shared surface-edge helper by `TestSurfaceEdgeNodes`, all in `test_trajectory_pathfinding.py`.
- **Measurement runs through a repository port, not the ORM ŌĆö and the engine/Celery imports are deferred on purpose.** `measurement_service` talks to `MeasurementRepository` (`app.domain.measurement`) and builds the `SqlAlchemyMeasurementRepository` adapter (`app.infra.measurement`) via `_repo(db)`; swapping the DB backing later is one new adapter class, no service change. The OpenCV engine and the Celery tasks are imported **inside** the engine/enqueue seam functions, never at module top, so `app.main` boots on `requirements.txt` alone (`cv2` / `celery` live in `requirements-video.txt`). The aggregate's status machine is the contract: `transition_to()` validates each hop and `fail(msg)` moves to ERROR from any non-terminal state. The worker runners wrap the heavy work in try/except and route any failure to `_mark_failed`, which re-reads the row on a **fresh** read so the ERROR transition + `error_message` persist even after the in-flight transaction was rolled back. Tests mock the engine seams (`extract_first_frame_and_detect`, `run_two_pass_processing`) and the enqueue seams, so don't inline the lazy imports back to module scope ŌĆö that breaks both the import-safety guarantee and the monkeypatch points.
