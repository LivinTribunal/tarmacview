# hooks package

## Purpose

Reusable React hooks for the operator/coordinator/admin frontend — map tools and drawing, mission edit history, persistence helpers, and async UI state. Anything stateful that needs to be shared across components but isn't a global context lives here.

## Public API surface

- `useMapTools` (default + named `MapTool` enum, `EDITING_TOOLS` set) — owns the active map tool plus 2D/3D mode, with keyboard shortcuts. The `MapTool` enum is the canonical list of tool IDs (SELECT, MOVE_WAYPOINT, MOVE_FEATURE, MEASURE, HEADING, ZOOM, ZOOM_RESET, PLACE_TAKEOFF, PLACE_LANDING). There is no PAN tool — empty-canvas drag pans under any active tool. Keyboard shortcuts: `s` SELECT, `w` MOVE_WAYPOINT, `m` MEASURE, `h` HEADING, `z` ZOOM (input/textarea/select focus suppresses them).
- `useMapDrawing` — high-level drawing controller used by surface/obstacle editors; layers `useUndoRedo` underneath.
- `useMapViewport` (no default; named exports) — `getSavedViewport`, `saveViewport`, `getSavedLayers`, `saveLayers`, `buildInitialLayerConfig`, plus the `MapViewportState` type. Persists per-airport viewport + layer toggles to `localStorage` under `tarmacview_mapViewport_<id>` / `tarmacview_mapLayers_<id>`. `KNOWN_LAYER_KEYS` is the round-trip allowlist for the saved-layers JSON; when a new `MapLayerConfig` key lands, add it here AND to `DEFAULT_LAYER_CONFIG` so legacy saved configs hydrate the new key from the default rather than dropping it (since #771 the list includes `airportBoundary`, which defaults to `true`).
- `useCesiumSync` — bidirectional camera sync between MapLibre and CesiumJS; see `components/map/cesium/cesiumUtils.ts` for the unit conversions.
- `useFocusFeature({ mapRef?, cesiumViewerRef? })` — the shared "locate"/recenter intent router for the 2D and 3D maps. Returns a single `locateFeature(feature)` that dispatches to whichever viewer is currently live (Cesium preferred when a non-destroyed `cesiumViewerRef` is present, else MapLibre). Also exports the pure helpers `computeMapLibreFocus`, `flyMapLibreToFeature`, `flyCesiumToFeature`, `cesiumRangeForFeature`, `computeFeatureAltitude`. This is the only sanctioned recenter path — side list panels and the map dblclick handlers call `locateFeature`, never `map.flyTo` / `viewer.flyTo` directly.
- `useFlyAlong(waypointCount)` — status / speed / progress controller for the 3D fly-along (idle | playing | paused, speed multiplier, normalized 0–1 progress). Exposes `play`, `pause`, `stop`, `setSpeed`, `setProgress`. Auto-stops when `waypointCount` drops below 2. Motion itself is driven by `viewer.clock` inside `CesiumFlyAlong`; this hook owns the React-side state, the component pushes progress back via `setProgress` each `postRender`.
- Drawing primitives — `useDrawCircle`, `useDrawPolygon`, `useDrawRectangle`, `useHeadingTool`, `useMeasureDistance`, `usePlacePoint`, `useVertexEditor`. Each still owns its own MapLibre source/layer ids (constants `SRC_*`/`LYR_*` at the top of the file or in a sibling sources module) and exposes GeoJSON FeatureCollections for rendering. `useDrawCircle` / `useDrawPolygon` / `useDrawRectangle` / `usePlacePoint` no longer hand-roll the add/clear/remove plumbing — they declare a `DrawingSourceSpec[]` and delegate it to `useDrawingSources` (see below). `useHeadingTool` / `useMeasureDistance` keep their own inline wiring. `useVertexEditor` keeps its React effect + drag state machine in `useVertexEditor.ts`, but its MapLibre source/layer ids and add/clear/remove/style-load helpers live in `vertexEditorSources.ts` (`SRC_NODES` / `SRC_MIDPOINTS` / `LYR_CORNERS` / `LYR_CENTER` / `LYR_MIDPOINTS`, `ensureSources`, `clearSources`, `removeSources`, `waitForStyleLoaded`), and the pure circle/polygon detection + radius-edge + `buildVertexGeometryUpdate` rebuilder live in `vertexEditorGeometry.ts` (`EditState`, `VertexGeometryUpdate`, `extractEditState`, `radiusEdgePoint`, `buildVertexGeometryUpdate`). The hook's public signature `useVertexEditor(map, feature, isSelectTool, onGeometryUpdate) → { isEditing }` and the re-exported `VertexGeometryUpdate` are byte-identical to the pre-#634 inline version.
- `useDrawingSources(map, spec)` (default) + named `createDrawingLayer`, `clearDrawingSources`, `removeDrawingSources`, the `DrawingSourceSpec` type, and the shared `EMPTY_FC` empty feature collection. One declarative `DrawingSourceSpec[]` per tool (`{ source, layers[] }`, layers in paint order); the hook returns `{ ensure, clear }` and owns an unmount-only source teardown via effect cleanup. `ensure` is idempotent per source (skips if `map.getSource(id)` already exists), `clear` resets every source to `EMPTY_FC`, teardown removes layers then sources in reverse (LIFO) order with missing-id errors swallowed.
- `useDirtyHistory` — pending-change tracker with undo/redo for entity-level mission edits. Exposes `markDirty`, `clearAll`, `getPendingChanges`, `getPendingChange(entityType, entityId)` (live, render-safe lookup of one entity's staged patch), `undo`, `redo`, plus `isDirty` / `canUndo` / `canRedo`.
- `useUndoRedo<T>` — generic past/future stack, default depth 10.
- `useDownloadMissionReport` — fetches PDF report and triggers browser save.
- `useOnComputationCompleted` — fires once on `COMPUTING → COMPLETED` transition from `ComputationContext`.
- `useFieldLinkStatus()` — polls `GET /field-link/status` via `@/api/fieldLink` every `FIELD_LINK_POLL_INTERVAL_MS` (10 s, `@/constants/ui`) while mounted. Since #109 it returns a `FieldLinkPoll` object — `{ status, lastChecked, checking, refresh }` — not the bare status. `status` is `null` until the first response (consumers hide/disable until then); a failed check degrades to the no-hub shape (`hub_online: false, rc_connected: false, broker_connected: false, devices: [], connect_url: null, public_host: null`) instead of throwing. `refresh()` forces an immediate re-check (the Field Hub dialog's **Heartbeat Check** button); `checking` flips `true` only for those manual checks (background poll ticks never set it, so the button doesn't spin every 10 s); `lastChecked` is epoch ms of the last completed check (poll or manual). Deliberately a plain `setInterval` hook (the `ComputationContext` pattern), NOT a `queries/` React Query hook — that layer is restricted to the two read resources listed in `api/CLAUDE.md`. Consumed only by `components/mission/ExportPanel`, which shares the one poll result three ways: the presentational `FieldLinkStatusChip` (takes `status` as a prop), the `SendToDroneSection` dispatch gate, and the `FieldHubDialog` (also gets `refresh` / `checking` / `lastChecked` for the heartbeat button) — all take props; don't call the hook from the children, that would multiply the polling.
- `useLastVisitedPath` — remembers per-role last route in `localStorage` key `tarmacview_lastPath`.
- `useElevationResolver(airportId)` — returns a memoized `ElevationResolver` (`(lat, lon) => Promise<number | null>`) bound to one airport. Wraps `fetchElevationAt` and swallows network errors to `null` so callers fall through to the existing-marker / airport-elevation chain inside `computePlacementUpdates`. Returns `undefined` when `airportId` is falsy. `MissionConfigPage` and `MissionMapPage` both consume the same hook so the resolver instance is shared per airport.
- `useResolvedAltitude({ effectiveEntityType, showAltInput, altResolveLat, altResolveLon, resolver, airportElevation })` — owns the coordinator point-entity **Altitude** input auto-fill. Returns `{ manualAlt, altLoading, altFallback, handleAltChange }`. Prefills `manualAlt` from the passed-in `ElevationResolver`; a `null` result drops to `airportElevation` and raises `altFallback`. `handleAltChange` flips an internal `userEditedAlt` that freezes the field against later resolver responses / marker drags; an internal request-id ref makes the effect last-write-wins on rapid re-clicks; state resets when `effectiveEntityType` changes. Consumed only by `components/coordinator/hooks/useCreationFormState` (the `CreationForm` state hook, extracted in #623) — that hook owns the resolver, the lat/lon inputs, and all submit logic; `useResolvedAltitude` owns just the resolve state.
- `useAngleLock({ angleLocked, configOverride, onChange, distanceFromLha, cameraGimbalAngle })` — the hover-point-lock distance/height/angle coupling. Returns `{ onDistanceChange, onHeightChange, onAngleChange }` change builders; each sets its own field on a fresh `InspectionConfigOverride` and, when `angleLocked` is on, runs `solveTriangle` (`@/utils/angleLock`) to keep the other two consistent (distance→height, height→angle, angle→height). With the lock off every builder is a plain passthrough. Consumed only by `components/mission/HoverPointLockFields`; it owns no state of its own.
- `useAirportLookup({ isOpen, icaoCode, importRadius, setErrors, t, setName/City/Country/Lat/Lon/Alt })` (default) + named `ICAO_REGEX`, `MIN_IMPORT_RADIUS_KM` / `MAX_IMPORT_RADIUS_KM`, the `SuggestionState` type — owns the OpenAIP lookup + suggestion-checkbox state machine for the create-airport dialog (`suggestions`/`looking`/`lookupError`/`lookupEmpty`/`expanded`). Exposes `handleLookup`/`applyLookup`, the section/item toggles (`toggleSection`, `toggle{Runway,Obstacle,SafetyZone}`, `setSectionChecked`, `setAllChecked`) and `createCheckedSuggestions(airportId) → failureCount`. Same form-owns-the-fields split as `useResolvedAltitude`/`useAngleLock`: the dialog owns the ICAO/name/location field state and passes its setters in, the hook owns only the lookup/suggestion state. Reset-on-reopen is the hook's own `[isOpen]` effect (stable setters, no ESLint disable). Consumed only by `components/coordinator/CreateAirportDialog` (#626 extraction).
- `useInspectionConfig({ inspection, template, agls, droneProfile, mission, configOverride, onChange, selectedLhaIds, directionBearing })` — the value/derived/camera engine behind the per-inspection override form (extracted from `InspectionConfigForm` in #627; behavior byte-identical, asserted by the `InspectionConfigForm.test.tsx` regression oracle). Owns the 3-tier `resolveNumber` resolver, every derived memo (`horizontalDistanceToLha`, `lhaSpanM`, `computedOpticalZoom`, `targetAgls`, `computedMehtHeight`, `computedObservationAngle`/`missingSettingAngleUnits`, `verticalProfilePapiMissing`, `verticalProfilePreview`, `aglOfSelectedLha`, `speedWarning`, the direction/bearing-flip block), the zoom live-bind state + both zoom effects, the camera-preset list/select/save handlers + `handleCameraModeChange`, the `hoverAglId` sync, and `handleNumberChange`. Returns a flat bag of resolved scalars + handlers — APPROACH_DESCENT adds `descentStartDistance` and `descentGlideSlopeOverride` to the bag, consumed by `mission/ApproachDescentFields` via `MethodSpecificSections`; SURFACE_SCAN joins the `showDirectionSection` method set (its serpentine path flips on REVERSED) and renders `mission/SurfaceScanFields` instead of the LHA-selection block. Since #867 `showDirectionSection` is `methodCaps(inspection.method).usesDirection` - a lookup into the exhaustive `METHOD_CAPABILITIES` table in `@/utils/methodAglCompatibility`, not a 4-method OR-chain. The sibling inspection-config components gate the same way (`FlightParametersSection` speed/density/hover via `caps.usesMeasurementSpeed`/`usesDensity`/`usesHover`, `LhaSelectionSection` surface-target via `caps.target === "SURFACE"`, `MethodSpecificSections` form-slot via `caps.formSlot`), so adding a method's UI gating is one new table row - a forgotten method is a compile error. The parent `InspectionConfigForm` is a thin orchestrator owning only UI-only state (`collapsed`, `lhaSelectionOpen`, `angleLocked`), and the `mission/{LhaSelectionSection,MethodSpecificSections,FlightParametersSection,CaptureModeSection,DirectionSection}` components render off the bag.

- `useMapPickingTools({ selectedFeature, airport, pendingGeometry, pendingPointPosition, getMap, t })` (default) — owns the coordinator pick-on-map state for **touchpoint**, **threshold**, **end**, and the first/last LHA: the four `picking*` flags and their `picked*Coord` results, plus `anyPicking` and `handlePickingMapClick(lngLat) → boolean` (the click handler the `AirportEditPage` calls before falling through to the normal map-click branch — it consumes a click for an active picker and returns `true` so the caller stops). Calls `onMissionUpdated(missionData)` then `refreshMissions()` on every successful fetch so the parent tab nav and mission cache stay in sync. `isInitialLoad`, `mission`, `flightPlan`, and `warnings` all reset when `id` changes (separate effect from the fetch) so the loading skeleton shows on mission switch; `focus` and `visibilitychange` listeners trigger background refetches. `handleExport` is the anchor-download dance: `exportMissionFiles` → branch on `result.kind` → if `"clamp_warning"`, set `clampWarning` and `return` (no anchor, no `fetchData`); otherwise `URL.createObjectURL(result.blob)` → `<a download={result.filename ?? ${mission.name}.${formats[0].toLowerCase()}}>` → `a.click()` inside try / `revokeObjectURL` + `removeChild` in finally → clear `clampWarning` → `fetchData()`. Backend-provided `filename` wins over the client fallback (DJI flight hub 2 sanitizer rules live server-side). On failure `handleExport` extracts the FastAPI `DomainError` body via the module-local `extractApiErrorMessage(err)` (reads `err.response.data.detail` as either a string or `{message}` object) and surfaces it through the toast; the generic `mission.validationExportPage.exportError` i18n string is only used when the response carried no usable message. The hook exposes `clampWarning: AltitudeClamp[] | null` plus `dismissClampWarning()` so `ExportPanel` can render `AltitudeClampWarning` and let the operator tick the acknowledge checkbox; the panel re-fires `handleExport` with `acknowledge_altitude_clamps: true` and a fresh `clamp_warning` response overwrites the stored list (the operator cannot silently re-confirm a different clamp set). `handleDelete` returns `boolean` so the page can decide whether to navigate; the other action handlers (`handleValidate`/`handleComplete`/`handleCancel`) call `fetchData()` on success and toast via `SLOW_NOTIFICATION_TIMEOUT_MS`-keyed `showNotification` on failure. `inspectionIndexMap` is a sequence-order-sorted 1-based index from inspection id, used by the validation map to label inspections.
- `useMeasurementList({ airportId })` — **orphaned after #180**: the airport-scoped `operator-center/MeasurementsListPage` it backed was deleted when Results moved into the mission workspace, so the hook + `MeasurementListTable` now sit in the tree with no live caller (kept pending human follow-up). Was shared list state mirroring `useMissionList`. Fetches via `listAirportMeasurements(airportId)` (every measurement across the airport's missions/inspections, newest first), layers `useListSort` over `mission` / `inspection` / `status` / `created_at` / `result` (the numeric keys sort on sequence-order / parsed date / `pass_count`), client-side search over mission name + inspection method, and a `FilterSpec` bar (status pills `defaultMode: "all-active"`, a mission select derived from the loaded rows, a `created_at` range). Owns auto-refresh polling: a silent background refetch every `MEASUREMENT_POLL_INTERVAL_MS` (`@/constants/ui`) while any row is QUEUED / FIRST_FRAME / PROCESSING (no table blanking, last-good rows kept on a transient poll failure), stopping when none are active so a finished run flips to DONE on its own. Also feeds those active row ids into `MeasurementProgressContext` via `sync()` on each fetch (a `useMemo` over the QUEUED / FIRST_FRAME / PROCESSING ids), so the corner progress toast picks up runs on direct navigation / reload and re-tracks a run just confirmed back into processing - `sync` is additive, the context's own per-id poll is what drops finished runs (see `contexts/CLAUDE.md`). Exposes the row-action handlers `handleDelete(row)` (calls `deleteMeasurement` then `fetchRows`) and `handleRename(row, label)` (calls `updateMeasurement` with the trimmed label or `null` to clear it back to the inspection fallback, then `fetchRows`) - both swallow errors and refetch on success, mirroring `useMissionList`. Returns `paged` / `sorted` / `filterBar` plus the search, sort, and pagination handlers the page spreads into the shared `ListPageLayout`.

## Invariants

- **MapTool enum is the source of truth** for tool IDs. Components compare with `MapTool.MOVE_WAYPOINT`, never raw strings.
- **`MOVE_*` is a superset of `SELECT`.** `MOVE_WAYPOINT` and `MOVE_FEATURE` keep all of `SELECT`'s behavior (single-click to select + open info panel, double-click to recenter) on top of their drag/edit affordances. Don't gate select-only handlers on `activeTool === SELECT`; gate them on a "selectable tools" set that includes the move tools, otherwise users have to toggle back to SELECT just to inspect a feature.
- **`buildInitialLayerConfig` is the only correct way** for the operator mission map to seed `MapLayerConfig` from defaults + persisted layers + props. It deliberately drops a saved `simplifiedTrajectory: true` so the full trajectory renders by default on reload, and clears the stale trajectory-family children that were forced false by the toggle handler.
- **Trajectory family keys** (`trajectory`, `transitWaypoints`, `measurementWaypoints`, `path`, `takeoffLanding`, `cameraHeading`, `pathHeading`) must stay in sync between `useMapViewport.ts` (`TRAJECTORY_FAMILY_KEYS`) and `components/map/hooks/useMapLayerSync.ts` (`TRAJECTORY_CHILDREN`, moved there from `AirportMap.tsx` in #571). Adding a new child requires updating both.
- **localStorage access** is wrapped in try/catch in every hook that uses it — Safari private mode and quota errors must not crash the UI.
- **Drawing hooks own their MapLibre layer/source IDs.** Two drawing tools must not reuse the same ID; the constants at the top of each file are the contract. `useDrawingSources` only centralizes the add/clear/remove *plumbing* — it never invents ids; the per-tool `DrawingSourceSpec[]` (with its own `SRC_*`/`LYR_*` constants and layer order) is still the source of truth. Preserve each tool's exact source/layer ids and the spec ordering (point single-source, circle dual-filter stroke, polygon solid/dashed + snap-target, rectangle single stroke) when refactoring.
- **Hook files are tested co-located** as `<name>.test.ts` and run under Vitest with a jsdom DOM. Never reach into `window` outside hooks; pass dependencies in.
- **`useAngleLock` is intentionally stateless.** The `angleLocked` toggle and the `configOverride` it patches are owned by the parent form (`InspectionConfigForm`) and passed in, so they survive inspection/method switches. Do not pull that state into the hook — a `useState` inside would reset the lock and lose the override every time the operator changes inspection or method.
- **`useInspectionConfig.resolveNumber` precedence is a hard contract: dirty override → saved config → template default.** The override branch keys on `field in configOverride`, not truthiness, so an explicit `null` in the override means "the operator cleared this field" and must NOT fall through to saved/default. Every camera/direction/preset read uses the same `!== undefined` / `in` guard for the same reason. Changing this to `??`-style coalescing silently revives cleared fields and breaks the regression oracle.
- **The operator mission pages own the cross-hook state; the mission hooks receive it as params/setters.** `updateMissionState`, `lastSaved`, `setMission`, `setFlightPlan`, and the selected-inspection state live on `MissionConfigPage` / `MissionMapPage` and are passed into `useInspectionEditing` / `useMissionSave` / `useWaypointEditing`. This is deliberate — it is the only behavior-safe way to break the genuine `handleSave` ↔ `inspectionDirty` ↔ `updateMissionState` cycle. Do not pull that shared state into a hook with its own `useState`; it desyncs the page and the sibling hooks.
- **`useDirtyHistory.getPendingChange` is a live, render-safe read.** Call it during render to overlay the staged patch onto the server record (`{ ...feature.data, ...(pendingPatch ?? {}) }`). Do not stash it in a ref or memoize across renders — its identity intentionally changes when the underlying state changes so dependents re-render. Local `formData` mirrors of edited values are an antipattern here: they trap pending edits when a parent switches the selected entity. The history depth cap is `MAX_HISTORY = 20`.

## Gotchas

- **Persisted layer state is per-airport**, keyed on `airport.id`. Switching airports loads a different snapshot — do not memoize across airports.
- **Saved `simplifiedTrajectory: true` is intentionally not honored on reload.** If you "fix" that to round-trip cleanly, the full trajectory stops rendering by default on reload (the operator opens a mission and sees only the simplified dot view, with no obvious way back). The user can still toggle it in-session via `LayerPanel`.
- **Undo/redo depth is 10 by default.** Mission map and surface drawing both rely on the same `useUndoRedo<T>`; deeper history is a session-wide change, not per-tool.
- **`useFlyAlong` does not own any timer.** The previous `setTimeout` scheduler was deleted when `CesiumFlyAlong` switched to a `viewer.clock`-driven `SampledPositionProperty`. Tests in `useFlyAlong.test.ts` drive `setProgress(p)` directly to verify status / progress transitions — do not reintroduce a timer here, and do not push per-frame state from React (the Cesium component is the per-frame writer).
- **`useLastVisitedPath` ignores `/login`, `/setup-password`, `/maintenance`** — those should never be remembered as a return target. Add new auth-only routes to `IGNORED_PATHS` if you create them.
- **`flyCesiumToFeature` must scan `viewer.dataSources`, not just `viewer.entities`.** Trajectory waypoint dots and path arrows live in `CustomDataSource`s, so a feature match has to walk every `viewer.dataSources` entry too — searching only `viewer.entities.values` regresses 3D double-click-to-recenter on waypoints (it silently falls through to the coord branch and frames the wrong spot). The no-match fallback frames a `BoundingSphere` via `camera.flyToBoundingSphere` with the same `HeadingPitchRange` orbit offset as the entity path; don't park `camera` destination straight on the point or the camera ends up inside the feature.
- **`useDrawingSources` removes sources only on unmount, not on `clear`.** `clear()` resets data to `EMPTY_FC` (keeps the source/layers alive for the next draw); the source/layer removal runs once in the effect cleanup when `map` or `spec` identity changes. Pass a stable `spec` (the per-tool `DrawingSourceSpec[]` is module-constant) — rebuilding it every render would thrash the teardown effect and drop the layers mid-draw. The add path is per-source guarded so calling `ensure()` repeatedly is safe.
- **`flyMapLibreToFeature` defers the `flyTo` via `queueMicrotask`.** Called from inside a map dblclick listener, MapLibre's HandlerManager runs `map._stop(true)` at the end of the event loop and would cancel the animation; the microtask wrapper runs after that. Don't strip it (same note lives in `components/map/CLAUDE.md`).
- **`useResolvedAltitude` needs a stable `resolver`.** Its resolve effect lists `resolver` in its deps, so pass the memoized `useElevationResolver(airportId)` result, not a fresh inline closure — a new closure each render re-fires the effect and re-hits the DEM endpoint on unrelated keystrokes. The "frozen once edited" / fallback / last-write-wins behavior is owned by this hook, not `CreationForm`; behavior must stay byte-identical to the pre-#597 inline version (a regression-guard suite asserts this).
- **`useAirportLookup.createCheckedSuggestions` keeps a flattened runway → obstacle → safety-zone `Promise.all` with a per-promise `.catch`.** That exact ordering, the skip-unchecked filter, and "count failures, never throw" are a #626 behavior-neutrality contract - `CreateAirportDialog`'s partial-failure path (modal stays open, failure count shown) and its tests assert it. Don't reorder, and don't collapse the per-call catch into one try/catch.
- **`STATUS_ORDER` / `TERMINAL_STATUSES` live in `@/constants/mission`, not in a page.** They moved out of `MissionConfigPage` in #599 so the page orchestrator and `useInspectionEditing` / `useMissionSave` can both import them without a page↔hook import cycle. Don't re-add a local copy in a hook.
- **The mission hooks keep `*Ref` stale-closure mirrors** (`inspectionDirtyRef`, the seven refs in `useTakeoffLandingPicker` - including `handleMissionChangeRef` so its `useCallback([])` click handler always invokes the latest `handleMissionChange`, the `handleUndo`/`handleRedo` ref mirror in `useMapInteractionTools`). A map-click or keydown listener is registered once with a `[]` dep array and reads through the ref, so the handler stays correct without re-subscribing. When you add state such a one-shot listener must see, mirror it into a ref the same way — reading the state directly inside the `useCallback([])` reads a stale value.
- **`useDroneEditor` flushes its autosave on drone-switch through refs, not state.** `handleSelectDrone` clears the pending debounce and, if `latestFormRef` vs `droneToForm(droneRef.current)` shows a dirty form with a non-empty `name`, fires `performSave` synchronously before `navigate` — without that flush, switching drones drops the last edit. `latestFormRef` mirrors `formData` and `droneRef` mirrors `drone` precisely because the `setTimeout` autosave callback and the navigate path would otherwise read stale closures; keep both refs written on every state update. The autosave gate is the `FIELDS.some(...)` dirty check plus a `name.trim()` guard (empty name → no save, no `nameError`). `beforeunload` only cancels the timer, it does not flush. Debounce/tick come from the shared `AUTOSAVE_DEBOUNCE_MS` / `AUTOSAVE_INTERVAL_MS` in `@/constants/ui` (the #597 alignment) — do not reintroduce a local `AUTOSAVE_DELAY`.
- **`useInspectionConfig` zoom auto-binds until the operator touches the slider.** While `zoomTouched` is false, an effect pushes `computedOpticalZoom` into the override on every change. Switching inspection re-seeds `zoomTouched` from whether the new inspection has an explicit `optical_zoom`; changing only the horizontal distance for the *same* inspection deliberately releases the flag (`zoomKeyRef`) so the derived zoom re-runs. Do not collapse the two effects or drop the `zoomKeyRef` keying — a single touched flag without the per-inspection/per-distance key either freezes zoom after the first manual edit or wipes a deliberate operator zoom on an unrelated distance tweak.
- **`useTemplateAutosave`'s LHA re-seed freezes once the coordinator edits the selection.** The `[allAgls, template]` effect re-seeds `selectedLhaIds` from `template.target_agl_ids[0]` only while `userTouchedLhaRef` is false; the three LHA handlers (`handleToggleLha`/`handleSelectAllLhas`/`handleDeselectAllLhas`) set it. `initializeFromTemplate` clears the flag *only* when `seededTemplateIdRef` shows a different template id, so the legitimate initial seed and a genuine template switch still re-seed, but an autosave-driven `template` identity bump (same id, refetched) no longer clobbers a manual selection. Don't reset the flag on every `initializeFromTemplate` or drop the `seededTemplateIdRef` keying — either reintroduces the #791 clobber (fixed #793). The same shape (auto-fill until touched) guards the template-name suggestion in `components/mission/CreateTemplateDialog` via a `nameTouched` ref.
- **`camera_mode: null` means "inherit from mission", not "AUTO".** `effectiveCameraMode = cameraMode ?? mission.camera_mode ?? "AUTO"`. `handleCameraModeChange` maps the UI's INHERIT/AUTO/MANUAL to `null`/`"AUTO"`/`"MANUAL"`; the MANUAL branch only fills *empty* camera fields from the default preset (explicit user/preset values are kept, template defaults and auto-derived focus/zoom are overwritten) and always fills geometry-zoom when untouched. Treating `null` as AUTO breaks the mission-level camera-mode override.
- **`useMissionValidation.handleExport` surfaces the real backend `DomainError` in the toast, not a generic string.** The catch block extracts `err.response.data.detail` (string form *or* `{message}` object form) through the module-local `extractApiErrorMessage`. The generic `mission.validationExportPage.exportError` i18n string is the fallback only when the response carried no usable message. Don't reintroduce a blanket `showNotification(t("...exportError"))` here — the unrelated failure modes (status gate, missing waypoints, drone-enum fallback warnings, etc.) were silent for months because every reason got swallowed into the same generic line. Same shape belongs in any future action handler that hits a route which raises `DomainError`.
- **The clamp-warning branch in `useMissionValidation.handleExport` is a success path, not an error path.** `exportMissionFiles` already parses the 409 blob body and returns `{kind: "clamp_warning", clamps}`; the hook stores those in `clampWarning` and returns *before* the anchor-download dance, so no file leaves the browser and `fetchData()` does not fire (mission status is still VALIDATED server-side). Only a genuine throw — a 409 without `altitude_clamps`, a 400, a 500 — reaches the catch block and the toast. When the operator ticks the `AltitudeClampWarning` checkbox, `ExportPanel` calls `handleExport` again with `acknowledge_altitude_clamps: true`; a fresh `clamp_warning` response overwrites the stored list so the ack flag never silently carries over to a different clamp set. `dismissClampWarning()` clears the panel manually; do not auto-clear on every export attempt — that would discard a still-pending acknowledgment.
- **`useAirportCrud.handleSave` dispatches same-AGL LHA reorders serially, everything else concurrently.** Surface / obstacle / safety-zone / AGL / airport updates and LHAs under *distinct* AGLs all fan out through one `Promise.all`. But 2+ LHA updates sharing one parent AGL are grouped and run in a single serial chain ordered by ascending target `sequence_number` (seq-less position/tolerance-only edits sort stably last and don't reorder). Each LHA update's target `sequence_number` is computed against the *original* order, and the backend `_shift_lha_sequence` already serialises each call under the parent-AGL `SELECT FOR UPDATE` — so firing a multi-light reorder concurrently let the stale-relative targets land in a nondeterministic final order (#100, fixed #104). Don't collapse the per-AGL chains back into the flat `Promise.all`; the serial ascending dispatch is what makes each move lock its final slot before the next applies. Single-item groups behave exactly like the old fully-concurrent path.
- **`useAirportMapHistory` vertex drags persist only `geometry` (+ `boundary`).** `length`/`width`/`heading` are user-facing surface/obstacle metadata and must not be written from a vertex drag — they re-sync only via the explicit "recalculate dimensions" button, otherwise a drag would silently clobber the operator's typed dimensions. And `handleUndo`/`handleRedo` must re-push the rolled-back/replayed geometry into the MapLibre sources via `syncEntityGeometryToMap` (`pages/coordinator-center/syncEntityGeometryToMap`): the live preview writes geometry straight to the map during edits, so a bare `undo()`/`redo()` leaves the map showing the wrong shape. The AIRPORT_BOUNDARY safety zone, obstacle (boundary + re-centroided icon point), and surface (polygon + runway/taxiway centerline) cases each branch to different source ids in `handleVertexGeometryUpdate`; keep those id branches in sync with `safetyZoneLayers` / the surface layer ids. Don't "simplify" any of this away — it is byte-identical to the pre-#629 inline page code on purpose.
