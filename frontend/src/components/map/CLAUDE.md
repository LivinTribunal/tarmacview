# map components package

## Purpose

All map rendering for the app — 2D MapLibre GL view, 3D CesiumJS viewer, and the panels that overlay them. `AirportMap` is the heavyweight component used wherever the operator/coordinator needs interactive map rendering.

## Public API surface

- `AirportMap.tsx` — main interactive map. Default-exports the component; named-exports `AirportMapHandle` (imperative ref API: `flyTo`, `recenter`, `locateFeature(feature)`, layer toggles, etc.). `locateFeature` is the entrypoint pages use for the "double-click to recenter" intent. Two declarative props drive the highlight ring without an imperative call: `focusFeature?: MapFeature | null` (single feature, any type) and `focusLhaIds?: string[] | null` (multi-LHA highlight set — when non-empty it takes precedence over `focusFeature` for the LHA layer). The coordinator inspection editor uses these to mirror its `selectedAglId` / `target_agl_ids` form state onto the map without needing a click. A third declarative prop, `flightPlanScope?: FlightPlanScope | null`, is forwarded to `CesiumMapViewer` and drives the 2D takeoff/landing-layer visibility effect; any page mounting `AirportMap` for a mission must pass `mission.flight_plan_scope` (see the "3D never invents geometry" invariant for why). The scope union is `"FULL" | "MEASUREMENTS_ONLY"` post-#755 - both scopes are airborne and the T/L layers stay hidden in both; the synthetic-bookend path is dead code today but the prop is still threaded through so a future ground-takeoff scope can re-enable it without re-wiring every page. After #685 the file is a thin orchestrator (~823 lines): layer/source bootstrap, infra-rebuild on airport change, waypoint-sync, takeoff/landing visibility effects, and terrain-mode swap all moved to `hooks/useMapBootstrap.ts`; the pure layer-id group tables moved to `mapLayerGroups.ts`. `AirportMap.tsx` keeps the orchestrator role: viewport setup, tool wiring, draw/measure/heading hooks, JSX overlays, and the public `AirportMapHandle` surface.
- `mapLayerGroups.ts` — pure constants module (no React, no MapLibre side effects). Exports `layerGroupMap` (the LayerPanel-toggle-key → MapLibre layer-id table consumed by every visibility loop in `useMapBootstrap` and `useMapLayerSync`), `INTERACTIVE_LAYERS` (layer ids the click router queries for picks), `POINTER_LAYERS` (layers that flip the cursor on hover), and `TOOL_CURSORS` (per-`MapTool` CSS cursor map). When you add a new layer that should toggle from the LayerPanel, register its id here under the right group key — don't fork a parallel table inside the parent component. `safetyZones` and `airportBoundary` are split groups since #771: the `safetyZones` key covers the four `SAFETY_ZONE_*` layers, `airportBoundary` covers `AIRPORT_BOUNDARY_LINE_LAYER` only, so users can hide regular zones while keeping the dashed-boundary outline (or vice versa). Both ids stay in `INTERACTIVE_LAYERS` so click-to-select keeps working independently.
- `hooks/useMapBootstrap.ts` — owns the heavy layer bootstrap that used to live in `AirportMap.tsx`. Exports the `useMapBootstrap(...)` hook (returns `addAllLayers`, `addWaypointLayers`, `handleTerrainChange`) plus the `PENDING_PREVIEW_SOURCE` / `PENDING_PREVIEW_*_LAYER` ids used by feature-creation previews. The hook owns `layersAddedRef` / `cancelStylePollRef` / `appliedTerrainRef`, runs the infra-rebuild effect when `airport.id` changes (remove → re-add → restore LayerPanel visibility → move vertex-edit overlays last → `triggerRepaint`), runs the waypoint-sync + takeoff/landing-visibility effects, and snapshots/restores the viewport across `setStyle(...)` terrain swaps via `waitForStyleLoaded`. `INFRA_SOURCES` and `removeInfraLayers` are file-private — outside callers don't reach in; they call the returned `addAllLayers` / `handleTerrainChange` instead. The hook is exercised by `useMapBootstrap.test.tsx`, which pins the adder call order (`registerAllMapImages → addSafetyZoneLayers → addSurfaceLayers → addObstacleLayers → addBufferZoneLayers → addAglLayers → addMeasureLayersToMap → addHeadingLayersToMap → addHighlightLayers → addPendingPreviewLayers → syncHighlight`), the rebuild sequence, and the trailing `triggerRepaint`. Post-#755 both `FlightPlanScope` values (`FULL`, `MEASUREMENTS_ONLY`) are airborne, so the takeoff/landing visibility branch keeps the T/L layers hidden in every case; the branch survives so a future ground-takeoff scope can flip it back on without re-wiring.
- `CesiumMapViewer.tsx` — the 3D viewer / fly-along wrapper (the only surviving page-level wrapper; `MapViewer.tsx` and `MapPreview.tsx` were unused and removed). Mirrors `AirportMap`'s click + dblclick handlers so 2D and 3D behave identically. Mission pages don't mount it directly — they mount `AirportMap`, which forwards `flightPlanScope` and the fly-along model state into the Cesium pipeline. The viewer itself is now an orchestrator (~312 lines after #664): one-shot viewer setup (depth test, click handlers, initial fly, terrain config) lives in `cesium/useCesiumViewerSetup.ts`; the declutter `postRender` listener and screen-space projection live in `cesium/runDeclutterPass.ts`; type-guard + airport/waypoint lookup for click-hit→`MapFeature` resolution lives in `cesium/featureLookup.ts` (`isMapFeatureType`, `lookupFeature`).
- `obstacleIcons.tsx` — SVG icon set keyed by `ObstacleType`.
- `layers/` — pure MapLibre layer adders: `addAglLayers`, `addObstacleLayers`, `addSafetyZoneLayers`, `addSurfaceLayers`, `addWaypointLayers`, `registerAllMapImages`. Each exports its source/layer-id constants (`AGL_SOURCE`, `WAYPOINT_LAYER`, `WAYPOINT_INSPECTION_HIGHLIGHT_LAYER`, etc.) so callers can target them when toggling visibility. `waypointLayers.ts` is now a pure re-export barrel (split in #628, same precedent as the #598 geo split — the `.../layers/waypointLayers` import path is unchanged and all 5 importers are untouched); the implementation lives in `layers/waypoint/`: `waypointFullLayers.ts` (full trajectory MapLibre coupling: all `WAYPOINT_*` ids — including `WAYPOINT_RECORDING_BOOKEND_LAYER`, the orange ring around the first/last MEASUREMENT of each video pass — the 16-layer `addWaypointLayers`, highlight-filter mutators `updateSelectedFilter` / `updateInspectionHighlightFilter` / `updateWarningHighlightFilter`, `removeWaypointLayers`, `getWaypointLayerIds`), `waypointFullGeoJSON.ts` (pure `FeatureCollection` builders peeled out in #678 — `waypointsToGeoJSON`, `waypointsToLineGeoJSON`, `waypointsToCameraLineGeoJSON`, `waypointsToCameraTargetGeoJSON`; no `map.*` calls, no MapLibre types, leaf module that only depends on `./waypointShared` and is unit-testable without a fake map), `waypointSimplifiedLayers.ts` (all `SIMPLIFIED_*` ids/builders — including `SIMPLIFIED_BOOKEND_SOURCE` / `SIMPLIFIED_BOOKEND_LAYER` + `waypointsToSimplifiedBookendGeoJSON` for the simplified-view bookend dot — the 7-layer `addSimplifiedTrajectoryLayers`, `removeSimplifiedTrajectoryLayers`, `getSimplifiedTrajectoryLayerIds`), and `waypointShared.ts` (colour/key/label helpers). The barrel also re-exports `updateInspectionHighlightFilter(map, inspectionId)` and `updateWarningHighlightFilter(map, ids, severity, simplified)` — the only sanctioned ways to drive the inspection / warning highlight rings. `obstacleLayers.ts` also exports `bufferPolygon(coords, bufferMeters)` — per-edge mitered offset (miter clamped at 2× buffer) used everywhere a buffer ring is drawn around an obstacle or surface footprint; `cesiumUtils.ts` re-exports it so the 3D viewer paints the same shape. `surfaceLayers.ts` is now an orchestrator (~281 lines after #664) that wires runways/taxiways + touchpoint/threshold/end-position marker layers; the pure math (`bufferLineString`, `dedupPairedRunways`, `pairedRunwayLabel`) moved to `layers/surface/surfaceGeometry.ts` and the runway marker adders (`addTouchpointLayers`, `addThresholdLayers`, `addEndPositionLayers` + their source/layer-id constants) moved to `layers/surface/surfaceMarkers.ts`. The parent re-exports the three geometry helpers and all marker source/layer-id constants so existing importers (`AirportMap.tsx`, `vertexEditorGeometry.ts`) keep working unchanged. The MapLibre sprite factories (`createTriangleIcon`, `createTowerIcon`, `createAntennaIcon`, `createTreeIcon`, `createRoundedSquareIcon`, `createHoverIcon`, `createRecordingStartIcon`, `createRecordingStopIcon`, `createAglSquareIcon`, `createPathArrowIcon`, `createThresholdIcon`, `createEndPositionIcon`, `createHatchPattern`) live in `layers/icons/iconFactories.ts`; `mapImages.ts` is now a thin orchestrator (~56 lines) that owns the per-type obstacle icon table, the sprite name/size/pixelRatio call sites, and the `safeAddImage`/`registerAllMapImages` orchestration. `pairedRunwayLabel(surface, byId)` returns `RWY 01/19` when coupled (falls back to the lone identifier) and `dedupPairedRunways` drops the higher-id direction of every paired RUNWAY pair so the map renders one merged feature per physical runway; the planner is unaffected.
- `overlays/` — React panels that float over the map (LayerPanel, LegendPanel, PoiInfoPanel, WaypointListPanel, WarningInfoPanel, SafetyZonesPanel, ObstaclesPanel, GroundSurfacesPanel, AGLPanel, MapHelpPanel, TerrainToggle, MapControlsToolbar, MapStatsPanel, MapWarningsPanel, plus info cards). `PoiInfoPanel` is the single feature-info panel for *every* clicked feature, waypoints included — pages mount it and nothing else. After #683 it is a thin (~66-line) router under `overlays/PoiInfoPanel.tsx` that switches on `feature.type` and forwards `editable` / `surfaces` / `onCoordinateChange` / `onDeleteTakeoffLanding` to per-type sub-panels under `overlays/poi/`: `SurfaceInfoPanel`, `ObstacleInfoPanel`, `SafetyZoneInfoPanel` (handles both the `AIRPORT_BOUNDARY` polygon fork and regular safety arms), `AglInfoPanel`, `LhaInfoPanel`, and `WaypointInfoPanel` (single + stacked waypoints, the `editable` Enter-commit branch, and the `TAKEOFF`/`LANDING` delete-confirm gate). Shared internals live alongside: `poi/rows.tsx` exports `InfoRow`, `CoordRows`, `PolygonCoordRows`, `EditableCoordRows`, `DeleteButton`; `poi/polygonMetrics.ts` exports `computePolygonAreaPerimeter`, `formatArea`, `formatLength` (unit boundaries at 1 km² / 1 km). The sub-panels and shared helpers are package-internal — outside code never imports `overlays/poi/*` directly (the orphaned top-level `overlays/WaypointInfoPanel.tsx` removed in #598 is unrelated to the new `poi/WaypointInfoPanel.tsx`, which only ever renders via the router). List panels emit a `{ feature, mode }` intent — `select` highlights only, `selectAndLocate` also recenters. `MapWarningsPanel` and the side `WarningsPanel` share `warningsPanelGrouping.ts` (in `components/mission/`) so both render the same severity-grouped, dedup-by-constraint shape with their own density classes. `LegendPanel.tsx` is now a thin orchestrator (~108 lines after #680): the seven item arrays + `STATUSES_WITH_FULL_WAYPOINTS` + the `SwatchType` / `LegendItem` types live in `overlays/legend/legendEntries.ts`; the SVG swatch renderer + the shared `SectionChevron` live in `overlays/legend/Swatch.tsx`; the generic collapsible row group lives in `overlays/legend/LegendSection.tsx`; and the nested PAPI/REL section lives in `overlays/legend/AglSystemsSection.tsx`. The import path `@/components/map/overlays/LegendPanel` is unchanged — pages still mount `LegendPanel` directly and never reach into `legend/`. The "Safety Zones & Boundary" section renders when *either* `layers.safetyZones` OR `layers.airportBoundary` is on (since #771) — the legend stays useful while only one side is visible. Since #202 the panel chrome is shared from `components/common/`: the four collapsible panels (`AGLPanel`, `GroundSurfacesPanel`, `ObstaclesPanel`, `SafetyZonesPanel`) render their title-pill + count-badge + rotating chevron via `CollapsiblePanelHeader`; the `HeadingInfoCard` / `MeasureInfoCard` shells wrap `MapInfoCard`; and `MapControlsToolbar` composes the zoom %-field from `ZoomDropdown` and the 2D/3D pill from `Map2D3DToggle`. `MapHelpPanel` gained a `"coordinator"` variant (`<MapHelpPanel variant="coordinator" />`, used by `AirportEditPage`) that renders the airport-editor drawing/vertex-edit shortcuts under the `coordinator.airports.help.*` i18n namespace - the old `components/coordinator/CoordinatorMapHelpPanel` was deleted in its favor. Reuse these `common/` primitives for new panels; don't re-fork a header/card/toggle per overlay.
- `cesium/` — `CesiumFlyAlong`, `CesiumInfrastructure`, `CesiumTrajectory`, plus shared `cesiumColors.ts`, `cesiumUtils.ts`, `terrainSampling.ts`, and `labelDeclutter.ts`. CesiumColors is the canonical mapping from app palette tokens → Cesium `Color` objects. `terrainSampling.ts` exports `resolveWaypointHeights(viewer, points)`, `cartFromAgl(lng, lat, agl, heights)`, and `terrainKey(lng, lat)` — the only sanctioned way to convert mission MSL altitudes into Cesium ellipsoidal positions. `labelDeclutter.ts` exports `DECLUTTER_PRIORITY` (priority constants), `applyDeclutter` (greedy first-fit screen-space pass), `collapseWaypointStacks` (group ground-shared waypoints), `estimateLabelRect`, and `rectsIntersect` — all pure helpers, testable without a live scene. `useTerrainProviderReady.ts` exports `useTerrainProviderReady(viewer): TerrainProvider | null` — the single shared subscription to `viewer.scene.globe.terrainProviderChanged` that all three terrain-consuming hooks (`useTrajectoryTerrainSampling`, `useFlyAlongTerrain`, `useTerrainSettled`) build on (consolidated in #677). Seeds from `viewer?.terrainProvider`, mirrors the live provider after async `setTerrain()` swaps, guards on `viewer.isDestroyed()`, cleans up on unmount. `CesiumTrajectory.tsx` is an orchestrator (~185 lines): the scene-builders moved to `cesium/trajectoryEntityBuilders.ts` (`addPathSegments`, `addPathArrows`, `addWaypointDots`, `addCornerDots`, `addStackedMeasurementDots`, `addTakeoffLanding`, `addCameraHeadingLines`, plus `scopeIncludesTakeoffLanding` and `buildPolylineOptions` — the latter two are the only exports outside this package's consumers and are exported for unit testing) and the terrain-sampling effects moved to `cesium/useTrajectoryTerrainSampling.ts` (a `renderHook`-testable hook returning `sampledHeights: Map<terrainKey, number>`; consumes `useTerrainProviderReady` to trigger re-sampling on provider swap). `CesiumInfrastructure.tsx` is also an orchestrator (~81 lines after #664): the `buildSurfaceEntities`, `buildSafetyZoneEntities`, `buildAirportBoundaryEntities`, `buildObstacleEntities`, `buildBufferZoneEntities`, `buildAglSystemEntities` JSX-tree builders live in `cesium/infrastructureEntities.tsx`, and the "wait for real terrain provider" gate lives in `cesium/useTerrainSettled.ts` (`useTerrainSettled(viewer)` derives readiness from `useTerrainProviderReady` — true once the provider is anything other than `EllipsoidTerrainProvider`, or after an 8 s fallback timer for closed-network / missing-Ion-token deploys; one-shot, stays true). `CesiumFlyAlong.tsx` is an orchestrator (~140 lines after #664): bearing math (`bearing(lat1,lng1,lat2,lng2)`, `bearingBetweenWaypoints`, `bodyTracksTarget`, `BODY_TRACKS_TARGET_TOLERANCE_DEG`) lives in `cesium/flyAlongBearing.ts`; the `SampledPositionProperty` + `CallbackProperty` builder lives in `cesium/buildFlyAlongTimeline.ts` (exports `buildFlyAlongTimeline`, the `Timeline` interface, and the model-render constants `USER_SCALE`, `MIN_PIXEL_SIZE`, `MAX_WORLD_SCALE`; the in-module-only `MODEL_FORWARD_OFFSET_DEG`, `MODEL_NATIVE_DIAMETER_M`, `MODEL_VISUAL_CENTER_Y_M`, and `MAX_PIXEL_SIZE` were de-exported in #794); pre-sampling terrain ellipsoidal heights for every waypoint + re-sampling on provider swap lives in `cesium/useFlyAlongTerrain.ts` (also via `useTerrainProviderReady`). Future agents touching trajectory rendering edit the trajectory builders; touching infra entity construction edits `infrastructureEntities.tsx`; touching fly-along motion edits `buildFlyAlongTimeline.ts`.
- `cesiumProviderSelection.ts` — pure selectors `pickTerrainConfig(url?)` and `pickImageryConfig(url?)` that read the optional `VITE_CESIUM_TERRAIN_URL` / `VITE_CESIUM_IMAGERY_URL` build-time env vars and return either `{ kind: "ion" }` (cloud default) or `{ kind: "url", url }` (closed-network override). `CesiumMapViewer` consumes them so the env-var branching stays unit-testable without booting a viewer.
- `mapStyles.ts` — pure MapLibre style builders, no React: `GLYPHS_URL` (env-overridable glyphs endpoint), `waitForStyleLoaded(map, cb)` (rAF-polls `isStyleLoaded()` then runs `cb`, returns a cancel fn; use it after every `setStyle()` to re-add layers once the new style is live), `makeSatelliteStyle()`, `makeMapStyle()`. Consumed by `AirportMap.tsx` and the measure/heading hooks.
- `hooks/` — per-tool hooks lifted out of `AirportMap.tsx` in #548/#571, behavior-neutral. `useMeasureTool` / `useHeadingTool` / `useWaypointDragTool` / `useInfraDragTool` / `useZoomTool` each own one tool's MapLibre event wiring + data→source sync (heading/measure also export an `add*LayersToMap` adder); `useMapHighlightLayers` exports `addHighlightLayers` / `syncHighlight` / `HIGHLIGHT_*` ids, imported back into `AirportMap`'s layer bootstrap; `usePickAndSelect` owns click/dblclick pick-and-select wiring and re-exports `resolveTransitInsertion`; `useMapLayerSync` owns `TRAJECTORY_CHILDREN`, the LayerPanel toggle cascade, and the layer-visibility effects. After #664 the pick-handler internals split out: `hooks/pickFeatureBuilders.ts` owns the pure pieces (`WAYPOINT_QUERY_LAYERS` and `ALL_WP_HOVER_LAYERS` layer-id tables, `resolveTransitInsertion(properties, lngLat)`, `buildWaypointFeature(hit, waypoints)`, `buildInfraFeature(features, airport)`), and `hooks/useMapHoverCursor.ts` owns the pointer-cursor effect (hover→pointer for `SELECT` / `MOVE_WAYPOINT` / `MOVE_FEATURE`). After #685 the layer-bootstrap core moved here too: `hooks/useMapBootstrap.ts` (described above) owns the adders, infra-rebuild, waypoint-sync, takeoff/landing visibility, and terrain-mode swap. These are package-internal; pages still only touch `AirportMap` + `AirportMapHandle`.

## Invariants

- **Colors** come from `frontend/src/constants/palette.ts` (and `cesium/cesiumColors.ts` for 3D). Raw hex literals (`#abc`, `#aabbcc`, `#aabbccdd`) inside `components/` and `pages/` are blocked by `scripts/check-palette.sh`. The legacy hex sites are tracked in `scripts/check-palette-allowlist.txt` as soft warnings until they migrate; new code must use palette tokens.
- **Animation timings** — durations and easings live in `frontend/src/constants/mapAnimations.ts`. Don't hard-code `duration: 800` in a `flyTo` call; reference the constant.
- **Earth radius / metres-per-degree / WGS84 bounds** come from `frontend/src/constants/geo.ts` (`EARTH_RADIUS_M`, `METRES_PER_DEGREE`, `LAT_BOUNDS`, `LON_BOUNDS`). Don't redeclare a local `EARTH_RADIUS = 6371000` in a layer or cesium helper: `obstacleLayers.ts`, `surfaceLayers.ts`, and `cesium/cesiumUtils.ts` import the shared constant; the `utils/geo` barrel (split into `geo/distance.ts` / `geo/polygon.ts` / `geo/centerline.ts` / `geo/centerlineDistance.ts` behind `geo/index.ts` in #598 — the `@/utils/geo` import path is unchanged) re-exports `EARTH_RADIUS_M` under the legacy `EARTH_RADIUS` name for back-compat. `LAT_BOUNDS` / `LON_BOUNDS` are `{ min, max }` objects (since #597); `overlays/PoiInfoPanel.tsx` validates edited coordinates against them — any overlay panel that range-checks a lat/lon must pull these, never inline `-90/90` / `-180/180`.
- **Tile URLs** are centralized in `frontend/src/constants/mapTiles.ts`. Switching between tile providers happens there, not inline. The four exports (`ESRI_WORLD_IMAGERY_TILES`, `ESRI_REFERENCE_TILES`, `OSM_TILES`, `ESRI_ATTRIBUTION`) read optional `VITE_TILE_*` env vars at build time and fall back to the public ESRI/OSM defaults — closed-network deployments override the URLs without code changes. The 3D viewer's "map" terrain mode reuses `OSM_TILES` so overrides apply to both viewers.
- **Cesium terrain and satellite imagery** route through `pickTerrainConfig` / `pickImageryConfig` in `cesiumProviderSelection.ts`. `CesiumMapViewer` calls `viewer.scene.setTerrain(new Terrain(CesiumTerrainProvider.fromUrl(...)))` for the URL branch and `Terrain.fromWorldTerrain()` for the Ion default; satellite imagery is `UrlTemplateImageryProvider` for the URL branch and `IonImageryProvider.fromAssetId(2)` for Ion. Don't reintroduce inline `import.meta.env.VITE_CESIUM_*` reads in the viewer — that is the whole reason the selectors exist.
- **i18n**: all user-facing strings in panels and overlays use `react-i18next`'s `t()` — never hard-code English in JSX.
- **Layer / source IDs** are the named exports from each `layers/*.ts` module. Always import the constants instead of repeating string literals; that is what keeps add/remove/toggle calls consistent across the codebase.
- **Orchestrator parents stay the public surface.** After the #664 re-baseline (and the #685 `AirportMap` split), `AirportMap`, `CesiumMapViewer`, `CesiumInfrastructure`, `CesiumFlyAlong`, `surfaceLayers`, `mapImages`, and `usePickAndSelect` are thin orchestrators that re-export (or compose) helpers from sibling modules. `LegendPanel` joined this group in #680 — the four `overlays/legend/*` files are package-internal. `PoiInfoPanel` joined the group in #683 — routes by `feature.type` to per-type sub-panels and shared helpers under `overlays/poi/`. Outside code keeps importing them by their parent paths — `./AirportMap` (not `./hooks/useMapBootstrap` or `./mapLayerGroups`), `.../layers/surfaceLayers` (not `.../layers/surface/surfaceGeometry`), `.../layers/mapImages` (not `.../layers/icons/iconFactories`), `.../hooks/usePickAndSelect` (not `.../hooks/pickFeatureBuilders`), `.../overlays/LegendPanel` (not `.../overlays/legend/LegendSection`), `.../overlays/PoiInfoPanel` (not `.../overlays/poi/WaypointInfoPanel` / `poi/rows` / `poi/polygonMetrics`). The extracted sibling modules are package-internal; promoting them to the public surface means moving the symbol onto the orchestrator, not changing import paths in pages/components.
- **`waypointLayers.ts` barrel boundary.** Outside code imports the waypoint layer surface from `.../layers/waypointLayers` and nothing else — never reach into `layers/waypoint/waypointFullLayers` / `waypointFullGeoJSON` / `waypointSimplifiedLayers` directly from a hook/component. The `waypoint/waypointShared.ts` helpers (`coordKey`, `segmentKey`, `offsetSegmentLeft`, `resolveWaypointColor`, `resolveSegmentColor`, `resolveLabel`, `TRANSIT_PATH_COLOR`, `DEFAULT_MEASUREMENT_COLOR`) were file-private in the old single file and are deliberately NOT re-exported from the barrel — keep them off it (same private/public boundary as `toRad` in the #598 geo split). Intra-package edges inside `waypoint/` stay acyclic: `waypointFullGeoJSON` is a leaf that only depends on `waypointShared`; `waypointFullLayers` imports the 4 builders back from `waypointFullGeoJSON` for use in `addWaypointLayers` and also imports the single id `SIMPLIFIED_WARNING_HIGHLIGHT_LAYER` from `waypointSimplifiedLayers` (one-directional, no cycle); don't add a back-import from `waypointFullGeoJSON` → `waypointFullLayers` (the leaf must stay a leaf) or from simplified → full.
- **MapLibre / Cesium SDKs**: import from `maplibre-gl` and `cesium` only inside this package. Higher-level components (pages, contexts) interact via the exported handle APIs, not by reaching into the underlying SDK.
- **Click vs double-click**: single click selects (no camera move), double click selects and recenters. Both maps and side list panels obey this. Recenter goes through `useFocusFeature` (in `frontend/src/hooks/`); never call `flyTo` from a list panel directly. Click-to-select fires under `SELECT`, `MOVE_WAYPOINT`, and `MOVE_FEATURE` — the move tools are supersets of select, not separate modes.
- **`dragPan` stays on by default.** There is no PAN tool; empty-canvas drag pans the map under every active tool. The waypoint and infra-point drag handlers (`hooks/useWaypointDragTool.ts` / `hooks/useInfraDragTool.ts`) are responsible for `dragPan.disable()` on mousedown-on-feature and `dragPan.enable()` on mouseup so feature drags don't fight the pan handler; `AirportMap.tsx` keeps only a defensive `dragPan.enable()` on tool change. If you add a new feature-drag hook, mirror this pattern or panning will trigger mid-drag.
- **Numeric display**: float values shown to users in overlay panels go through `formatNumber(value, decimals)` from `@/utils/format`. Never bind a raw `number` straight into JSX (that surfaces 14-digit floats). Coordinates route through `@/utils/coordinates` — `formatLat`/`formatLon` (9 dp default, optional decimals override; AGL LHA rows pass `8`) and `formatAlt(value, decimals=1)` — never inline `.toFixed(6)` / `.toFixed(4)`. Everything else picks 0/1/2 decimals via `formatNumber`. All helpers return `""` for null/undefined/NaN, so wrap the row in a `value != null` guard if you want to hide the field entirely. Coordinate and altitude readouts in the feature-info panels are click-to-copy: wrap the raw numeric string (no units, no commas) in `@/components/common/CopyableValue`. The one exception is `PoiInfoPanel`'s polygon-vertex row, which copies the whole `lat, lon, alt` triplet. Feature-info coordinate/altitude rows are read-only everywhere (no mount passes `onCoordinateChange`; `EditableCoordRows` keeps click-to-edit only for its display formatting). The waypoint altitude row renders `X m MSL` / `Y m AGL` as two independently-labeled `CopyableValue` segments when AGL is known - those values come from `MapFeatureWaypoint.data.agl` / `camera_target_agl`, which must be populated at every `MapFeature` construction site (`AirportMap`, `MissionMapPage`, `CesiumMapViewer`) or AGL silently goes missing; waypoint drag nulls the moved point's own `agl` (stale until recompute). `CopyableValue` ships its own base color (`text-tv-text-primary-soft`) and hover target (`hover:text-tv-text-primary-hover`) — never pass `text-tv-text-primary` in the wrapper `className` (same specificity, bundle-order wins and the dim base color silently regresses, killing the hover delta). Pass typography classes only (`font-medium`, `tabular-nums`, alignment); leave color to the component. The `<span>` around the MSL/AGL pair in the waypoint row is the deliberate exception — it keeps `text-tv-text-primary` because it colors the literal `" / "` separator that lives outside the two `CopyableValue` children.
- **Cesium label declutter tagging**: every label-bearing entity in the 3D viewer must set `properties.declutterPriority` (use a constant from `cesium/labelDeclutter.ts → DECLUTTER_PRIORITY`, never a magic number) and a stable `properties.declutterGroup` string. The `CesiumMapViewer` postRender pass projects all tagged labels and hides lower-priority ones whose screen rects overlap. Untagged labels are ignored by the pass and will overlap unconditionally. Selection boost (`+1000`) is applied by the viewer pass at render time, not at construction.
- **3D never invents geometry the backend didn't emit.** The 3D path is built from the trajectory waypoint array, same as 2D. The only geometry that could be derived from mission-level `takeoff_coordinate` / `landing_coordinate` (rather than from waypoints) is the synthetic takeoff→first / last→landing legs and the T/L marker entities, and that synthesis is currently gated off for every scope: `scopeIncludesTakeoffLanding(scope)` in `cesium/trajectoryEntityBuilders.ts` (called from `CesiumTrajectory.tsx` and `addPathSegments`) returns `false` unconditionally because both surviving scopes (`FULL` airborne, `MEASUREMENTS_ONLY`) carry no ground TAKEOFF/LANDING waypoints post-#755. The 2D predicate in `AirportMap.tsx` likewise hides `WAYPOINT_TAKEOFF_LAYER` / `WAYPOINT_LANDING_LAYER` in every case. The `flightPlanScope` prop is still threaded through `AirportMap → CesiumMapViewer → CesiumTrajectory` so a future ground-takeoff scope can flip the predicate back on at one site; don't rip out the plumbing.
- **Cesium trajectory builders never mutate the caller's waypoint array.** `addPathSegments` / `addPathArrows` in `cesium/trajectoryEntityBuilders.ts` copy-then-sort (`waypoints.slice().sort((a, b) => a.sequence_order - b.sequence_order)`), so the array they receive keeps its original order and element identity; they always draw in `sequence_order` regardless of how the input is ordered. This is a T3 contract pinned by `trajectoryEntityBuilders.test.tsx` (asserts both non-mutation - order + identity - and sequence-order rendering off an out-of-order input). Don't switch to an in-place `.sort()` on the passed array, and apply the same copy-first rule to any new builder that orders waypoints.

## Gotchas

- **`AirportMap` is large and stateful.** Forwarded ref via `AirportMapHandle` is the only public way to drive imperative actions (camera moves, layer toggles, draw tool changes). Do not poke into internal state.
- **Layer order matters.** `registerAllMapImages` must complete before `addObstacleLayers` (icon-based) is called; the layer adders themselves are order-sensitive because MapLibre paints in declaration order. When inserting a new layer, decide its z-index by where you call its adder.
- **Cesium <-> MapLibre coordinate sync.** Both viewers consume the same WGS84 coordinates but Cesium uses radians internally — go through `cesiumUtils.ts` helpers, don't `toRadians` inline.
- **Cesium altitude is per-waypoint terrain + AGL, not a global offset.** Mission waypoint altitudes are stored MSL. Cesium positions need ellipsoidal heights, which differ from MSL by the geoid undulation and vary point-to-point. The renderer consumes the per-WP `wp.agl` and `wp.camera_target_agl` shipped by the backend's `build_enriched_response` (one batched provider lookup over wp + camera_target points). `CesiumTrajectory` (via the `useTrajectoryTerrainSampling` hook) and `CesiumFlyAlong` (via the `useFlyAlongTerrain` hook) sample `viewer.terrainProvider` per waypoint via `resolveWaypointHeights`, then render at `sampled_cesium_terrain(lng, lat) + agl` - the helper is `cartFromAgl(lng, lat, agl, heights)` in `cesium/terrainSampling.ts`. No `airport.elevation` math in the renderer. FLAT-mode airports stay byte-identical to the prior behavior because `FlatElevationProvider.get_elevation` returns `airport.elevation` and the backend computes `agl = wp.alt - airport.elevation` for FLAT. Do not reintroduce a single airport-ARP `terrainOffset`; it was wrong everywhere except the ARP and broke at airports with negative geoid undulation. Re-sample whenever the live terrain provider swaps (the swap is async; `setTerrain(...)` returns before the world-terrain provider is live, so reading the provider once at mount yields the ellipsoid-zero default). Both hooks track the live provider via the shared `useTerrainProviderReady(viewer)` hook in `cesium/useTerrainProviderReady.ts`, which owns the `terrainProviderChanged` subscription and returns the current `TerrainProvider | null`; the sampling effect lists that value in its dep array so re-sampling fires on every swap. Don't reinstate a local `useState<TerrainProvider>` + `terrainProviderChanged` listener inside an individual hook, and don't read `viewer.terrainProvider` once and cache it.
- **Simplified trajectory dots.** The simplified renderer draws one small black dot per `TRANSIT` waypoint in both 2D (`layers/waypoint/waypointSimplifiedLayers.ts:waypointsToSimplifiedCornersGeoJSON`) and 3D (`CesiumTrajectory.tsx:addCornerDots`). Do not gate on a corner-angle threshold — straight transits between AGL groups never crossed it and looked unmarked.
- **Simplified takeoff/landing markers belong to the `simplifiedTrajectory` layer group only.** `AirportMap.tsx` still toggles a `takeoffLanding` group, but the simplified marker layer ids must not appear in it; the visibility loop iterates `Object.entries` and a later `takeoffLanding=false` would otherwise overwrite the simplified group's `true` and hide the markers.
- **Dblclick on MapLibre.** `flyMapLibreToFeature` defers the `flyTo` to a microtask so MapLibre's HandlerManager doesn't cancel the animation with `_stop(true)` at the end of the dblclick event loop. Don't strip the `queueMicrotask` wrapper.
- **Hex-color allowlist exit criteria**: when an allowlisted file finally migrates to `palette.ts` tokens, drop the entry from `scripts/check-palette-allowlist.txt` in the same change so the script becomes strict for that path.
- **Performance ceiling**: target is 250 waypoints comfortable, 500 max. Adding a per-waypoint React node (instead of a layer feature) regresses this fast.
- **Transit insert altitude.** Clicking a transit segment under MOVE_WAYPOINT inserts a new transit waypoint whose altitude is linearly interpolated between the segment's two endpoints (the click is projected onto the segment line). The interpolation helper lives in `frontend/src/utils/altitudeInterpolation.ts`. Transit-segment line features built by `layers/waypoint/waypointFullGeoJSON.ts:waypointsToLineGeoJSON` (moved from `waypointFullLayers.ts` in #678) carry `from_alt`, `to_alt`, `from_lng`, `from_lat`, `to_lng`, `to_lat` so the pick handler in `hooks/usePickAndSelect.ts` (`resolveTransitInsertion`) can do the projection without re-reading the waypoint model. If you change the segment feature schema, update `resolveTransitInsertion` and the ghost preview together so the previewed altitude matches what gets sent to the backend.
- **`focusFeature` / `focusLhaIds` race on first paint.** The highlight effect runs whenever the props change, but on the very first render it fires before `addAllLayers` has added the `highlight-agl` / `highlight-lha` layers, so the call silently no-ops and the ring never appears until the prop changes again. `AirportMap` keeps `focusFeatureRef` and `focusLhaIdsRef` mirrors of the latest props and `addAllLayers` re-runs `syncHighlight` against those refs once the layers exist (`addHighlightLayers` / `syncHighlight` / the `HIGHLIGHT_*` ids now live in `hooks/useMapHighlightLayers.ts`; `AirportMap`'s bootstrap imports them back and owns the post-add re-sync). If you add a new highlight layer or refactor the layer-bootstrap path, route the post-add re-sync through these refs the same way; otherwise the dropdown-driven highlight in the coordinator inspection editor regresses to "only paints after the second change."
- **Bearing hydration on first paint.** Both 2D and 3D viewers must seed the compass on mount: `AirportMap` lazy-inits its `bearing` state from `getSavedViewport(airport.id)?.bearing` and fires `onBearingChangeRef.current?.(map.getBearing())` once right after map construction; `CesiumMapViewer` invokes its `bearingListener()` once after attaching the `postRender` listener. MapLibre's rotate event and Cesium's `postRender` only fire on change, so without the initial push the parent-owned compass (e.g. `MapDrawingToolbar`) renders at 0° until the user rotates. Don't drop these one-shot calls when refactoring viewer setup.
- **Cesium has no built-in label collision.** Unlike MapLibre's `text-allow-overlap=false`, Cesium happily renders overlapping labels as illegible glyph stacks. The `CesiumMapViewer` declutter pass (a `scene.postRender` listener, with the per-frame projection logic factored into `cesium/runDeclutterPass.ts`) handles this — throttled to ~10 Hz and gated on actual camera movement / entity-count changes, otherwise idle frames re-run the O(n²) projection for nothing. Don't hoist the throttle, and don't tag entities with priorities pulled from thin air; use the `DECLUTTER_PRIORITY` constants so the priority order stays consistent (surface > takeoff/landing > safety zone > AGL > LHA > obstacle > waypoint > arrow).
- **Inspection highlight is part of the `trajectory` layer group.** `WAYPOINT_INSPECTION_HIGHLIGHT_LAYER` (blue ring around `MEASUREMENT` waypoints whose `inspection_id` matches `AirportMap`'s `highlightedInspectionId` prop) is registered in the trajectory group inside `AirportMap.tsx` so the trajectory visibility toggle hides it consistently with the rest of the trajectory render. The filter is reapplied on every layer rebuild via `updateInspectionHighlightFilter` — pages drive it by passing the prop, never by calling the helper directly. The ring color (`INSPECTION_HIGHLIGHT_COLOR = "#3b82f6"`) lives in `frontend/src/constants/palette.ts`.
- **Warning highlight covers every affected waypoint, not the representative one.** `pickGroupForSelection` in `warningsPanelGrouping.ts` merges every `waypoint_ids` across the group (deduped, in input order); the panels hand the full id list to `updateWarningHighlightFilter` so clicking a row that bundles N backend violations highlights all N waypoint sets, not just the first slice. Multi-violation groups set the synthetic `waypoint_ref` to `null` so the detail panel falls into the count branch — keep this contract intact, otherwise the displayed number stops matching the on-map highlight.
- **`bufferPolygon` is a per-edge mitered offset, not a centroid scale.** A long thin runway with a 5 m buffer must draw a uniform 5 m ring on every side and end - the earlier centroid-radial scaling stretched the buffer along the long axis and undershot it on the ends. The helper projects to local meters at the polygon centroid (equirectangular), offsets each vertex along the average of its two adjacent edge normals, clamps miter length at 2× buffer for near-collinear vertices on long thin shapes, then converts back to lon/lat. Pass an open or closed ring; the helper strips a closing duplicate if present and detects orientation from signed area. Both `obstacleLayers.ts` and `cesium/infrastructureEntities.tsx` (via the `cesiumUtils.ts` re-export) consume it - keep the 2D and 3D buffer geometry in sync by routing both through this helper, never inline a different offset scheme.
- **Safety zones and the airport boundary are independent layer toggles since #771.** `MapLayerConfig.safetyZones` drives the four `SAFETY_ZONE_*` MapLibre layers and the regular-zone rows in `SafetyZonesPanel`; `MapLayerConfig.airportBoundary` drives `AIRPORT_BOUNDARY_LINE_LAYER` and the 3D `buildAirportBoundaryEntities` gate in `CesiumInfrastructure`. Both default to `true`. The combined "Safety Zones & Boundary" legend section renders when either flag is on. `SafetyZonesPanel` grays its boundary row when `airportBoundary` is off and its zone rows when `safetyZones` is off, so each side stays clickable while the other is hidden - don't merge the two `*Grayed` branches back into one. When adding a new boundary-only layer id, register it under the `airportBoundary` group in `mapLayerGroups.ts`, not `safetyZones`.
- **Paired runways render as one feature, but `GroundSurfacesPanel` shows both rows.** The map dedup in `surfaceLayers.ts` keeps the lower-id direction of each pair and emits an `identifier_label` MapLibre property via `pairedRunwayLabel` so the label reads `RWY 01/19`. The side panel uses the shared `pairAwareSurfaceOrder` helper from `frontend/src/utils/surfacePairing.ts` (also consumed by the coordinator panel) to keep paired rows adjacent (lower identifier first) with a single chain icon spanning their boundary. AGL `aglColorForType` is the canonical color seam for both 2D (`AGLPanel.tsx`) and 3D (`CesiumInfrastructure.tsx` via `aglCesiumColor`); don't hardcode PAPI's magenta for every AGL marker.
- **3D infra markers must disable terrain depth-test and wait for terrain.** Every `CesiumInfrastructure` AGL + LHA `point`/`label` entity sets `disableDepthTestDistance = Number.POSITIVE_INFINITY` (matching obstacles and trajectory waypoints) - without it, with `globe.depthTestAgainstTerrain` on, the terrain mesh occludes them from some camera angles and they read as "under the ground" (the JARO PAPI A/B-vs-C/D report - the stored elevations were identical, it was purely a depth-test gap). The markers are also gated on terrain readiness: `useTerrainSettled(viewer)` in `cesium/useTerrainSettled.ts` returns `true` once the live terrain provider becomes anything other than `EllipsoidTerrainProvider` (or an 8 s fallback elapses, so a missing-Ion-token / closed-network deploy still shows them), because the entities are `CLAMP_TO_GROUND` and `setTerrain()` is async - mounting before world terrain streams in clamps them to the WGS-84 ellipsoid metres below real ground (the first-open flicker). All three terrain-aware hooks (`useTerrainSettled`, `useTrajectoryTerrainSampling`, `useFlyAlongTerrain`) share the same provider subscription via `useTerrainProviderReady` — mirror both invariants (depth-test disabled + readiness gate) when adding any new clamped infra marker, and route the readiness gate through the shared hook rather than re-implementing the listener.
- **Stacked trajectory waypoints collapse at construction time, not in the declutter pass.** `CesiumTrajectory` runs `collapseWaypointStacks` over visible waypoints, keeps the lowest-altitude one as the visible label, and forces deeper waypoints to `label.show=false` with no declutter tag — so they stay hidden across camera moves regardless of the screen-space pass. Tie-break on equal altitude is by waypoint id (deterministic). If you change waypoint label construction in `addWaypointDots` (in `cesium/trajectoryEntityBuilders.ts`), route the input through `collapseWaypointStacks` first and pass the resulting `stackMap` into the builder; otherwise stacked columns regress to overlapping labels because the declutter pass alone can't recover the visibility decision.
- **Waypoint labels mirror the 2D map exactly.** `addWaypointDots` (in `cesium/trajectoryEntityBuilders.ts`) resolves a per-ground-position label by finding any MEASUREMENT at that `(lng,lat)` and using `inspectionIndexMap[wp.inspection_id]` — same convention as `layers/waypoint/waypointShared.ts:resolveLabel`. Transits, hovers, and measurements without an inspection-index entry render as unlabeled dots. The map must flow through both viewers: `AirportMap` passes `inspectionIndexMap` to `addWaypointLayersFn` for 2D and to `CesiumMapViewer` (then forwarded to `CesiumTrajectory`) for 3D. Don't reintroduce sequence-order labels or a `+N` stack suffix — both diverged from the 2D rendering and confused users into thinking 3D was showing a different flight plan.
- **Recording-bookend waypoints use two different colors on purpose.** The first/last MEASUREMENT of each video pass (`camera_action = RECORDING_START` / `RECORDING_STOP`) carries the recording dwell after #754 collapsed the bookend HOVER into the MEASUREMENT. The full-trajectory view (`WAYPOINT_RECORDING_BOOKEND_LAYER`) paints a transparent orange `#e5a545` ring around the green measurement dot — it matches the 3D HOVER color, leaves the inspection-number label readable, and reads as a "seam affordance" (here is where editing the recording dwell makes sense). The simplified view (`SIMPLIFIED_BOOKEND_LAYER`) paints the same waypoint as a filled green dot using `DEFAULT_MEASUREMENT_COLOR` because the simplified view exists for "where are the measurement nodes" rather than "where can I edit the seam" — it must read as a measurement node, not an editing handle. The full-view filter also includes `stack_count <= 1` so a vertical profile column doesn't falsely paint the seam ring on its collapsed representative. Don't unify the colors and don't drop the `stack_count` clamp; both diverged on purpose. The 3D path keeps the per-WP orange fill via `isRecordingBookend(wp)` / `getWaypointColor` in `cesium/trajectoryEntityBuilders.ts` — when adding a new bookend visualization, paint all three views together so the seam stays visible across 2D full, 2D simplified, and 3D.
- **Stacked `WaypointInfoPanel` shows ranges, not single values.** For a vertical-profile column (more than one waypoint at the same ground position) `WaypointInfoPanel`'s stacked variant drops the Type, Camera Action, separate Altitude, and Hover Duration rows — those describe a single waypoint and would lie about a column. Altitude renders as a range inside the coords block (`100.00 → 140.00 m MSL` / `8.33 → 24.49 m AGL`) and gimbal pitch widens to `→`-separated min/max. The range fields come from new `MapFeatureWaypoint.agl_min` / `agl_max` / `gimbal_pitch_min` / `gimbal_pitch_max` populated by `buildWaypointFeature` in `hooks/pickFeatureBuilders.ts`; every site that constructs a stacked-feature `MapFeatureWaypoint` must populate them or the range row silently falls back to the single-value display. The 2D pick path also prefers a recording-bookend MEASUREMENT when resolving the representative of a stack, so `hover_duration` and `camera_action` read from the seam waypoint instead of whichever id happens to be first in the comma-joined list.
- **`inspectionIndexMap` threads through both viewers and the waypoint list.** The 1-based index that paints labels on the map (`1`, `2`, ...) is the same lookup that `WaypointListPanel` uses for its "Inspection N (count)" bundle header. `AirportMap` owns the map and forwards it through `MapLeftPanelCluster` → `WaypointListPanel`; the 3D path receives it via `CesiumMapViewer` → `CesiumTrajectory`. Don't reconstruct a per-panel index from `inspection.sequence` or array position — that diverges from the on-map labels the moment inspections are reordered or filtered.
- **Fly-along motion is clock-driven, not setTimeout-stepped.** `CesiumFlyAlong` is a thin orchestrator (clock + status/speed effects + progress listener) over `buildFlyAlongTimeline(viewer, waypoints, segmentDurations, heights)` in `cesium/buildFlyAlongTimeline.ts`, which assembles the `SampledPositionProperty` from per-segment 1× durations (`computeSegmentDurations` in `frontend/src/utils/flyAlongTiming.ts` — 3D distance / per-WP speed plus `hover_duration`, with two samples per hovering waypoint so the drone holds position). Playback runs through `viewer.clock` (`shouldAnimate`, `multiplier`, `clockRange = CLAMPED`). Position is per-frame; do not reintroduce a per-waypoint `flyTo` chase or a `setInterval`/`setTimeout` stepper. Heading is a `CallbackProperty`: target-tracking (HR / VP / HPL / MEHT — `bodyTracksTarget(wp)` in `cesium/flyAlongBearing.ts`, true when `wp.heading` matches bearing to `camera_target` within `BODY_TRACKS_TARGET_TOLERANCE_DEG = 5`) recomputes yaw every frame from the interpolated position; row-sweep (FO / PSS) honors `wp.heading` verbatim; everything else (TRANSIT, TAKEOFF, LANDING, exit-from-inspection) uses `bearingBetweenWaypoints` source→next so transits never inherit a stale neighbor heading. The placeholder `.glb` files were exported with +X forward, so `MODEL_FORWARD_OFFSET_DEG = 90` shifts the HPR heading; remove it only if a fresh export uses +Y forward. Pixel size is bounded via Cesium's `minimumPixelSize = 64`, a `maximumScale = 20` world-space cap, and a `MAX_PIXEL_SIZE = 280` scale `CallbackProperty` so the drone never inflates to city-sized at extreme zoom-out. `MissionMapPage` resolves the model URL via `getBundledModel(droneProfile.model_identifier)?.path` with a `/models/drones/generic_quadcopter.glb` fallback and threads state through `AirportMap` → `CesiumMapViewer` → `CesiumFlyAlong`. Bearing must be propagated from the Cesium pipeline up to the parent compass — do not let the parent read its own state and assume it tracks 3D camera yaw.
