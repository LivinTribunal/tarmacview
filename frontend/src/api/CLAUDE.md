# api package (frontend)

## Purpose

Single source of truth for the browser → backend HTTP boundary. Wraps Axios with the JWT interceptor, exposes one module per resource, and seeds the React Query client and key registry. Components and hooks call these functions; nothing else in `frontend/src/` talks to `/api/v1/*` directly.

## Public API surface

- `client.ts` — default axios instance with `baseURL: "/api/v1"`, request interceptor that attaches `Bearer <access>` from `tokenStore` (skips `PUBLIC_PATHS`), and a response interceptor that handles 401-refresh-and-retry (queued during in-flight refresh, single retry per request via `_retry`) plus 503 → `triggerLogout()` + redirect to `/maintenance`. Re-exports `isAxiosError`.
- `queryClient.ts` — `QueryClient` singleton (60s `staleTime`, 5 min `gcTime`, retry 1, `refetchOnWindowFocus: false`) plus `queryKeys` — the canonical key registry. Use `queryKeys.missions.flightPlan(id)` etc.; do not hand-roll keys.
- `missions.ts` — `listMissions`, `getMission`, `createMission`, `updateMission`, `deleteMission`, `duplicateMission`, status transitions (`validateMission`, `completeMission`, `cancelMission`), `exportMissionFiles(id, formats, { include_geozones?, include_runway_buffers?, dji_heading_mode_override?, acknowledge_altitude_clamps? }) -> ExportMissionResult` where `ExportMissionResult = { kind: "file"; blob; filename } | { kind: "clamp_warning"; clamps: AltitudeClamp[] }` (the union mirror of the 200-blob / 409-altitude-clamp split — callers branch on `kind`), `dispatchMission(id, { acknowledge_altitude_clamps? }) -> DispatchMissionResult` where `DispatchMissionResult = { kind: "dispatched"; dispatch: WaylineDispatchResponse } | { kind: "clamp_warning"; clamps }` (same 409-clamp union shape as export — dispatch builds the same KMZ server-side; `WaylineDispatchResponse` lives in `@/types/fieldLink.ts`), `downloadMissionReport`, inspection child resources (`addInspection`, `updateInspection`, `removeInspection`, `reorderInspections`), trajectory + flight plan (`generateTrajectory`, `generateAndFetchTrajectory`, `getComputationStatus`, `getFlightPlan`, `revalidateFlightPlan`, `batchUpdateWaypoints`, `insertTransitWaypoint`, `deleteTransitWaypoint`).
- `airports/` — package (barrel `airports/index.ts` + per-sub-resource modules: `core`, `terrain`, `surfaces`, `obstacles`, `safetyZones`, `agls`, `lhas`) — airport CRUD plus surface / obstacle / safety-zone / AGL / LHA child resources, terrain-DEM upload/delete/download, bulk drone change. The historic `@/api/airports` import path is preserved by the barrel; every name/signature/URL is identical to the pre-split single file (see #634). Add new sub-resource calls to the matching module and re-export from `index.ts`; do not reintroduce a flat `airports.ts`. Pair-link writes: `createReverseSurface(airportId, surfaceId, { identifier? })`, `coupleSurface(airportId, surfaceId, { target_surface_id, primary })`, `decoupleSurface(airportId, surfaceId)` — all return the freshly-fetched `SurfaceResponse`; consumers re-sync `selectedFeature.data` from the parent airport so the UI reflects the new pair state without a second click. `fetchElevationAt(airportId, lat, lon)` is the per-point ground lookup used by `useElevationResolver` for takeoff/landing placement; returns `{ elevation, source }` where `source ∈ {"FLAT", "DEM_UPLOAD", "DEM_API", "API_FALLBACK"}`. `updateAirport`, `uploadTerrainDEM`, `deleteTerrainDEM`, and `downloadTerrainData` accept an optional `{ rewriteExisting?: boolean }` (defaults to `true`) that maps to the backend's `rewrite_existing` query param — the coordinator `TerrainSettingsCard` checkbox controls it. `extractPhotoMetadata(airportId, files)` (in `airports/photoMetadata.ts`, re-exported from the barrel) POSTs a `FormData` of one or more photos to `/airports/{id}/extract-photo-metadata` and returns `PhotoMetadataResponse` (per-image coordinates + PAPI lens heights, read-only) for the coordinator "extract from image" seed flow.
- `admin.ts` — super-admin user management, audit-log read/export, system settings. `listAuditLogs` and `exportAuditLog` accept an optional `airport_id` to scope the feed (used by the super-admin airport detail page and the audit-log filter chip).
- `auth.ts` — `getMe()` / `updateMe(payload)` against `GET` / `PUT /api/v1/auth/me`. The `UserUpdatePayload` shape is `{ name?, password?, current_password? }`; the backend requires `current_password` whenever `password` is sent. Used by `UserSettingsDialog` (Account Settings modal); `AuthContext` calls `getMe()` directly inside `refreshUser()` so the post-update name appears in the navbar without a remount. The initial post-login `/auth/me` hit is still done inline in `AuthContext` because it has to attach a freshly-issued bearer before the interceptor knows about it - don't route that one through `auth.ts`.
- `fieldLink.ts` — `getFieldLinkStatus()` against `GET /field-link/status`, the backend's field-hub proxy. The backend returns the degraded `{hub_online: false, devices: []}` shape (HTTP 200) when no hub is configured or reachable, so a thrown error here means the backend itself is unreachable — `useFieldLinkStatus` maps that to the same no-hub shape. Types in `@/types/fieldLink.ts` (kept in lockstep with `backend/app/schemas/field_link.py`).
- `droneMedia.ts` — `listDroneMedia()` (mission-grouped media + unassigned bucket), `assignDroneMedia(mediaId, missionId | null)` (manual reassign; null parks the file in the unassigned bucket), `confirmDroneMediaIngest(missionId)` (idempotent ingest confirm). Types in `@/types/droneMedia.ts`, kept in lockstep with `backend/app/schemas/drone_media.py` + the media shapes in `schemas/field_link.py`. Consumed by `UploadDroneMediaDialog`.
- `droneProfiles.ts`, `cameraPresets.ts`, `inspectionTemplates.ts` — reference-data CRUD.
- `queries/` — read-only React Query hooks for the only two resources that use them: `queries/airports.ts` exports `useAirportSummaries` (airport-summary list) and `queries/droneProfiles.ts` exports `useDroneProfiles` (drone-profile list). Both are plain `useQuery` wrappers with a 5-min `staleTime` and nothing else; the detail-read hooks and the create/update/delete mutation hooks that used to live here were removed in #794 as unused. Every other resource (missions, admin, inspection templates) is called through the direct `@/api/*` functions. Add a hook here only when a component genuinely needs React Query caching for the airport-summary or drone-profile read; don't reintroduce the mutation wrappers, the detail-read hooks, or wrappers for the raw-function resources.

## Invariants

- **Every backend call goes through the shared `client` from `client.ts`.** Don't `import axios from "axios"` in feature code — that bypasses the JWT interceptor and the 401-refresh flow.
- **All paths are relative to `/api/v1`.** The axios instance prepends the prefix; functions pass paths like `/missions/${id}`, never `/api/v1/missions/${id}`.
- **Read TypeScript types from `@/types/*`, not hand-rolled inline shapes.** Schemas there are kept in sync with the Pydantic DTOs (`MissionResponse` ↔ `MissionResponse`, `FlightPlanResponse` ↔ `FlightPlanResponse`, etc.). Inline shapes drift.
- **React Query keys come from `queryKeys` in `queryClient.ts`.** The registry is what `queryClient.invalidateQueries(...)` calls match against — a hand-rolled key won't invalidate after a mutation.
- **Mutating endpoints don't auto-invalidate.** Raw `@/api/*` functions never call `invalidateQueries`, and after #794 the `queries/` layer is read-only — there are no mutation hooks left. Every mutation, airports and drone profiles included, goes through a raw `@/api/*` function, so the calling page/component owns any refresh (e.g. the `refetch` returned by `useAirportSummaries`). The `queryKeys` registry still carries `airports.detail` / `droneProfiles.detail` / list keys, but nothing invalidates them today; if you add a write that must refresh a cached read, invalidate from the calling component, not from a new hook.
- **Blob endpoints return `{ blob, filename }`.** `exportMissionFiles` and `downloadMissionReport` parse `Content-Disposition` (rfc-5987 `filename*` preferred) via `parseContentDispositionFilename`. Don't read `res.headers["content-disposition"]` ad-hoc — reuse the helper.
- **`PUBLIC_PATHS` in `client.ts` is the only allowlist that bypasses the bearer header and the refresh retry.** Adding a new unauthenticated endpoint requires adding its prefix there; otherwise the interceptor will attach a stale token or trigger a refresh on a public route.

## Cross-package dependencies

- Imports from: `axios`, `@tanstack/react-query`, `@/auth/tokenStore` (token getters + `triggerLogout`), `@/types/*` (DTO types).
- Imported by: `@/pages/**`, `@/components/**`, `@/hooks/**`, `@/contexts/**`. The router and contexts wire the `QueryClient` and call functions from `missions.ts` / `airports/` directly. Most pages also call the direct `@/api/*` functions; only the airport-summary read (`useAirportSummaries`, in `AirportSelectionView`) and the drone-profile-list read (`useDroneProfiles`, in `DashboardPage` and `useMissionList`) go through the `queries/` hooks.

## Gotchas

- **`client` resets `Content-Type` to `multipart/form-data` for a `FormData` body.** The instance default is `application/json`, and axios 1.x `transformRequest` json-stringifies a `FormData` body when that header is present — so a multipart upload reaches the server with no file fields (422). The request interceptor detects `config.data instanceof FormData` and sets the multipart header so axios emits the correct boundary. This covers every FormData upload through the shared client: photo-metadata extract, terrain-DEM upload, drone-profile thumbnail. Don't remove the reset or hard-code a boundary.
- **401 refresh is single-flight.** While one refresh is in progress, every subsequent 401'd request is queued in `refreshQueue` and resolved with the new token after `/auth/refresh` completes. `processQueue` dispatches both success and failure paths — don't add a second refresh trigger or you'll race the token swap.
- **`generateAndFetchTrajectory` is the convenience wrapper for "compute + reshape"** — it calls `generateTrajectory` and unpacks `{ flight_plan, mission_status }` into `{ flightPlan, missionStatus }` for the camelCase frontend. The raw `generateTrajectory` returns the backend snake_case shape verbatim; pick one and stick with it per call site.
- **`generateTrajectory` accepts an `AbortSignal`** so the operator can cancel a long compute. Pass the signal from the page's `AbortController`; the backend treats the disconnected request as a normal cancellation and the polling endpoint surfaces the failed state.
- **`revalidateFlightPlan` ≠ `generateTrajectory`.** Revalidate (`POST /missions/{id}/revalidate`) re-runs the safety pipeline against the persisted plan and returns the same `FlightPlanResponse` shape with a refreshed `validation_result` — waypoint UUIDs and lon-lat-alt are byte-identical pre/post. The "Validate Trajectory" button uses this; the "Recompute Trajectory" button uses `generateTrajectory`. Disable the validate button while there are dirty waypoint edits or `mission.has_unsaved_map_changes` — those need a recompute first. Returns 409 when the mission has no flight plan.
- **`batch_update_waypoints` caps at 200 updates per call.** The backend rejects larger batches with 400. Chunk in the page layer if you ever need more.
- **503 force-logs out and redirects.** The response interceptor reads 503 as "system in maintenance"; it calls `triggerLogout()` and sets `window.location.href = "/maintenance"`. Don't surface 503 errors to UI toast handlers — they will never run.
- **`exportMissionFiles` defaults the geozone flags to `false`.** Pass `{ include_geozones: true, include_runway_buffers: true }` only after the panel has confirmed both format capability (`@/constants/exportCapabilities`) and `mission.supports_geozone_upload === true`. The backend gate rejects mismatches with 400; the disabled-checkbox UI is the first line of defense, not the only one. MAVLink output extension switches from `.waypoints` to `.plan` server-side when `include_geozones` is on, so consume `filename` from the response — don't synthesize it.
- **`exportMissionFiles` forwards `dji_heading_mode_override` even when the picker is hidden.** The wire shape always sends the key (defaults to `null`), so callers don't need to omit it conditionally. Only pass a non-null value when `(KMZ || WPML) ∈ formats` AND the active drone profile is DJI — the backend ignores it for every other generator, but sending a non-null override on a non-DJI export would still write back to `mission.dji_heading_mode` as a side effect, polluting the operator's last-used preference. Pydantic rejects unknown values at 422.
- **`dispatchMission` reuses the export route's 409-clamp contract.** Its catch branch parses the 409 body through the same `readClampWarningFromError` helper as `exportMissionFiles` (handles both a Blob and an already-parsed dict) and returns `{kind: "clamp_warning", clamps}` — callers branch on `kind` and re-fire with `acknowledge_altitude_clamps: true` after the operator confirms (`SendToDroneSection` owns that retry). Every other failure (status-gate 409, hub-unreachable 502, etc.) propagates as a thrown axios error; the section surfaces `response.data.detail` (string or `{message}` form) inline. Don't fork a second clamp parser.
- **`exportMissionFiles` parses a 409 blob body as a `clamp_warning`, not an error.** The route uses `responseType: "blob"` so a 409 body still comes back as a blob; the catch branch reads the body with `body.text() + JSON.parse`, looks for `detail.altitude_clamps`, and returns `{kind: "clamp_warning", clamps}` instead of re-throwing. Callers must branch on `result.kind` before reaching for `result.blob` — the union is the explicit "two valid outcomes" shape and the type system enforces it. Any other 409 (mission status gate, no flight plan, etc.) still propagates as a thrown axios error and goes through the generic `extractApiErrorMessage` toast in `useMissionValidation`. The retry path is the caller's: `ExportPanel` re-fires `exportMissionFiles` with `acknowledge_altitude_clamps: true` once the operator ticks the checkbox; this module never auto-retries.
